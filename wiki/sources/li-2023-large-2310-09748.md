## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

This paper proposes LAIL, a learning-based in-context example selector for code generation that treats the target LLM itself as the supervision source. Instead of retrieving demonstrations only by lexical similarity, LAIL scores each candidate example by the normalized log-probability of the ground-truth program conditioned on the requirement and that example, then uses the resulting positive/negative labels to train a GraphCodeBERT-based retriever with contrastive learning. Across MBJP, MBPP, and MBCPP, LAIL consistently improves Pass@k over heuristic selectors such as AceCoder, TOP-k-SBERT, and TOP-k-GraphCodeBERT on both CodeGen and GPT-3.5. The paper also reports better human judgments and nontrivial transfer across models and datasets, framing example selection as an LLM-preference alignment problem rather than pure text matching.

## Problem & Motivation

The paper studies instability in in-context learning for code generation: with frozen LLM parameters, output quality depends heavily on which requirement-code demonstrations are placed in the prompt. Prior selectors either sample randomly or retrieve by textual overlap, so they optimize surface similarity rather than whether a candidate example actually helps the target model generate the correct program. The authors argue that good demonstrations should be defined by the LLM's own preferences and prior knowledge, especially under strict prompt-length budgets where only a few examples can be shown.

## Method

- **Task setup**: code generation is cast as in-context learning, where a model predicts `y_t ~ P(y_t | x_1, y_1, ..., x_m, y_m, x_t, Θ)` with frozen parameters `Θ`.
- **Two-stage candidate construction**: for each training example `e_i`, BM25 first narrows the candidate pool to `|S_i| = 50` examples with high lexical overlap, trading coverage for efficiency.
- **LLM-aware scoring**: each candidate example `(x_q^i, y_q^i)` is scored by the normalized ground-truth log-probability `M = (1 / |y_i|) * P_LLM(y_i | x_q^i, y_q^i, x_i)`, where `P_LLM(y_i | ...) = Σ_{u=1}^{|y_i|} log p(t_{i,u} | x_q^i, y_q^i, x_i, t_{i,<u})`.
- **Positive/negative labeling**: examples in `S_i` are ranked by `M`; the top `z = 5` become the positive set `S_i^p`, and the bottom `v = 5` become the negative set `S_i^n`.
- **Retriever training**: requirements are encoded with GraphCodeBERT, and a contrastive objective pulls the query embedding toward a sampled positive and pushes it away from sampled negatives: `L = -log( exp(s(E_[CLS]^i, E_[CLS]^{i,p}) / τ) / Σ_{E_[CLS]^{i,n} in N_i} exp(s(E_[CLS]^i, E_[CLS]^{i,n}) / τ) )`, where `s(.)` is cosine similarity.
- **Negative sampling**: the contrastive negative set size is `|N_i| = 64`, implemented as one sampled hard negative from `S_i^n` plus `h = 63` additional negatives from `H = R \\ S_i`.
- **Training details**: the retriever uses learning rate `5e-5`, batch size `32`, and trains for about `1` hour on `4 x NVIDIA A100`.
- **Inference**: the trained retriever computes `E_[CLS]^t` for a test requirement, ranks all training examples by cosine similarity, and selects top-`r` examples as the prompt in decreasing relevance order.
- **Generation protocol**: candidate scoring uses greedy decoding with max length `500`; final generation uses nucleus sampling with `temperature = 0.8`, `top-p = 0.95`, max length `500`, and `5` sampled programs per test case for Pass@`{1,3,5}` evaluation.

## Key Results

- On **CodeGen**, LAIL reaches Pass@1 `21.30` on MBJP, `18.60` on MBPP, and `19.08` on MBCPP, improving over the SOTA baseline by `11.58%`, `6.89%`, and `5.07%`, respectively.
- On **GPT-3.5**, LAIL reaches Pass@1 `53.14` on MBJP, `50.60` on MBPP, and `54.71` on MBCPP, yielding relative gains of `4.38%`, `2.85%`, and `2.74%` over AceCoder.
- Human evaluation on `400` generated programs reports LAIL scores of correctness `1.764`, code quality `1.839`, and maintainability `1.782`, improving over the SOTA baseline by `11.504%`, `5.811%`, and `7.027%`.
- Using probability-based feedback beats alternative labeling signals: compared with Match-CodeBLEU, LAIL improves Pass@1 by `4.05%` on MBJP, `5.86%` on MBPP, and `3.77%` on MBCPP under GPT-3.5.
- Transfer across models is nontrivial: a retriever trained from GPT-3.5 feedback improves CodeGen Pass@1 by `22.30%`, `14.47%`, and `16.94%` over random selection on MBJP, MBPP, and MBCPP.
- Transfer to ChatGPT is also useful despite missing token probabilities: GPT-3.5-derived LAIL improves ChatGPT Pass@1 by up to `4.31%` relative to the SOTA baseline.

## Limitations

- LAIL still depends on a BM25 pre-filter, so the method can miss helpful demonstrations that are semantically relevant but lexically dissimilar if they never enter `S_i`.
- The labeling pipeline requires token-level probability feedback from the base LLM, which is unavailable for many black-box chat systems; the paper addresses this only indirectly through transfer experiments.
- Experiments cover only three benchmark datasets and three base models, all in code generation, so broader generalization to other code tasks or stronger post-2023 models is unverified.
- The paper fixes prompt format, decoding hyperparameters, and candidate-pool construction empirically rather than tuning them jointly, leaving some performance headroom unexplored.
- Hard negatives come from BM25-near examples, and the authors explicitly note possible false negatives in `S_i^n`, which can hurt retriever training when too many are sampled.
