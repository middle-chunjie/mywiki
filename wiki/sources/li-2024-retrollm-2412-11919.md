## Summary

RetroLLM proposes a unified retrieval-augmented generation framework in which a single autoregressive large language model first generates corpus-constrained clues, then generates fine-grained evidence directly from the corpus, and finally answers the question without a separate retriever or chunk re-injection step. The core design combines hierarchical [[fm-index]] constraints, clue-based candidate document scoring, and [[forward-looking-decoding]] that boosts tokens whose future windows are predicted to be relevant. On five open-domain QA datasets, RetroLLM consistently beats standard RAG and iterative retrieval baselines while using far fewer tokens: for example, it reaches `61.6/49.8` Acc/F1 on NQ, `74.3/72.8` on TriviaQA, and `48.9/36.2` on 2Wiki, while consuming roughly `2.1x` fewer tokens than Naive RAG on average.

## Problem & Motivation

Existing [[retrieval-augmented-generation]] pipelines usually rely on a separate dense retriever plus retrieved text chunks appended to the generator input. The paper argues this design introduces deployment overhead, redundant input tokens, rigid retrieval granularity, and a training mismatch between retrieval and generation. Prior [[generative-retrieval]] methods remove the retriever by generating document identifiers, but they still require mapping those identifiers back to document text before answer generation. RetroLLM instead aims to unify retrieval and generation inside one decoding trajectory so the model can decide when to retrieve more evidence, how much evidence to retrieve, and when to answer. A key obstacle is [[false-pruning]] during constrained evidence generation over a large corpus, where early locally plausible prefixes steer decoding into irrelevant documents and eliminate correct evidence paths.

## Method

- **Task decomposition**: model retrieval as constrained evidence generation `P(e|q, C) = ∏_{t=1}^{T_e} P(e_t | e_{<t}, q, I_c)` and answer generation `P(a|q,e) = ∏_{t=1}^{T_a} P(a_t | a_{<t}, q, e)`, where `I_c` is a corpus-level [[fm-index]].
- **Hierarchical FM-Index constraints**: build one global index `I_c = FM-Index(C)` over the whole corpus and one document-level index `I_d = FM-Index(d)` per document. The global index constrains clue generation; document-level indexes constrain evidence generation inside a reduced candidate set.
- **[[clue-generation]] stage**: generate clue phrases `c_i` under corpus constraints, then compute document scores `S_gen(d) = Σ_i w_i · f(c_i, d)` with clue weights `w_i = log(N / CF(c_i)) + log(N / DF(c_i))` and term score `f(c_i, d) = log(1 + TF(c_i, d))`.
- **Auxiliary lexical clues**: expand generated clues with a sparse lexical model and combine generated-clue ranking `R_1(d)` with lexical ranking `R_2(d)` through [[reciprocal-rank-fusion]], `S(d) = w_1 Σ_{r∈R_1(d)} 1/r + w_2 Σ_{r∈R_2(d)} 1/r`.
- **[[forward-looking-decoding]]**: locate candidate future windows around clue occurrences using document FM-index operations `Loc` and `Ext`, score each window with a reranker `S_w(w) = f_rel(q,w)`, and adjust allowed-token logits as ``l~(t) = l(t) + λ · max_{w∈W_t} S_w(w)``.
- **Generation protocol**: clues are emitted in the format ``<|clue|> c1 <|sep|> c2 ... <|/clue|>``, evidence in ``<|evidence|> e1 <|sep|> e2 ... <|/evidence|>``, and final answer generation is unconstrained.
- **Training construction**: for each labeled QA pair, use SPLADE-v3 to retrieve top documents and clues, rerank evidence, keep evidence that both contains the answer and is judged relevant by Llama3.1-70B-Instruct, then train on `(q, clues, evidence, answer)` with answer-loss weight `γ = 2`.
- **Optimization details**: backbone is `Mistral-7B-Instruct`; LoRA uses `lora_r = 16`, `lora_alpha = 64`, `epochs = 3`; the middle `80%` of evidence tokens is masked out during training to focus learning on critical boundary decisions.
- **Inference hyperparameters**: maximum generated clues `= 5`, expanded clues `= 8`, maximum evidence `= 5` for single-hop and `10` for multi-hop QA, rank-fusion weights `w1 = 1`, `w2 = 2`, and future-window coefficient `λ = 100`.
- **System implementation**: knowledge source is the `2018-12-20` Wikipedia dump; FM-index is implemented with `sdsl-lite`, called from Python through SWIG, and experiments run on `8 × NVIDIA A800-80GB` GPUs plus a `64`-core Intel Xeon Platinum 8358 CPU.

## Key Results

- On NQ, RetroLLM scores `61.6` Acc / `49.8` F1 with `302` tokens, versus Naive RAG at `52.4` / `41.1` with `919` tokens.
- On TriviaQA, RetroLLM reaches `74.3` Acc / `72.8` F1 with `287` tokens, outperforming Iter-RetGen (`71.0` / `69.9`) and Naive RAG (`69.3` / `65.9`).
- On HotpotQA, RetroLLM achieves `61.9` Acc / `47.2` F1, far above Naive RAG (`37.8` / `35.8`) and IRCoT (`37.3` / `41.5`).
- On out-of-domain PopQA and 2Wiki, RetroLLM obtains `65.7` / `43.0` and `48.9` / `36.2`, showing stronger generalization than prior RAG baselines.
- Retrieval quality also improves: averaged across datasets, RetroLLM gets single-hop `R@1 = 56.6`, `R@5 = 67.9`, and multi-hop `R@1 = 29.3`, `R@5 = 49.6`, while using only `3.29` and `4.24` passages on average.
- Efficiency remains strong: RetroLLM uses about `2.1x` fewer total tokens than Naive RAG and about `6x` fewer than Iter-RetGen on average, while total latency (`786 ms`) is much lower than Self-RAG (`3269 ms`) and Iter-RetGen (`2332 ms`).
- Ablations show each component matters: removing future windows drops in-domain Acc/F1 from `66.0/56.6` to `44.3/43.2`, while naive constraints collapse to `27.2/28.0` in-domain and `21.8/20.7` out-of-domain.

## Limitations

- The method still depends on clue expansion with an external sparse lexical model, so the full RAG pipeline is not yet completely end-to-end optimized.
- Although it is faster than most complex RAG systems, it remains slightly slower than Naive RAG because constrained evidence generation increases output-token overhead.
- The unified decoding only covers retrieval and answer generation; the paper does not yet fold in broader reasoning steps such as intent analysis, decomposition, retrieval-necessity prediction, or citation planning.
- Evidence generation and evaluation are studied mainly on open-domain QA with a fixed 2018 Wikipedia corpus, so transfer to fresher or non-Wikipedia knowledge sources is not established.
