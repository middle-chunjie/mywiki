## Summary

This paper studies tool retrieval as a distinct bottleneck in tool-augmented LLM systems and proposes an iterative feedback framework that uses the tool-usage LLM to improve a separate dense retriever. At each round, the LLM first comprehends the instruction and retrieved tools, then assesses missing or mis-ranked tools, and finally rewrites the instruction to better reflect user intent and tool affordances. The retriever is further trained with iteration-aware tokens and hard negatives so it can benefit from those refined instructions. On the authors' TR-bench benchmark, the method substantially outperforms BM25, OpenAI Ada embeddings, and ToolRetriever in both in-domain and out-of-domain evaluation, while also improving downstream ToolLLaMA tool-use success rates.

## Problem & Motivation

Existing tool-learning approaches either rely on in-context learning over a small visible tool set or fine-tune tool-usage models, but both break down when real-world tool inventories are large and frequently updated. The paper argues that simply reusing document retrieval methods is insufficient because tool retrieval must understand ambiguous user instructions, structured tool descriptions, and the preferences of the downstream tool-usage model. The core motivation is therefore to make the retriever absorb LLM-level understanding without turning retrieval into a prohibitively expensive full-LLM ranking step.

## Method

- **Task setup**: given instruction `q` and tool set `D = {d_1, ..., d_N}`, a retriever `R(q, d_i)` returns top-`K` candidate tools. The paper sets `K = 10` and iterates refinement for `T = 3` rounds.
- **Dense retriever backbone**: the base retriever uses a dual-encoder / bi-encoder formulation with embeddings `E(q)` and `E(d)` and relevance `R(q, d) = sim(E(q), E(d))`, trained from a contrastive objective over positive and negative tools.
- **Three-step LLM feedback**: at iteration `t`, the LLM consumes refined instruction `q^t` plus retrieved tools `{d_1^t, ..., d_K^t}` and generates: comprehension `F_C = LLM(P_C, q^t, {d_k^t})`, assessment `F_A = LLM(P_A, q^t, {d_k^t}, F_C)`, and refinement `F_R = LLM(P_R, q^{t-1}, {d_k^{t-1}}, F_A)`.
- **Instruction refinement target**: the LLM enriches unresolved user goals and injects scenario-specific tool-usage cues; if retrieval is already adequate, it emits the special token `N/A` and stops further rewriting.
- **Iteration-aware training**: each instruction is prefixed with a token such as `Iteration 0`, `Iteration 1`, ... so the retriever can distinguish refinement stages rather than treating all rewritten instructions as IID queries.
- **Hard negatives**: for each instruction, the model samples an incorrect tool from the retrieved top-`K` list as a hard negative, extending the loss beyond standard in-batch negatives to sharpen discrimination among semantically similar tools.
- **Joint objective**: feedback training sums losses across iterations as `L_feedback = Σ_{t=1}^{T} α^t L(q^t)`, with balancing factor `α = 1`, so the retriever learns from both earlier and later refined instructions.
- **Implementation**: the retriever is Sentence-BERT over `BERT-base`, trained with learning rate `2e-5`, `500` warm-up steps, and batch size `64`; the feedback LLM is `gpt-3.5-turbo-0125`, and experiments run on `4 x NVIDIA A6000 (48 GB)`.

## Key Results

- **TR-bench scale**: training uses `195,937` ToolBench instructions; in-domain evaluation covers `1,587` instructions over a tool set of up to `13,954` tools, while out-of-domain evaluation uses `553` T-Eval instructions and `1,000` UltraTools instructions.
- **In-domain overall gains**: on ToolBench-All, the method reaches `NDCG@1 = 88.53`, `NDCG@3 = 87.00`, and `NDCG@5 = 88.83`, beating ToolRetriever's `75.73 / 83.19 / 83.06`.
- **Hard case improvement**: on the intra-category multi-tool (`I2`) split, it achieves `89.01 / 85.46 / 87.10` at `NDCG@1/3/5`, versus ToolRetriever's `68.24 / 77.43 / 77.90`, a `30.44%` relative gain at `NDCG@1`.
- **Out-of-domain robustness**: on T-Eval it reaches `84.45 / 73.31 / 74.45 / 80.25` at `NDCG@1/3/5/10`; on UltraTools it reaches `49.30 / 47.50 / 54.30 / 59.92`, slightly or clearly above the strongest baseline depending on the cutoff.
- **Downstream tool use**: with ToolLLaMA on `I2`, retrieval improvements translate into better execution metrics: pass rate rises from `68.5%` to `75.6%`, and win rate from `60.8%` to `65.9%`.
- **Ablations**: removing warm-up, hard negatives, or joint training all hurts performance; the strongest degradation is `w/o joint & hard-neg`, which drops to `83.77 / 77.67 / 81.21 / 83.69` on `NDCG@1/3/5/10`.
- **Iteration analysis**: performance improves monotonically from 1 to 3 iterations (`NDCG@1: 85.69 -> 87.78 -> 89.01`) while average latency rises from `6.12s` to `8.59s` to `10.30s` per instruction.

## Limitations

- The method adds extra inference latency because each retrieval round may require another LLM call; the paper explicitly frames this as an inherent cost of iterative feedback.
- Feedback is generated offline before retriever training, so the current system does not test whether online feedback during training or deployment would be even better.
- Like many retrieval datasets, TR-bench contains possible false negatives, meaning unlabeled but actually useful tools can distort both training and evaluation.
- The retriever still depends on a separate LLM critic (`gpt-3.5-turbo-0125` here), so quality and cost are partly tied to the external feedback model.
