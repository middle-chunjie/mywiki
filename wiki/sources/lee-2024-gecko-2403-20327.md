## Summary

Gecko is a `1.2B`-parameter text embedding model that distills retrieval behavior from large language models into a compact dual encoder. The paper builds a two-step data pipeline called FRet: first an LLM generates diverse `(task, query)` pairs from web passages with few-shot prompting, then the same LLM reranks retrieved neighbors to relabel better positives and hard negatives. Gecko is pre-finetuned on large unlabeled text-pair corpora and fine-tuned on FRet plus academic QA, NLI, classification, and multilingual retrieval data. On MTEB, Gecko-1B-`256` reaches `64.37` average and Gecko-1B-`768` reaches `66.31`, outperforming several larger or higher-dimensional baselines while maintaining strong retrieval and semantic similarity performance.

## Problem & Motivation

General-purpose embedding models need large and diverse supervision, but manually labeled retrieval and similarity data are expensive to collect and often narrow in task coverage. The paper asks whether LLMs can transfer their broad task knowledge into a smaller retriever more directly. It targets a unified embedding model that can serve retrieval, semantic similarity, classification, clustering, reranking, and multilingual retrieval without training a separate encoder per task. A central motivation is that synthetic `(query, seed passage)` pairs are often suboptimal because the seed passage is not necessarily the globally best answer passage for the generated query.

## Method

- **Backbone and stages**: Gecko starts from a pretrained transformer language model with `1.2B` parameters, then applies pre-finetuning followed by task-unified fine-tuning.
- **Pre-finetuning representations**: for text-pair data `D_pre = {(q_i, p_i)}`, the model encodes `q_i = mean_pool(M(t || q_i))` and `p_i = mean_pool(M(p_i))`, where `t` is a dataset-specific task prefix such as question answering or search result.
- **Pre-finetuning loss**: optimize in-batch contrastive learning `L_pre = -(1/B) * sum_i log exp(sim(q_i, p_i)/tau) / sum_j exp(sim(q_i, p_j)/tau)` with cosine similarity, no explicit hard negatives, and the largest batch that fits on device.
- **Step 1, query generation**: sample a web passage `p_seed` and use a few-shot prompted LLM to generate a task description and query, `LLM(P_QG, p_seed) -> (t, q)`, so the synthetic data spans QA, search result, fact checking, and sentence similarity behaviors.
- **Step 2, positive/negative mining**: an initial embedding model retrieves top-`N` neighbors for each generated query; the same LLM reranks them using query likelihood and relevance classification prompts.
- **Rank fusion**: the paper combines the two LLM rankings with `R(q,p) = 1 / r_QL(q,p) + 1 / r_RC(q,p)` and chooses a relabeled positive `p+ = argmax_p R(q,p) = p_1`; hard negatives are selected either as `p_N` or from remaining neighbors.
- **FRet dataset**: this process creates `6.6M` quadruples `(task, query, positive, negative)`, and the paper reports that positive relabeling `p+ != p_seed` happens in about `15%` of examples.
- **Unified fine-tuning mixture**: FRet is mixed with [[natural-questions]], [[hotpotqa]], [[fever]], MedMCQA, [[snli]], [[mnli]], classification datasets from HuggingFace, and MIRACL for the multilingual model, all normalized into a shared task-query-target format.
- **Fine-tuning objective**: Gecko optimizes an in-batch cross-entropy objective where each query separates `p_i+` from `p_i-`, other positives in the batch, and other queries as same-tower negatives; for classification triples, the paper appends unique IDs to avoid false negatives.
- **Flexible dimensions**: the model adds [[matryoshka-representation-learning]] loss so one encoder supports `d = 768` and `d = 256` embeddings from the same trained model.

## Key Results

- On MTEB, `gecko-1b-256` scores `64.37` average, beating `text-embedding-3-large-256` at `62.00` despite the latter using OpenAI's stronger proprietary system.
- On MTEB, `gecko-1b-768` scores `66.31` average with `81.17` classification, `55.70` retrieval, `85.06` STS, and `32.63` summarization, outperforming `text-embedding-3-large` (`64.59`) and matching or exceeding several `7B`-scale baselines.
- A zero-shot model trained only on FRet still reaches `62.64` average on MTEB, suggesting the synthetic pipeline alone transfers substantial general-purpose embedding behavior.
- For multilingual retrieval, `gecko-multilingual-1b` reaches `56.2` average nDCG@10 on MIRACL, higher than `text-embedding-3-large` (`54.9`) and `mContriever-X` (`55.4`).
- Data mixture matters: the baseline recipe scores `58.42`, FRet-only reaches `62.64`, and the full mixture reaches `66.31`; removing the unified formatting drops a `300K` FRet setting from `60.87` to `57.45`.
- LLM relabeling improves quality: on FRet-based training, using `p_1` as positive and `p_20` as hard negative yields `53.39` BEIR nDCG@10 and `83.14` STS, compared with `52.33` and `82.66` when keeping `p_seed` and no hard negative.

## Limitations

- The paper omits several recipe details in the main text, including the concrete temperature `tau`, batch sizes, and exact neighbor count `N`, so exact reproduction still requires additional implementation choices.
- FRet is generated only in English; the multilingual result relies on adding MIRACL training data rather than showing that the English synthetic data alone is sufficient cross-lingually.
- Evaluation is centered on MTEB and MIRACL, which is strong evidence for general embeddings but leaves long-document retrieval, production latency, and domain-specific robustness less explored.
- The synthetic-data pipeline depends on repeated LLM inference and prompt design, but the paper does not analyze generation cost, annotation bias, or failure cases in depth.
