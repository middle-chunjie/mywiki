## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

The paper proposes two visual-programming frameworks for text-to-image generation and evaluation. `VPGen` turns generation into an explicit three-step program: predict object counts, predict a discrete bounding-box layout, and then render the image with a layout-conditioned diffusion model. `VPEval` similarly turns evaluation into a program that dispatches specialized visual modules for object presence, counting, spatial relations, scale, and text rendering, returning both scores and explanations. The core claim is that modular intermediate structure improves controllability and interpretability compared with end-to-end text-to-image systems. Empirically, VPGen substantially improves count, spatial, and scale accuracy on skill-based prompts, while VPEval correlates better with human judgments than single-model metrics such as CLIP, captioning, or VQA alone.

## Problem & Motivation

Existing text-to-image systems are strong at producing visually plausible single objects, but they often fail on prompt elements that require explicit structure: object counts, spatial relations, relative scale, and rendered text. At the same time, common automatic evaluation metrics are typically single-model scores whose reasoning is opaque and whose reliability varies by skill. The paper addresses both issues by introducing executable visual programs: one for generation with explicit layout control, and one for evaluation with modular, explainable scoring.

## Method

- **VPGen decomposition**: generation is split into `3` stages: `(1)` object/count generation, `(2)` layout generation, and `(3)` image generation.
- **Textual scene representation**: the first stage linearizes a scene as object-count pairs such as `obj1 (#obj1) obj2 (#obj2)`, keeping the interface language-native for the LM.
- **Bounding-box layout format**: the second stage predicts `xyxy` boxes with coordinates normalized to `[0,1]` and quantized into `100` bins, so one object can be emitted as `obj (xmin,ymin,xmax,ymax)` with each coordinate in `{0,...,99}`.
- **Why boxes instead of masks**: the paper argues that boxes are token-efficient; an `xyxy` box needs `4` tokens, whereas a `64 x 64` segmentation map would require `4096` tokens.
- **Layout LM**: the layout-aware generator is `Vicuna 13B`, fine-tuned with `LoRA` rather than full-model updates to preserve pretrained knowledge and reduce memory cost.
- **Training data**: the LM is trained on `1.2M` text-layout examples aggregated from [[flickr30k-entities]], `MS COCO Instances 2014`, and [[paintskills]], with `2000` validation examples; the maximum count for a single object class is capped at `7`.
- **Training setup**: per-GPU batch size is `96 = 24 x 4` gradient accumulation; the `Flickr30K+COCO+PaintSkills` model trains for `2` epochs, while the `Flickr30K`-only model trains for `6` epochs; reported training time is `26` hours on `4 x A6000` GPUs.
- **Layout-to-image renderer**: the final stage uses [[gligen]], a layout-conditioned diffusion model that consumes the prompt plus region descriptions and boxes.
- **GLIGEN inference hyperparameters**: the default configuration uses `gligen_scheduled_sampling_beta = 0.3`, `num_inference_steps = 50`, and `fp16`; end-to-end generation takes about `6s` per image on one `A6000` (`2s` for Vicuna and `4s` for GLIGEN).
- **VPEval skills**: evaluation is factorized into `5` skills: object, count, spatial, scale, and text rendering.
- **VPEval modules**: `objDet` uses [[grounding-dino]] plus `DPT` depth, `ocr` uses [[easyocr]], and `vqa` uses [[blip-2]]; higher-level evaluators such as `countEval`, `spatialEval`, `scaleEval`, and `textEval` are built from these primitives.
- **Open-ended evaluation programs**: for prompts requiring multiple criteria, the system uses [[gpt-3-5]] with in-context examples to generate an evaluation program dynamically, then averages module outputs while retaining visual and textual explanations.

## Key Results

- On skill-based prompts, `VPGen (F30+C+P)` reaches `51.0%` average VPEval accuracy, outperforming `Stable Diffusion v1.4` (`37.7%`), `Stable Diffusion v2.1` (`40.6%`), and `Karlo` (`40.8%`).
- The same model substantially improves controllability-sensitive skills: `Count 72.2%`, `Spatial 56.1%`, and `Scale 26.3%`, versus the best baselines of `59.5%`, `31.3%`, and `16.4%`, respectively.
- The paper reports explicit gains over strong baselines of `+12.7` points on Count versus Karlo, `+24.8` on Spatial versus Stable Diffusion v2.1, and `+9.9` on Scale versus Karlo.
- On open-ended `TIFA160` prompts, `VPGen (F30)` scores `71.0%`, close to `Stable Diffusion v1.4` (`70.6%`) and below `Stable Diffusion v2.1` (`72.0%`); on the more layout-sensitive PaintSkills subset it scores `71.0%` versus `63.5%` (`SD v1.4`) and `68.4%` (`SD v2.1`).
- Bounding-box quantization at `100` bins is sufficient: `100` bins gives `51.0%` skill-based / `68.3%` open-ended accuracy, slightly better than `1000` bins (`49.8%` / `68.2%`).
- Replacing the trained Vicuna layout model with `GPT-3.5-Turbo + 36` in-context examples hurts VPGen performance: `40.7%` skill-based and `65.5%` open-ended versus `51.0%` and `68.3%` for `Vicuna 13B + GLIGEN`.
- For skill-based human correlation, VPEval reaches overall Spearman `rho = 73.5`, better than `CLIP` (`20.4`), captioning metrics, and `BLIP-2 VQA` (`65.0`); the `VPEval dagger` variant reaches `79.0`.
- For open-ended human correlation, VPEval scores `rho = 56.9`, slightly above `TIFA (BLIP-2)` at `55.9`, and `VPEval dagger` further improves to `60.3`; human annotator agreement is high with `kappa = 0.85` and `alpha = 0.85`.

## Limitations

- VPGen's gains depend on the downstream layout-to-image model; the paper explicitly notes cases where [[gligen]] fails even when the generated layout is correct.
- Text rendering remains weak for all systems, including VPGen: the best reported text-rendering score in Table 1 is only `8.9%`, and `VPGen (F30+C+P)` gets `3.7%`.
- The LM and visual modules are trained mostly on English and natural-image data, so cross-lingual prompts and non-photographic domains are likely weaker.
- Open-ended program generation with [[gpt-3-5]] introduces API cost and external-model dependence, even though the paper later releases generated programs and a distilled local LM.
- Evaluation quality is bounded by module quality: object detection, OCR, VQA, and depth estimation errors can all propagate into VPEval scores.
