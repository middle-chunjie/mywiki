## Summary

⚠ Possibly outdated: published 2022; re-verify against recent literature.

This survey organizes the emerging literature on pre-trained models for source code (CodePTMs) into a comparative framework covering `20` models, `18` downstream software-engineering tasks, benchmark datasets, and evaluation metrics. The paper argues that source code should not be treated as plain natural language because it mixes token sequences, natural-language comments, and explicit structures such as ASTs and data-flow graphs. It classifies CodePTMs along four axes: architecture, input modality, pre-training objective, and programming-language coverage; then relates those choices to observed task performance. The main synthesis is that pre-training consistently improves software-engineering tasks, but gains depend on matching model architecture, modality, and pre-training task type to the downstream workload.

## Problem & Motivation

Pre-trained language models had already transformed NLP, but it was unclear how well that recipe transferred to software engineering, where data is multimodal and strongly structured. The authors aim to give AI and SE researchers a unified view of CodePTMs, the tasks they target, the datasets they use, and the kinds of inductive biases that matter. A survey was needed because the area had grown quickly, individual papers used incompatible task setups, and many researchers outside software engineering were unaware that code-specific pre-training had already produced strong results across understanding and generation workloads.

## Method

- **Survey scope**: review `20` CodePTMs and relate them to `18` downstream SE tasks spanning understanding and generation, with task I/O types over `code`, `natural language`, and predicted `values`.
- **Architecture taxonomy**: classify models into `4` families: `LSTM`, `Transformer-Encoder (TE)`, `Transformer-Decoder (TD)`, and full `Transformer (TF)` / [[encoder-decoder-architecture]]. The survey explicitly argues that encoder-only and decoder-only models are mismatched for some seq2seq and classification settings.
- **Modality taxonomy**: group models into `Uni`, `Bi`, and `Multi` according to whether they consume code only, code plus one extra modality, or code with multiple auxiliary modalities such as NL, [[abstract-syntax-tree]], and [[data-flow-graph]].
- **Cross-modal fusion**: distinguish `Together` vs `Standalone` input construction. `Together` concatenates modalities into one training instance to encourage cross-modal representation learning; `Standalone` uses separate instances per modality.
- **Pre-training objective taxonomy**: separate NLP-derived objectives into `LM`, [[masked-language-modeling]], [[denoising-autoencoding]], and contrastive learning, and code-specific objectives into code-aware, structure-aware, and cross-modal-aware tasks with subtypes `CN`, `CS`, and `CNS`.
- **Programming-language coverage**: compare `Mono` vs `Multi` pre-training and summarize the main corpora, including CodeSearchNet with `>6.4M` functions, BigQuery mined from `>3M` repositories, and CLCDSA with `26K` functions.
- **Quantitative synthesis**: aggregate best published CodePTM and non-CodePTM numbers per benchmark and compute relative error reduction as `` `ER = (error_non_codeptm - error_codeptm) / error_non_codeptm × 100%` ``.

## Key Results

- Across the benchmarks summarized in Table 5, the best result on every listed dataset is achieved by a CodePTM, with relative error reduction ranging from `0.9%` to `78.7%`.
- On function-docstring mismatch (`K1`), CuBERT reaches `98.0` accuracy versus `91.0` for the best non-CodePTM baseline, corresponding to `78.7%` relative error reduction.
- On variable-misuse localization and repair (`V1`), CuBERT obtains `95.2` accuracy versus `80.5`, for `75.4%` relative error reduction.
- On code search over CodeSearchNet (`C2`), SynCoBERT reaches `74.0` MRR versus `41.9` for the best non-CodePTM baseline, a `55.2%` relative error reduction.
- On Java-to-C# / C#-to-Java code translation (`C4`), CodeT5 reports `66.4` versus `35.4`, yielding `47.9%` relative error reduction.
- Gains are uneven: some generation benchmarks improve only modestly, such as `0.9%` on `H2` code summarization and `2.4%` on `A1` code completion, showing that not every task benefits equally from pre-training.

## Limitations

- The survey is a `2022` snapshot and predates the current code-LLM era, instruction tuning, retrieval-augmented coding agents, and many larger multilingual code models.
- The quantitative comparison compiles best numbers from different papers rather than rerunning all models in a controlled setting, so causal claims remain limited by inconsistent preprocessing, training budgets, and evaluation protocols.
- Table 3 gives a high-level mapping from models to tasks, but many confounders remain entangled, including corpus size, modality choice, and pre-training objective composition.
- Several forward-looking claims, such as code-specific tokenization, more efficient adaptation, and unified evaluation, are research suggestions rather than experimentally validated conclusions inside this paper.
