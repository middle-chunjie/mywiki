## Summary

The paper proposes Counterfactual Analysis for Programming Concept Predicates (CACP), a black-box evaluation framework for testing whether large code models actually understand core programming concepts rather than merely solving end tasks by pattern matching. CACP generates semantics-preserving counterfactual programs that perturb one concept at a time, then measures whether code-completion accuracy changes under the perturbation. The framework targets four predicates: control flow, data flow, data types, and identifier naming, instantiating them with five concrete mutations such as if-else flipping and def-use breaking. Across ten code models and three Python benchmarks, the authors report Average Mutation Effects as high as `34.07%`, indicating that even strong code LLMs remain brittle to concept-level perturbations despite good baseline completion accuracy.

## Problem & Motivation

Standard code-generation benchmarks such as HumanEval, MBPP, and CodeContests evaluate end-to-end task success, but they do not isolate which underlying programming concepts a model does or does not understand. The authors argue that this makes model failures hard to attribute: a bad completion could stem from weak handling of control flow, variable scope, naming, or type constraints rather than from generic lack of coding ability. To obtain a more fine-grained diagnosis, they formulate evaluation around Programming Concept Predicates (PCPs) and ask whether a model's output changes appropriately when only one predicate is perturbed through a controlled counterfactual transformation.

## Method

- **Framework setup**: define a code model `M: H x X -> Y`, where `H` is the natural-language instruction space, `X` is the partial-program input, and `Y` is the generated completion; correctness is scored by an attribution function `A(h, x, y) in {0,1}` based on unit-test success.
- **Counterfactual objective**: generate a mutated program `x'` that changes only one Programming Concept Predicate while preserving task validity, so differences in output quality can be attributed to that predicate rather than to unrelated syntax or semantics changes.
- **Target predicates**: CACP studies four PCP families: [[control-flow]], [[data-flow]], [[data-types]], and [[identifier-naming]].
- **Mutation 1, If-Else Flip**: negate the relational expression in an `if` condition using complementary operators and swap the then/else bodies; the mutation is accepted only when it preserves semantics and does not introduce side effects from method calls.
- **Mutation 2, Independent Swap**: use [[data-flow-analysis]] to identify independent statement blocks and swap them, probing whether code completion is invariant to statement order when no dependency exists.
- **Mutation 3, Def-Use Break**: rename the second definition-use chain of a variable and all downstream uses with a random identifier of length `5`, creating a semantics-preserving perturbation that tests scope and liveness sensitivity.
- **Mutation 4 and 5, Variable-Name Invariance**: rename user-defined variables either with random `5`-character strings or by shuffling names among variables; function parameters are excluded to avoid changing prompt semantics.
- **Prompt construction for completion**: mutated and original reference solutions are cut at matched positions to form partial programs. For Independent Swap, Def-Use Break, and Variable-Name mutations, the framework keeps the first `75%` of the program and removes the trailing `25%`; for If-Else Flip, both variants are cut at the start of the then body.
- **Validity checks**: every mutated full program is compiled and executed against the benchmark test suite so only semantics-preserving counterfactuals are retained.
- **Effect metric**: for a predicate `p_k`, the mutation effect is `ME^M_(p_k,h,x) = |A(h, x_(p_k), M(h, x_(p_k))) - A(h, x, M(h, x))|`; the dataset-level score is `AME^M_(p_k) = E_[h,x][ME^M_(p_k,h,x)]`, where lower AME indicates better predicate understanding.
- **Experimental setup**: evaluate `10` popular code models on Python tasks from HumanEval, MBPP, and CodeContests with temperature `0`, using unit-test correctness rather than string-match metrics such as CodeBLEU or chrF.

## Key Results

- CACP yields `1,724` valid counterfactual pairs for variable-name randomization, `1,724` for variable-name shuffling, `1,103` for if-else flip, `1,624` for independent swap, and only `299` for def-use break.
- On HumanEval+MBPP, original completion accuracy ranges from `43.20%` for Llama 2 `7B` to `72.65%` for Code Llama `34B`, but AME remains large: up to `29.08%` for variable-name shuffle and `25.18%` for if-else flip.
- On CodeContests, the largest observed degradation is `34.07%` for Llama 2 `7B` under def-use break; if-else flip reaches `32.53%` for Code Llama `13B` and `32.02%` for Code Llama `34B`.
- Independent Swap is consistently the easiest perturbation, with AME often near `5%` to `9%`, substantially lower than name-based or control-flow mutations.
- Larger models usually have lower AME than smaller ones for variable-name mutations, and code-specialized models such as StarCoder, Code Llama, and PaLM `2-S*` tend to outperform general LLM counterparts on these predicates.
- In the memorization analysis for StarCoder, replacing more frequent relational operators causes larger performance drops: `== -> !=` yields `13.21%` change versus `!= -> ==` at `7.37%`, and `> -> <=` yields `16.92%` versus `1.48%` in the reverse direction.

## Limitations

- The evaluation only covers the [[code-completion]] task, so the conclusions do not automatically transfer to repair, synthesis, translation, or multi-file coding settings.
- Only four predicate families and five concrete mutations are instantiated; concept coverage is therefore incomplete even within classical [[program-semantics]] and program-analysis literature.
- Def-use perturbations are data sparse, with just `22` valid HumanEval+MBPP cases and `299` total, making those estimates less stable than the other mutations.
- The study is limited to Python benchmarks and assumes access to executable unit tests, which constrains applicability to datasets without reliable test-based attribution.
- Some observed effects may still mix true reasoning failures with memorization artifacts from training-data frequency, as the relational-operator analysis suggests.
