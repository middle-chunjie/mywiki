## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

The paper proposes Structured Chain-of-Thought Prompting (SCoT prompting) for code generation. Instead of asking a model to emit free-form natural-language reasoning, it first asks the model to produce a structured intermediate plan built from program structures such as sequence, branch, loop, plus an explicit input-output specification, and only then generate code. The method is implemented as a two-stage prompt pipeline with three demonstrations per stage and evaluated on ChatGPT and Codex across HumanEval, MBPP, and MBCPP. Relative to standard CoT prompting, SCoT improves Pass@1 by up to `13.79%` on HumanEval, `12.31%` on MBPP, and `6.63%` on MBCPP, while also improving human-rated correctness, code smell, and maintainability.

## Problem & Motivation

Standard chain-of-thought prompting was developed for natural-language reasoning and describes solutions as free-form textual steps. The authors argue that this is a poor fit for code generation because source code is inherently structured and relies on explicit control flow and well-specified inputs and outputs. As a result, conventional CoT can leave iteration scopes, branch conditions, and output formats ambiguous, which weakens downstream code synthesis. The paper's central hypothesis is that intermediate reasoning for code should resemble programming structure rather than ordinary prose, so prompting should make models reason "toward code" before they generate the final program.

## Method

- **Structured CoT representation**: replace free-form CoT with a structured plan containing `sequence`, `branch`, and `loop` constructs, plus an explicit input-output block that specifies parameter names, parameter types, and expected return values.
- **Program-structure basis**: the method assumes any program can be decomposed into three basic structures, so SCoT constrains intermediate steps to use forms such as `if ...`, `if ... else`, `if ... elif ... else`, `for`, and `while`.
- **Two-stage prompting pipeline**: stage 1 prompts the LLM to generate a SCoT from `<requirement, SCoT>` examples; stage 2 prompts the LLM to generate code from `<requirement, SCoT, code>` examples.
- **Prompt construction**: each prompt contains `3` manually prepared demonstrations. The seed examples are sampled from benchmark training sets, then annotators manually write SCoTs to form the prompt exemplars.
- **Noise mitigation**: the second-stage prompt explicitly asks the model to double-check the generated SCoT and repair missing or incorrect steps before implementing code, reducing error accumulation from stage 1.
- **Base models**: evaluate one instruction-tuned LLM, `ChatGPT (gpt-3.5-turbo-0301)`, and one code LM, `Codex (code-davinci-002)`, showing that the prompting method is model-agnostic.
- **Benchmarks**: evaluate on `HumanEval` (`164` Python problems), `MBPP` (`500` test problems in Python), and `MBCPP` (`435` C++ test problems), covering both Python and C++ function-level generation.
- **Sampling setup for baselines**: all methods sample `n = 20` programs per problem. Zero-shot and few-shot use `temperature = 0.8`, `top-p = 0.95`, and `max length = 300`; CoT uses the same sampling settings but `max length = 600`.
- **Sampling setup for SCoT**: stage 1 samples `20` SCoTs per problem with `temperature = 0.8`, `top-p = 0.95`, `max length = 300`; stage 2 deterministically generates code for each SCoT with `temperature = 0` and `max length = 300`, yielding `20` candidate programs total.
- **Evaluation metric**: report unbiased `Pass@k` for `k ∈ {1, 3, 5}` using `n = 20` samples, where `Pass@k = E[1 - C(n - c, k) / C(n, k)]` and `c` is the number of correct samples for a problem.
- **Ablation design**: isolate the contributions of `basic structures` and the `input-output structure` by removing them independently while keeping the two-stage pipeline fixed.

## Key Results

- On **ChatGPT + HumanEval**, SCoT reaches `60.64 / 73.53 / 77.32` for `Pass@1/3/5`, versus CoT's `53.29 / 69.76 / 75.52`, a relative `13.79%` gain on `Pass@1`.
- On **ChatGPT + MBPP**, SCoT reaches `46.98 / 55.31 / 58.36`, versus CoT's `41.83 / 51.04 / 54.57`, a relative `12.31%` gain on `Pass@1`.
- On **ChatGPT + MBCPP**, SCoT reaches `57.06 / 65.70 / 68.70`, versus CoT's `53.51 / 63.84 / 67.03`, a relative `6.63%` gain on `Pass@1`.
- On **Codex + HumanEval**, SCoT reaches `49.82 / 66.56 / 75.14`, versus CoT's `43.79 / 63.41 / 71.56`, a relative `13.77%` gain on `Pass@1`; on **MBPP**, the relative `Pass@1` gain is `7.38%`, and on **MBCPP** it is `5.57%`.
- In human evaluation with `10` developers and `800` reviewed programs, SCoT scores `1.412` on correctness, `1.869` on code smell, and `1.873` on maintainability, outperforming CoT's `1.225`, `1.689`, and `1.616`.
- Ablation shows removing **basic structures** drops ChatGPT HumanEval `Pass@1` from `60.64` to `55.67`, while removing the **input-output structure** drops it to `59.65`; similar degradations appear on MBPP and MBCPP.
- Robustness studies show SCoT remains strong across three different example seed sets and three different annotators, with HumanEval `Pass@1` staying in the `59.36` to `60.64` range.

## Limitations

- The method depends on manually written SCoT demonstrations, so deployment quality still hinges on prompt authoring effort and example selection.
- Evaluation is limited to `2` base models, `3` benchmarks, and mostly short function-level code generation; it does not test repository-scale or tool-integrated programming.
- The paper compares primarily against prompting baselines and explicitly does not run a head-to-head comparison with execution-based reranking methods such as CodeT, so the ceiling under stronger post-processing remains unclear.
- The chosen base models, `gpt-3.5-turbo-0301` and `code-davinci-002`, are now dated, so absolute performance numbers should not be treated as current frontier results.
- The authors acknowledge possible benchmark contamination in pretrained LLMs and argue only that relative comparisons across prompting methods remain fair under the same base model.
