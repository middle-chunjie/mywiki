## Summary

CodeFill is a Python code autocompletion model that jointly learns token names, token types, and statement-level continuations instead of treating source code purely as flat text. The pipeline first converts each file into aligned value and AST-type sequences, adds indentation markers, and applies BPE to identifier names; it then trains three GPT-2-style decoder-only Transformers with soft-parameter-sharing multi-task learning over next-token value prediction, next-token type prediction, and multi-token statement completion. A heuristic post-processing stage re-ranks top predictions using visible file/class/function scope. On deduplicated Python benchmarks, CodeFill improves over prior baselines including GPT-C and TravTrans+, with especially large gains on realistic cardinal-point completion and multi-token statement completion.

## Problem & Motivation

IDE autocompletion systems traditionally rely either on grammar- and type-driven rules or on NLP-style next-token models. The first class is brittle in dynamically typed languages such as Python because type information is often unavailable; the second class ignores program structure and evaluates prediction on arbitrary masked tokens rather than the syntactic positions where developers actually invoke completion. The paper aims to combine naming and structural signals, model longer-range dependencies beyond one-token prediction, and evaluate at more realistic completion points so that the learned model better matches actual IDE usage.

## Method

- **Pre-processing**: each Python file is stripped of comments and blank lines, then converted into aligned sequences of token values and token types derived from the Python AST/tokenizer pipeline. Imports and aliases are normalized into categories such as `MODULE`, `LIBRARY`, and `ALIAS`; literals are normalized to `STRING` and `NUMBER`.
- **Indentation-aware encoding**: because Python block structure is indentation-sensitive, the pipeline inserts explicit `INDENT` and `DEDENT` markers, plus `EOS` to mark the end of a statement for multi-token completion.
- **Subword naming model**: identifier strings are segmented with [[byte-pair-encoding]] to reduce the open-vocabulary problem while still allowing unseen names to be composed from sub-tokens.
- **Architecture**: CodeFill uses three task-specific GPT-2-style [[decoder-only-language-model]]s under soft parameter sharing, all based on stacked self-attention and feed-forward Transformer blocks. Token prediction follows autoregressive factorization `P(v_0, ..., v_N | c_0, ..., c_T) = ∏_i P(v_i | c_0, ..., c_T, v_0, ..., v_{i-1})`.
- **Transformer computation**: the paper writes the hidden-state stack as `h_0 = C W_e + W_p`, `h_l = transformer_block(h_{l-1})`, and `P(v_t) = softmax(h_L W_e^T)`, where `C` is the left context, `W_e` token embeddings, and `W_p` position embeddings.
- **Three learning tasks**: Token Value Prediction (TVP) predicts the next token value from left context; Token Type Prediction (TTP) predicts the next AST/token type sequence; [[statement-completion]] (SC) keeps generating token values until `EOS`, enabling multi-token code suggestion.
- **Multi-task training**: the model samples tasks per epoch with probabilities `0.2` (TTP), `0.4` (TVP), and `0.4` (SC). During pre-training it optimizes the shared loss `Loss_final = |min(Loss_TVP, Loss_TTP, Loss_SC)|`; during fine-tuning it keeps only TVP and SC because TTP converges quickly on Python grammar.
- **Decoding**: for TVP and SC, CodeFill uses [[beam-search]] with beam width `|B| = 5`, selected after trying `3`, `5`, and `10`.
- **Post-processing / ranking**: the top-10 identifier predictions are re-weighted by visible scope. For example, attribute access receives weights `1.625 / 1.250 / 1.125` for function/class/file scopes; variable names `1.625 / 1.125 / 1.500`; function names `1.125 / 1.625 / 1.500`.
- **Data and training configuration**: pre-training uses `PY1690K`; fine-tuning uses `90%` of `PY117K`; evaluation uses the held-out `10%` of `PY117K`. The implementation sets learning rate `1e-5`, maximum sequence length `2048`, and trains for `100` epochs on `2 x GTX 1080 Ti` GPUs.

## Key Results

- **Any-token completion (TLP-A)**: CodeFill reaches `80.6` accuracy and `81.7` MRR, beating GPT-C (`79.8` / `80.0`) and TravTrans+ (`78.9` / `79.4`).
- **Token-type prediction (TLP-B)**: overall accuracy / MRR is `83.8` / `87.2`; on identifiers specifically it reaches `54.4` accuracy and `56.0` MRR, outperforming TravTrans+ (`51.1` / `53.7`).
- **Leaf-node prediction (TLP-C)**: overall accuracy / MRR is `66.3` / `69.5`; for function names it reaches `72.2` accuracy and `76.9` MRR.
- **Cardinal-point completion (TLP-D)**: on syntactic points where developers are more likely to trigger autocompletion, CodeFill gets `70.0` accuracy and `70.9` MRR versus GPT-C `66.1` / `67.8` and TravTrans+ `65.0` / `66.2`.
- **Statement-level completion (SLP)**: for `n = 4` tokens, CodeFill achieves `70.2` METEOR and `63.8` ROUGE-L, ahead of TravTrans+ at `64.5` and `52.4`. The paper summarizes this as more than `2/3` of statement completions of length `<= 4` being correct.
- **Ablation**: moving from single-task GPT-2 (`77.7` accuracy, `78.2` MRR) to soft-parameter-sharing MTL with value/type/statement tasks raises performance to `80.6` / `81.7`, with training time increasing from `12h` to `24h`.
- **Runtime**: prediction latency is `73 ms`, within the paper's target of under `100 ms`, though the final model has `258M` parameters.

## Limitations

- The study is restricted to Python; the authors explicitly note that generalization to other programming languages still needs validation.
- The re-ranking layer depends on manually tuned scope weights rather than learned calibration, so part of the final gain comes from heuristic engineering.
- CodeFill is comparatively large (`258M` parameters), making client-side deployment impractical and pushing the system toward centralized serving.
- Statement-level evaluation uses ROUGE-L and METEOR rather than online IDE studies or human productivity measurements, so user-facing utility is inferred rather than directly measured.
- The datasets are large and deduplicated, but they still come from open-source GitHub repositories; industrial codebases and private dependency contexts are not evaluated.
