## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

CoCoMIC addresses a practical failure mode of code language models: next-token completion often depends on APIs, classes, and signatures defined in other files of the same repository, but standard autoregressive models only see the current file. The paper introduces CCFINDER, a static-analysis retriever that builds a project context graph and collects relevant `k`-hop neighbors from imported entities, then proposes CoCoMIC to encode each retrieved entity into a compact vector via a `[SUM]` token and jointly attend to cross-file and in-file context at every Transformer layer. On a PyPI-based Python benchmark with 6,888 held-out prompts, the method improves exact-match code completion from `15.97` to `21.39` and identifier-match exact match from `24.29` to `31.26`, while also lowering perplexity from `2.87` to `2.69`.

## Problem & Motivation

Modern code completion rarely depends only on local tokens: developers import project-specific classes, functions, and globals whose signatures are invisible from the current file. Prior code LMs such as CodeGen therefore hallucinate method names, argument lists, and member accesses when same-repository context is required. The paper argues that naively prepending entire repositories is not viable because project context is structurally heterogeneous, often private to the model's pretraining corpus, and far too long for the model's input budget. The target problem is thus twofold: retrieve the right cross-file context efficiently, then fuse it with the in-file prefix in a way that preserves local causal generation while exposing project-level dependencies.

## Method

- **CCFINDER graph construction**: parse Python projects into a multi-relational directed graph `G = (V, E)` whose nodes are files, classes, functions, and global variables, and whose edges encode intra-file scope relations plus inter-file import dependencies.
- **Retrieval procedure**: extract same-project import statements from the incomplete file, locate matching nodes through locale-based lookup, run depth-first search over `k`-hop neighbors, deduplicate nodes, then reorder entities to follow source-file code order. The paper uses `k = 2`.
- **Entity representation**: each retrieved cross-file entity is serialized as `c_i = (locale_i, w_i^1, ..., w_i^m, [SUM])`, where locale text is prepended as a comment such as `#git.listtags\n`. The hidden state of `[SUM]` is used as the entity embedding.
- **Cross-file compression**: encode each entity with the backbone LM as `h_{c_i} = f_theta(c_i) in R^{d_h}` and stack them as `H_C = (h_{c_1}, ..., h_{c_n})`, so long retrieved snippets are compressed into one vector per entity rather than concatenated verbatim.
- **Joint attention**: for in-file prefix `s_t = (x_1, ..., x_{t-1})`, compute `H_S(t) = f_theta(s_t)`, project both contexts into keys and values, concatenate them as `K(t) = concat(K_C, K_S(t))` and `V(t) = concat(V_C, V_S(t))`, then predict with `Attn(t) = softmax(Q(t)K(t)^T / sqrt(d_k))V(t)`.
- **Backbone and training**: use `CodeGen-350M-Mono`, fine-tune for `5` epochs with maximum sequence length `2048`, learning rate `5e-5`, `5%` warmup, and cosine annealing.
- **Retrieval budget**: keep at most `128` retrieved entities, each truncated to `128` tokens, letting CoCoMIC compress up to `16384` cross-file tokens into `128` vectors.
- **Data and implementation**: build the benchmark from `60,891` permissively licensed PyPI projects, split `80/10/10`, and evaluate on `6,888` statement-level held-out prompts that require local API usage. CCFINDER uses `tree-sitter` plus `import-dep` for parsing and import analysis.

## Key Results

- Main comparison: exact-match code completion rises from `15.97` for fine-tuned CodeGen to `21.39` for CoCoMIC, a relative gain of `33.94%`.
- Identifier-match exact match rises from `24.29` to `31.26`, a relative gain of `28.69%`; identifier precision/recall reach `55.45` / `57.83`.
- BLEU-4 improves from `35.11` to `41.65`, and perplexity drops from `2.87` to `2.69`, indicating no degradation on general next-token prediction.
- Retrieval quality: adding cross-file context raises identifier recall from `75.19%` (in-file only) to `95.55%`, meaning CCFINDER recovers `27.07%` more identifiers relevant to the target completion.
- Retrieval ablation: random entities score `15.68` EM, `1`-hop CCFINDER reaches `18.47`, and `2`-hop reaches `21.39`, supporting the chosen `k = 2`.
- Context encoding ablation: mean pooling over entity tokens yields `16.78` EM, far below `[SUM]`-based CoCoMIC at `21.39`.
- Relation ablation: locale annotations outperform both no-relations (`20.27` EM) and multi-task edge prediction (`20.01` EM), while MTL reaches `97.2%` edge-prediction accuracy without helping code completion much.

## Limitations

- The full pipeline is demonstrated only on Python projects; extension to other languages is argued conceptually rather than validated experimentally.
- The method assumes access to same-project source files. When cross-file context is unavailable, CoCoMIC is reported to perform `5-7%` worse than fine-tuned CodeGen.
- Experiments are expensive: each training job uses `8` A100 GPUs for roughly `50` hours, and the paper reports only one run, limiting robustness analysis.
- Retrieved context still depends on static import structure; the paper does not model dynamic runtime dependencies or third-party package selection.
- The work evaluates statement-level completion on a custom PyPI benchmark, so generalization to broader IDE workflows is still uncertain.
