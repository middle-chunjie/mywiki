## Summary

The paper proposes Synthetic Programming Elicitation and Compilation (SPEAC), a text-to-code pipeline for very low-resource programming languages where direct generation is unreliable because the target language is barely represented in pretraining. Instead of forcing the model to emit the target language directly, SPEAC first selects an LLM-friendly parent language and a compilable child subset, then repairs generated programs into the child language before compiling them to the target language. The case study targets the UCLID5 formal verification language with Python as the parent language and an object-oriented Python subset as the child language. A prototype called Eudoxus substantially improves parse rates over prompting, self-repair, and fine-tuning baselines while keeping repair costs far below LLM call latency.

## Problem & Motivation

Large language models are strong at code generation for high-resource languages such as Python and Java, but they often fail to produce syntactically valid programs for very low-resource programming languages (VLPLs) such as domain-specific or formal verification languages. Existing remedies such as grammar prompting, constrained decoding, retrieval, and fine-tuning either assume simpler grammars, linear generation, or enough target-language data to adapt the model. The paper argues for a different interface: let the model generate code in a language it already handles well, then use formal repair and compilation to bridge that output into the target VLPL.

## Method

- **SPEAC pipeline**: define a target language `T`, a parent language `P`, a child language `C subset P`, and a compiler `f: C -> T`; the LLM generates `p in P`, repairs produce `p' in C`, and compilation emits code in `T`.
- **Synthetic programming elicitation study**: run a setup/execute/analyze process over `33` handwritten UCLID5 tasks using `2` LLM subjects (`gpt-4-turbo-2024-04-09`, `gpt-3.5-turbo-0125`) and compare direct target-language prompts with imaginary host-language API prompts.
- **Parent/child language selection**: direct UCLID5 prompting yields `660` non-parsing attempts (`10` samples x `33` benchmarks x `2` models), while Python prompts are more consistent; the paper chooses `P = Python` and defines `C` as an object-oriented Python subset whose methods map closely to UCLID5 transition-system structure.
- **Minimal prompting**: initial prompt `q` asks for parent-language code under the child-language constraints, and repair prompt `q'` asks the model to replace each `??` hole with valid code.
- **Largest consistent subtree**: build an error-tolerant AST `A`, prune nodes that cannot belong to `C`, encode child-language static checks as soft clauses `S`, solve for the largest satisfiable subset `S'` with MAX-SMT, and replace nodes responsible for `S \\ S'` with holes.
- **Model-based repair**: use the satisfying MAX-SMT model to fill holes when constraints determine a value, e.g. infer `?? = integer` from usage constraints such as `is_Integer(??)`.
- **Implementation details**: the prototype Eudoxus uses `tree-sitter` for parsing and search, `Z3` for MAX-SMT solving, and allows up to `5` repair iterations before giving up.

## Key Results

- The abstract reports syntactically correct UCLID5 generation `84.8%` of the time for SPEAC versus `9.1%` for fine-tuned `gpt-3.5-turbo` and `12.1%` for `gpt-4-turbo` in-context learning.
- Table 1 reports parse rates of `28/33` for Eudoxus with GPT3t (`84.8%`) and `24/33` for Eudoxus with GPT4t (`72.7%`); the best listed baselines are `2/33` for fine-tuned GPT3t and `4/33` for three-shot GPT4t.
- Across unique textbook problems, the two Eudoxus variants solve `30/33` problems that compile, while all `11` baseline variants together solve only `6/33`.
- On semantic ratings over parsed programs, GPT4t-based Eudoxus scores `4` or `5` on `19/24` outputs and GPT3t-based Eudoxus scores `4` or `5` on `14/28`; overall, `33/52` Eudoxus outputs that compile are judged useful (`>= 4`).
- Repair is computationally cheap relative to model calls: GPT3t Eudoxus averages `0.9 s` repair time and `7.2 s` LLM time, while GPT4t Eudoxus averages `1.8 s` repair time and `35.1 s` LLM time.

## Limitations

- The evaluation is limited to a single target VLPL, UCLID5, so cross-domain generalization to other low-resource or formal languages remains untested.
- Only two proprietary LLMs are evaluated, making model-family robustness unclear.
- Semantic correctness is assessed manually by a single author, which introduces evaluator bias and low inter-rater reliability.
- Eudoxus still fails on some benchmarks because of iteration limits and implementation bugs in snippet extraction or pretty-printing.
- The paper contains an internal reporting mismatch: the abstract states a `9.1%` fine-tuned GPT3t baseline, while Table 1 lists `2/33`.
