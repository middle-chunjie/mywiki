## Summary

⚠ Possibly outdated: published 2022; re-verify against recent literature.

This paper introduces ALERT, a black-box adversarial attack for pre-trained code models that explicitly optimizes the naturalness of identifier substitutions instead of preserving only operational semantics. ALERT uses CodeBERT or GraphCodeBERT to propose subtoken replacements with masked language modeling, ranks them by contextual embedding cosine similarity, then searches for successful adversarial renamings with an Overall Importance Score guided greedy procedure followed by a genetic algorithm fallback. Across vulnerability detection, clone detection, and source-code authorship attribution, ALERT consistently outperforms MHM-style baselines while changing fewer variables and issuing fewer victim-model queries. The paper also shows that adversarial fine-tuning on ALERT-generated examples substantially improves model robustness.

## Problem & Motivation

Prior adversarial attacks on models of code mainly preserve operational semantics, for example by renaming variables without breaking execution, but they ignore whether the resulting code looks natural to human developers. The paper argues that this is a major flaw because software is consumed by both machines and humans: an identifier substitution that is executable but semantically awkward or stylistically implausible is unlikely to survive code review or be merged into a real codebase. The core motivation is therefore to generate adversarial examples that preserve both machine-oriented operational semantics and human-oriented natural semantics while remaining effective against strong pre-trained code models.

## Method

- **Victim models and attack surface**: ALERT attacks fine-tuned [[codebert]] and [[graphcodebert]] classifiers on vulnerability detection, clone detection, and [[source-code-authorship-attribution]], assuming black-box access to predicted labels and confidences.
- **Naturalness-aware substitution**: variable names are split with [[byte-pair-encoding]] into subtokens `T = <t_1, ..., t_m>`. For each masked subtoken, the underlying pre-trained model returns top candidates; in experiments ALERT generates `60` candidates per variable occurrence and keeps the top `30` variable-name reconstructions ranked by contextual similarity.
- **Semantic ranking**: candidate names are scored by cosine similarity between concatenated contextual embeddings of the original and substituted subtokens, i.e. similarity between `T` and `T'`, to favor human-plausible replacements rather than arbitrary rare identifiers.
- **Variable extraction and filtering**: ALERT uses [[tree-sitter]] to extract local variables defined and initialized within the snippet scope, unions candidate sets across occurrences, and filters duplicates, invalid identifiers, language keywords, and names colliding with field names.
- **Importance scoring**: for token position `i`, ALERT defines `IS_i = M(c)[y] - M(c^*_{-i})[y]`, where `c^*_{-i}` replaces the variable token with `<unk>`. Variable priority is `OIS_var = Σ_{i in var[pos]} IS_i`, aggregating importance across all occurrences of the same variable.
- **Greedy-Attack**: variables are sorted by decreasing `OIS_var`; for each variable, ALERT evaluates all natural substitutes and either stops when `M(c') != M(c)` or keeps the variant that minimizes the confidence on the ground-truth label before moving to the next variable.
- **GA-Attack**: if greedy search fails, ALERT treats a solution as a chromosome of variable-to-substitute pairs, uses fitness `fitness = M(T)[y] - M(T')[y]`, and evolves the population with crossover probability `r = 0.7`, `child_size = 64`, and `max_iter = max(5 * |vars|, 10)`.
- **Experimental setup**: fine-tuned victim models use maximal input length `512`; tasks cover `27,318` vulnerability-prediction functions from FFmpeg/Qemu, `90,102/4,000/4,000` train/dev/test clone examples from [[bigclonebench]], and `660` Python files for authorship attribution.

## Key Results

- **Human naturalness**: in a user study over `100` sampled C snippets with `4` non-author CS participants, ALERT-generated adversarial examples received an average naturalness score around `3.95/5`, while classic MHM examples averaged only `1.86/5`.
- **Attack success on CodeBERT**: ALERT reaches `53.62%` ASR on vulnerability detection, `27.79%` on clone detection, and `35.78%` on authorship attribution, improving over MHM-NS by `17.96`, `7.74`, and `16.51` percentage points respectively.
- **Attack success on GraphCodeBERT**: ALERT reaches `76.95%`, `7.96%`, and `61.47%` ASR on the same three tasks, outperforming MHM-NS by `21.78`, `4.54`, and `29.36` percentage points.
- **Greedy vs. full ALERT**: Greedy-Attack alone already achieves `49.42% / 23.20% / 30.28%` ASR on CodeBERT and `71.98% / 6.75% / 46.79%` on GraphCodeBERT; GA-Attack provides the remaining lift when greedy search stalls in local optima.
- **Scalability**: relative to MHM-NS, Greedy-Attack reduces the number of victim-model queries by `82.57%`, and full ALERT still reduces query count by `49.62%` while also modifying fewer variables than the baseline.
- **Adversarial fine-tuning**: training on ALERT-generated examples raises adversarial-set accuracy to `87.59%` for CodeBERT-Adv and `92.32%` for GraphCodeBERT-Adv overall; for example, GraphCodeBERT-Adv reaches `91.70%` on clone-detection adversarial examples and `96.21%` on authorship-attribution adversarial examples.

## Limitations

- The evaluation covers only two pre-trained code models and three downstream tasks, so the claims may not transfer to newer code LMs, generative models, or tasks beyond classification-style settings.
- ALERT relies on black-box access to confidence scores and still issues many victim-model queries, which may be unavailable or rate-limited in real deployments.
- The naturalness study is limited to `100` C snippets and `4` non-author participants, so human-judgment evidence is suggestive rather than definitive.
- The attack space is restricted to local-variable renaming in syntactically valid `C`, `Python`, and `Java`; the parser-based extraction pipeline does not cover languages such as Ruby.
- The method is sensitive to hyperparameters such as candidate-pool size, maximum iterations, and victim-model fine-tuning settings, which the authors explicitly note as a threat to validity.
