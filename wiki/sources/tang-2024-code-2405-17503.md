## Summary

The paper argues that iterative LLM code repair is not just a prompting trick but a search problem with a real exploration-exploitation tradeoff. It introduces REx, a refinement policy that treats each current program as an arm in an arm-acquiring bandit, uses testcase pass rate as a heuristic `h(rho)`, and applies Thompson Sampling over Beta posteriors to decide which partial program to refine next. The method is evaluated on competition programming, loop invariant synthesis, and ARC visual reasoning. Across these domains, REx consistently solves at least as many problems as greedy, breadth-first, and fixed-width baselines while using substantially fewer model calls, with the biggest gains on harder tasks and a particularly strong result on nonlinear loop invariants.

## Problem & Motivation

Prior LLM refinement pipelines usually expand candidate programs using simple greedy or breadth-first rules, even though refinement generates an unbounded tree whose branching and depth both matter. The paper's key claim is that code repair exposes a genuine exploration-exploitation problem: the system should exploit programs that already pass many checks, but it should also explore less-refined branches because stochastic LLM repairs can discover better trajectories from weaker intermediate programs. The authors therefore seek a policy that minimizes expensive LLM calls while still finding correct programs in domains with efficiently checkable specifications.

## Method

- **Specification and heuristic**: the program objective is to satisfy a decomposable specification `Phi = phi_1 /\ ... /\ phi_K`; the paper uses testcase or constraint pass rate as the progress heuristic `h(rho) = (sum_k 1[rho |= phi_k]) / K`.
- **Refinement operator**: refining a candidate program `rho` samples a counterexample `phi` from the failed constraints and prompts the LLM to generate `rho'`, formalized as `P_refine(rho' | rho, Phi) = E_{phi ~ U(CounterExamples(rho, Phi))}[P_LLM(rho' | prompt(rho, phi, Phi))]`.
- **Bandit formulation**: each current program is an arm; pulling an arm means refining that program once. The Bernoulli reward parameter is `theta_rho = E_{rho' ~ P_refine}[1[rho' |= Phi]]`, so success means the new program fully satisfies the specification.
- **Beta prior from heuristic quality**: REx uses `theta_rho ~ Beta(alpha_rho^prior, beta_rho^prior)` with `alpha_rho^prior = 1 + C * h(rho)` and `beta_rho^prior = 1 + C * (1 - h(rho))`. Larger `C` makes the policy greedier because it trusts the heuristic more strongly.
- **Posterior update and action selection**: after `N_rho` unsuccessful refinements, the posterior becomes `Beta(1 + C * h(rho), 1 + C * (1 - h(rho)) + N_rho)`. REx samples from each program's posterior and refines the program with the highest sampled value, which automatically discounts over-refined branches while preserving nonzero exploration probability.
- **Expected decay with failures**: the posterior mean is `E[theta_rho | N_rho] = (1 + C * h(rho)) / (2 + 2 * C + N_rho)`, giving a simple quantitative account of why repeated failed repairs should become less attractive.
- **Evaluation instantiations**: the paper tests REx on `38` nonlinear loop-invariant tasks, a `40`-problem ARC subset, and APPS programming subsets; APPS runs mostly use GPT-4 with `temperature = 1.0`, ARC uses GPT-4 Turbo, and loop-invariant checking uses Z3 with a `2` minute timeout.
- **Hyperparameters**: REx is evaluated over `C in {5, 10, 15, 20, 25, 30}` for standard budgets and also `C in {10, 25, 50, 100}` for the scaled ARC setting; the paper highlights `C = 20` as a strong default on difficult benchmarks.

## Key Results

- Across competition programming, loop invariant synthesis, and ARC, REx reduces LLM/API calls by roughly `1.5x-5x` relative to alternative refinement policies.
- In the speedup analysis, REx needs about `2x` less compute on APPS-Competition, `2-5x` less on nonlinear loop invariants, and `1.1x-1.6x` less on ARC.
- On nonlinear loop invariants, REx reaches `73.7%` solved versus `60.5%` for the prior specialized solver, despite not using dynamic traces, handcrafted features, or per-instance tuning.
- The method is reported as state of the art on APPS-Competition with GPT-4 and also outperforms the Hypothesis Search baseline on ARC.
- Hyperparameter sensitivity is lower than for greedy, BFS, and fixed-width baselines; large values such as `C = 20` perform well across all hard benchmarks studied.

## Limitations

- The final improvement in total solved problems is described as modest in the large-compute limit; much of the gain comes from lower cost rather than dramatic accuracy jumps.
- REx only uses testcase or constraint pass rate as its heuristic, leaving richer contextual signals and more advanced bandit variants unexplored.
- The formulation assumes access to an efficiently checkable specification, which fits unit tests, ARC training examples, and loop-verification conditions but does not directly cover open-ended coding tasks.
- The paper explicitly notes that solving the hardest APPS or ARC problems likely depends more on stronger base models than on further refinement-policy tuning alone.
