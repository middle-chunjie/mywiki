## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

This paper studies whether dense text embeddings can be inverted back into their source strings and introduces Vec2Text, an iterative reconstruction method that recasts inversion as controlled generation with embedding feedback. Starting from a conditional decoder, the system repeatedly generates a text hypothesis, re-embeds it with the target encoder, and conditions on the target vector `e`, hypothesis vector `ê^(t)`, and residual `e - ê^(t)` to make discrete corrections. Across GTR-base and OpenAI `text-embeddings-ada-002`, the method dramatically outperforms single-step baselines, reaching `97.3` BLEU and `92.0%` exact match on 32-token Wikipedia passages while also recovering `89.2%` of full names from embedded clinical notes. The core claim is that embeddings should be handled as sensitive data rather than privacy-preserving surrogates for text.

## Problem & Motivation

The paper asks whether hosted vector databases are actually safe when they store only dense embeddings rather than raw text. In the authors' threat model, an attacker has black-box query access to the same embedding model `φ` and wants to recover a text `x` from its embedding `e = φ(x)`. If exact or near-exact reconstruction is feasible, then embeddings leak private content almost as directly as the original documents, which is especially concerning for retrieval systems and medical notes.

## Method

- **Objective and threat model**: formulate inversion as `x̂ = argmax_x cos(φ(x), e)` under black-box access to an embedding model `φ`, where the attacker can query `φ(x̂)` but cannot inspect gradients or weights.
- **Base inversion model**: first train a conditional language model `p(x | e)` to reconstruct text directly from the embedding, amortizing the combinatorial search problem into model parameters.
- **Vec2Text correction loop**: iteratively refine a hypothesis with `p(x^(t+1) | e, x^(t), ê^(t))`, where `ê^(t) = φ(x^(t))`; the model conditions on the target embedding, the current hypothesis embedding, and the residual vector `e - ê^(t)` to decide the next correction.
- **Embedding projection**: map each embedding to a token-like sequence with `EmbToSeq(e) = W_2 σ(W_1 e)`, then concatenate `EmbToSeq(e)`, `EmbToSeq(ê^(t))`, `EmbToSeq(e - ê^(t))`, and hypothesis token embeddings, giving encoder input length `3s + n` with projection length `s = 16`.
- **Backbone and decoding**: initialize from `T5-base`; each inversion model has about `235M` parameters including the projection head. Token decoding is greedy, while sequence-level search keeps the top candidates by cosine similarity in embedding space; the paper's search study goes up to `50` correction rounds and sequence beam width `8`.
- **Training setup**: train the GTR-base inverter on `5M` Natural Questions Wikipedia passages truncated to `32` tokens, and two `text-embeddings-ada-002` inverters on MSMARCO with maxima of `32` and `128` tokens. Optimization uses Adam, learning rate `2e-4`, warmup plus linear decay, `100` epochs, batch size `128`, and `4 × NVIDIA A6000` GPUs; the slowest run takes about `2` days.

## Key Results

- **In-domain, GTR-base / Natural Questions (32 tokens)**: the base model gets `31.9` BLEU, `67` token F1, `0.0%` exact match, and `0.91` cosine similarity; Vec2Text with `50` steps plus sequence beam search reaches `97.3` BLEU, `99` token F1, `92.0%` exact match, and `0.99` cosine similarity.
- **In-domain, OpenAI `text-embeddings-ada-002` / MSMARCO (32 tokens)**: performance rises from `26.2` BLEU, `61` token F1, `0.0%` exact match to `83.4` BLEU, `96` token F1, and `60.9%` exact match with `50` steps plus sequence beam search.
- **Longer inputs, OpenAI `text-embeddings-ada-002` / MSMARCO (128 tokens)**: exact reconstruction is harder, but Vec2Text still improves from `17.0` to `55.0` BLEU and from `0.6%` to `8.0%` exact match, while cosine similarity rises from `0.95` to `0.99`.
- **Out-of-domain recovery on BEIR**: the shortest corpus, Quora, reaches `95.5` BLEU and `98.6` token F1, and the paper reports exact recovery for `66%` of Quora examples; across all `15` BEIR datasets, token F1 stays at least `41` and cosine similarity at least `0.95`.
- **Clinical privacy leakage on MIMIC**: on pseudo re-identified clinical notes, Vec2Text recovers `94.2%` of first names, `95.3%` of last names, `89.2%` of full names, and `26.0%` of notes exactly, versus only `10.8%` full-name recovery for the base model.
- **Noise defense trade-off**: adding Gaussian noise with `λ = 0.01` only lowers mean BEIR `NDCG@10` from `0.302` to `0.296` but collapses reconstruction BLEU from `80.372` to `10.334`; stronger noise such as `λ = 0.1` destroys both retrieval (`0.002`) and reconstruction (`0.148` BLEU).

## Limitations

- The defense experiment is narrow: the attacker is trained on clean embeddings and tested against simple Gaussian perturbations, so adaptive attack-defense dynamics remain open.
- Search is incomplete; the paper does not explore beyond `50` correction rounds or sequence beam width `8`, and exact recovery keeps improving as search gets deeper.
- Exact recovery is shown mainly for short inputs (`32` tokens), with only partial success on `128`-token sequences and no study of the much longer contexts used in production vector stores.
- The method assumes continued black-box access to the same embedding model and requires a new embedding query at every refinement step, which can make attacks costly or rate-limited in practice.
