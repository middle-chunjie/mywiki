## Summary

The paper introduces SWE-RL, a reinforcement-learning pipeline for improving open large language models on real-world software issue resolution using open-source pull-request histories instead of proprietary teacher traces or executable competitive-programming tasks. From GitHub events and `4.6M` cloned repositories, the authors build `24M` aggregated PR instances, filter them to about `11M` unique cases, and derive seed bug-fixing examples containing issue text, full-file code context, and oracle patches. A rule-based reward penalizes malformed edits and otherwise scores patch similarity, while GRPO trains Llama-3.3-70B-Instruct into Llama3-SWE-RL-70B. The resulting model reaches `41.0%` on SWE-bench Verified and, unexpectedly, improves on out-of-domain reasoning tasks including code reasoning, mathematics, and MMLU.

## Problem & Motivation

Prior software-engineering LLM pipelines largely depended on strong proprietary models, prompt engineering, or supervised distillation from closed models. Existing RL successes in coding mostly targeted competitive programming, where execution-based rewards are cheap and exact, but those assumptions break for repository-scale issue resolution. The paper therefore asks whether open-source software evolution records, including issues, pull requests, commits, and review discussions, can provide a scalable supervision signal for reinforcement learning on realistic bug-fixing tasks. A second motivation is generalization: the authors want to test whether RL on software issue solving improves broader reasoning ability rather than overfitting a narrow repair distribution.

## Method

- **Raw software-evolution corpus**: collect GitHub events from `2015-01-01` to `2024-08-31` via GHArchive and clone `4.6M` repositories with history, then aggregate PR-level context such as linked issues, discussions, review comments, initial code snapshots, intermediate commits, and cumulative patches into `24M` instances; after filtering, keep about `11M` unique PRs.
- **Decontamination and filtering**: remove repositories appearing in SWE-bench, bot-generated PRs, empty or extremely large changes, and PRs whose hunks are dominated by lock-file or version-bump style edits.
- **Context construction**: for each PR seed, use the issue description plus full contents of changed files and LLM-predicted relevant but unchanged files, specifically prompting Llama-3.1-70B-Instruct to supply additional relevant file paths so the policy learns to reason under noisy repository context.
- **Training target**: the policy model solves an issue by generating search/replace style code edits through reasoning, with the oracle target represented as the merged PR patch `patch_gt`.
- **Reward function**: define `R(tau) = -1` when the output format is invalid; otherwise `R(tau) = compare(patch_pred, patch_gt)`, where `compare` is Python `difflib.SequenceMatcher` and returns a continuous similarity score in `[0, 1]`.
- **Policy optimization**: train with [[group-relative-policy-optimization]] using normalized within-group rewards and a KL penalty, optimizing a clipped objective with group size `G = 16` rollouts per problem.
- **Training configuration**: continue from Llama-3.3-70B-Instruct for `1,600` steps with `16k` context, global batch size `512`, and `32` problems per batch with `16` sampled rollouts each; each global step performs one optimization step.
- **Inference scaffold**: use Agentless Mini, a simplified pipeline that does file-level localization, repair generation over full-file context, reproduction-test generation, regression-test filtering, and consensus reranking.
- **Reranking rule**: for consensus groups of patches `P` passing the same reproduction-test set `T`, score groups by `|P| x |T|^2`, prioritizing patches that pass more tests while still rewarding agreement across samples.
- **Main evaluation protocol**: on SWE-bench Verified, sample `500` repair candidates per issue at temperature `1.0`, keep the top `30` reproduction tests for reranking, and submit only the highest-ranked patch for pass@1 evaluation.

## Key Results

- `41.0%` pass@1 on SWE-bench Verified, which the paper reports as state of the art among medium-sized open models (`<100B`) and higher than Lingma-SWE-GPT-72B (`28.8%`), SWE-Fixer-72B (`30.2%`), SWE-Gym-32B (`32.0%`), and the paper's Llama3-Midtrain-8B beta (`31.0%`).
- In oracle-file repair-only evaluation, Llama3-SWE-RL-70B reaches `34.8` repair performance with `95.6%` correct formatting, versus `29.6` and `96.2%` for the SFT baseline, and `5.4` / `12.2%` for greedy base Llama-3.3-70B-Instruct.
- Majority voting helps the base model's format compliance (`44.6%`) and repair score (`16.6`), but it still remains far behind RL training.
- Scaling repair samples from `20` to `160` raises final resolve rate from `33.6%` to `40.0%`; further scaling to `320` and `500` only nudges the score to `41.0%`, indicating saturation.
- Increasing reproduction-test samples improves results from `38.8%` to `41.0%` up to about `20` tests, with no further gain from `20` to `30`.
- Out-of-domain performance improves rather than regresses: HumanEval+ `79.9` vs `76.2` base and `73.2` SFT; CRUXEval-I `71.6` vs `60.5` and `68.4`; MATH strict `73.7` vs `63.2` and `54.0`; MMLU `86.82` vs `86.49` and `85.26`.
- Reward ablation shows continuous similarity rewards outperform exact-match discrete rewards on repair (`34.8` vs `29.0`) while keeping similar format accuracy (`95.6%` vs `94.2%`).

## Limitations

- The reward compares lexical patch similarity rather than semantic equivalence, so functionally correct but textually different fixes may be under-rewarded.
- Agentless Mini uses a simplified file-level localization step, which can miss richer repository context and still relies on external scaffold structure.
- The pipeline decomposes issue solving into separate stages, so the model does not learn through interactive environment feedback during training.
- Strong performance depends on heavy inference-time sampling and reranking, which can be expensive in practice.
- The RL setup is trained only on issue-solving trajectories, so broader claims about generalized reasoning still rest on benchmark transfer rather than mechanistic evidence.
