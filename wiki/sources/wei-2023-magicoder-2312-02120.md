## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

The paper introduces Magicoder, a family of open-source code large language models built from OSS-INSTRUCT, a data-generation pipeline that asks a teacher model to synthesize self-contained programming problems and solutions from randomly sampled open-source code snippets. Instead of relying only on a small set of seed tasks or heuristic instruction rewrites, the method uses diverse real-world code as a controllable source of inspiration, then instruction-tunes CodeLlama-Python-7B and DeepSeek-Coder-Base-6.7B on about `75K` cleaned synthetic samples. A second stage with `110K` Evol-Instruct examples yields MagicoderS. Across HumanEval, HumanEval+, MBPP, MultiPL-E, and DS-1000, the resulting `7B` models outperform stronger open models and, on HumanEval+, MagicoderS-CL slightly exceeds ChatGPT under `pass@1`.

## Problem & Motivation

Prior open-source code LLM alignment pipelines often generated instruction data from a narrow prompt template, a tiny seed set, or a small number of handcrafted evolution heuristics. The authors argue that such synthetic corpora inherit teacher-model bias and under-cover realistic programming scenarios, which limits downstream code-generation quality.

Magicoder addresses this by grounding instruction synthesis in abundant open-source code. Random seed snippets provide controllability and diversity without directly turning raw code into training targets. The paper's hypothesis is that real code can inspire more realistic and less biased programming tasks, while still letting a strong teacher model generate coherent problem-solution pairs for instruction tuning.

## Method

- **Seed corpus**: OSS-INSTRUCT samples snippets from `starcoderdata`, a filtered variant of [[the-stack]], using `80K` source documents total: `40K` Python and `5K` each from C++, Java, TypeScript, Shell, C#, Rust, PHP, and Swift.
- **Snippet sampling**: from each document, the pipeline randomly extracts `1-15` consecutive lines and injects them into a prompt that asks the teacher to emit two sections: `[Problem Description]` and `[Solution]`.
- **Teacher model**: data generation uses [[gpt-3-5-turbo]] (`gpt-3.5-turbo-1106`) with greedy decoding, chosen for cost-effectiveness and for consistent problem-solution pairing.
- **Cleaning and decontamination**: the authors remove duplicates and benchmark overlaps against HumanEval, MBPP, APPS, DS-1000, and GSM8K; only `9` extra samples are filtered after benchmark decontamination, leaving about `75K` OSS-INSTRUCT examples.
- **Category analysis**: dataset diversity is analyzed with INSTRUCTOR embeddings over `10` manually designed coding categories, plus TF-IDF similarity against HumanEval to show lower benchmark-style imitation than Self-Instruct or open Evol-Instruct reproductions.
- **Base models**: Magicoder first fine-tunes [[codellama-python]]-`7B`; the same recipe is later applied to [[deepseek-coder]]-Base `6.7B` to obtain Magicoder-DS.
- **Training setup**: finetuning runs for `2` epochs on `2 x NVIDIA A100-80GB` with DDP, Adafactor, batch size `512`, initial learning rate `5e-5`, `15` warmup steps, a linear scheduler, and truncation length `1216`.
- **Second-stage tuning**: MagicoderS continues finetuning from Magicoder using open `evol-codealpaca-v1` with about `110K` samples; the setup stays the same except the max sequence length is `1024`.
- **Evaluation protocol**: Python benchmarks use greedy decoding and `pass@1` via [[evalplus]], while MultiPL-E and DS-1000 use sampling settings aligned with prior WizardCoder evaluations.
- **Ablation logic**: the paper separately studies Python-only vs non-Python OSS-INSTRUCT subsets and compares OSS-INSTRUCT against direct finetuning on `75K` mined comment-function pairs from the same open-source corpus.

## Key Results

- On HumanEval / HumanEval+, Magicoder-CL-`7B` reaches `60.4 / 55.5` `pass@1`, up from CodeLlama-Python-`7B` at `37.8 / 34.1`.
- MagicoderS-CL-`7B` reaches `70.7 / 66.5` on HumanEval / HumanEval+, compared with ChatGPT at `72.6 / 65.9`; it is slightly worse on HumanEval but better on the more rigorous HumanEval+.
- On MBPP / MBPP+, MagicoderS-CL-`7B` reaches `68.4 / 56.6`, versus Magicoder-CL at `64.2 / 52.6` and CodeLlama-Python-`7B` at `57.6 / 45.4`.
- On MultiPL-E, MagicoderS-CL-`7B` achieves `42.9` (Java), `57.5` (JavaScript), `44.4` (C++), `47.6` (PHP), `44.1` (Swift), and `40.3` (Rust), consistently beating Magicoder-CL and approaching WizardCoder-CL-`34B`.
- On DS-1000, MagicoderS-CL-`7B` scores `37.5`, versus `29.9` for Magicoder-CL, `28.4` for WizardCoder-CL-`7B`, and `29.2` for WizardCoder-SC-`15B`.
- On DeepSeek-Coder-Base-`6.7B`, MagicoderS-DS reaches `76.8 / 70.7` on HumanEval / HumanEval+ and beats DeepSeek-Coder-Instruct-`6.7B` on all reported benchmarks with about `8x` fewer finetuning tokens.
- In the data-source ablation, training on only non-Python OSS-INSTRUCT still improves HumanEval+ from `34.1` to `44.5`, suggesting nontrivial cross-language transfer.
- Direct finetuning on `75K` comment-function pairs hurts the base model on MultiPL-E (`29.6` to `24.1`) and leaves HumanEval+ unchanged at `34.1`, while OSS-INSTRUCT raises the same metrics to `55.5` and `37.8`.

## Limitations

- The comparison with [[deepseek-coder]] is necessarily partial because the paper states that DeepSeek-Coder's technical details and instruction-data construction were not yet publicly disclosed.
- The strongest results are still concentrated on code-generation benchmarks; the paper does not test broader software-engineering settings such as repository-level repair, agentic coding, or long-horizon issue resolution.
- The synthetic corpus is generated by a single teacher model and intentionally keeps some noisy samples, including potentially incomplete solutions, so data quality is not fully controlled.
- Most experiments center on `7B` or `6.7B` backbones; scaling behavior to larger base models is proposed as future work rather than demonstrated.
- The argument against direct open-source finetuning is evaluated only through comment-function pairs, so it does not exhaust all possible ways of using raw code supervision.
