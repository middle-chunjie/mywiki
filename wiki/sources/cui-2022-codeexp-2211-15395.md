## Summary

⚠ Possibly outdated: published 2022; re-verify against recent literature.

This paper introduces code explanation generation, a task aimed at producing long-form docstrings that cover both global function intent and local implementation behavior. It builds the CodeExp resource with three partitions: `2,285,387` raw GitHub code-docstring pairs, `158,024` refined pairs selected by a learned BERT filter, and `13,186` human-annotated examples for supervision and evaluation. The modeling study frames the task as [[sequence-to-sequence]] generation and compares GPT-2, GPT-Neo, CodeT5, and zero-shot [[codex]] under raw-only, refined-only, and two-stage raw-then-refined [[fine-tuning]]. The main finding is that data quality dominates raw scale: refined data markedly improves generated explanations, and CodeT5 fine-tuned on the refined split reaches near-human performance while BLEU and METEOR align best with human judgments.

## Problem & Motivation

Existing code summarization systems mostly generate short high-level descriptions, but developers often need explanatory docstrings that spell out parameters, return values, control-flow behavior, and design rationale. The paper argues that this gap matters for software maintenance, education, and verification of machine-generated code. Building such systems is hard because large public code-comment corpora contain many short, stale, or misleading comments, and the field lacks both high-quality training data and evaluation protocols that reflect explanatory adequacy rather than surface-level summary quality.

## Method

- **Task formulation**: define [[code-explanation-generation]] as mapping a function's signature and body to an explanatory docstring, and cast it as a [[sequence-to-sequence]] generation problem.
- **Annotated seed set**: start from code-doc-corpus and keep examples with `6-30` code lines, docstrings longer than `3` lines, and cyclomatic complexity `> 3`, yielding `13,186` candidate pairs for [[human-evaluation]].
- **Annotation protocol**: collect `0-3` scores for general adequacy, coverage of outer `if/else` or `try/except` blocks, and coherence of parameter / return descriptions; branch-level span alignments are also marked.
- **Raw corpus construction**: crawl roughly `55,000` GitHub repositories with more than `60` stars, parse Python files with Tree-Sitter ASTs, and extract `2,285,387` function-docstring pairs as [[codeexp]](raw).
- **Learned [[data-filtering]]**: fine-tune BERT-base uncased on `11,208` annotated examples to predict normalized step-1 and step-2 scores in `[0, 1]`; validation MSE is `0.027` for adequacy and `0.018` for coverage.
- **Refined corpus selection**: reapply the same length / complexity heuristics to raw data, then keep pairs whose predicted adequacy and coverage scores exceed `1.0` after rescaling, producing `158,024` refined examples.
- **Backbone models**: evaluate GPT-2-base (`117M`), GPT-Neo-1.3B, GPT-Neo-2.7B, CodeT5 (`220M`), and zero-shot [[codex]]; GPT-2 / GPT-Neo are decoder-only, while [[codet5]] is encoder-decoder.
- **Training strategies**: compare S1 raw-only, S2 refined-only, and S3 raw-then-refined [[curriculum-learning]]; hold out `1%` of raw and `5%` of refined for validation, optimize cross-entropy on `16` Nvidia V100 `32GB` GPUs, and continue S3 from the best raw checkpoint.
- **Hyperparameters**: GPT-2 and CodeT5 use max input length `1024`; GPT-Neo uses `2048`; AdamW starts at learning rate `5e-5` with linear decay; batch size is `16` for GPT models and `32` for CodeT5; epochs are `3` for GPT models and `10` for CodeT5.
- **Inference and metrics**: generate with temperature `0.1`, max generated tokens `512` for CodeT5 / GPT-Neo and `256` for GPT-2; evaluate with BLEU, ROUGE-1/L F1, METEOR, CER, BERTScore, CodeBERTScore, and adapted [[kendalls-tau]] where `τ = |#Con - #Dis| / |#Con + #Dis + #Tie|`.

## Key Results

- CodeExp contains `2,285,387` raw pairs, `158,024` refined pairs, and `13,186` annotated pairs; the refined split is about `1/15` the size of the raw split but consistently better for training.
- The BERT filter achieves validation MSE `0.027` on adequacy and `0.018` on coverage before being used to mine the refined corpus.
- On automatic metrics, CodeT5-(r+r) leads `6/7` metrics with ROUGE-1 `0.3415`, ROUGE-L `0.3256`, CER `0.5695`, METEOR `26.87`, BERTScore `84.98`, and CodeBERTScore `79.52`; GPT-Neo27-(r+r) has the top BLEU at `10.72`.
- On human evaluation, CodeT5-(refined) reaches overall `3.446` versus human reference `3.515`, and its coverage score `3.061` exceeds the reference score `2.994`.
- Data refinement matters strongly: CodeT5-(refined) improves over CodeT5-(raw) from BLEU `5.39` to `8.02`, METEOR `17.51` to `24.38`, and overall human score `2.614` to `3.446` (`+31.8%`).
- All fine-tuned models beat their off-the-shelf counterparts by large margins; for example, GPT-Neo13 BLEU rises from `0.40` without fine-tuning to `10.45` with two-stage training.

## Limitations

- The paper only studies Python function-docstring pairs, so cross-language and class- / module-level documentation remain untested.
- Automatic metrics are still weak semantic proxies: every reported Kendall `τ` is below `0.5`, which the authors explicitly identify as insufficient.
- Coverage of detailed code semantics remains difficult for both humans and models; even the best systems still miss branch-specific behavior in some cases.
- The data pipeline inherits heuristic biases from repository-star thresholds, docstring-length filters, and cyclomatic-complexity constraints.
- The work studies fine-tuning strategies but does not explore large-scale pretraining specialized for code explanation.
