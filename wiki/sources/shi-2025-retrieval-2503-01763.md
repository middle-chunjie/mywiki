## Summary

The paper argues that tool retrieval is a major hidden bottleneck in tool-using LLM agents and introduces ToolRet, a benchmark explicitly designed to measure this stage at realistic scale. ToolRet aggregates and standardizes tool-use datasets into `7,615` retrieval tasks over a merged corpus of `43,215` tools, then adds target-aware instructions generated from `100` human-written seeds with GPT-4o. Across sparse, dense, reranking, and LLM-based retrievers, even strong conventional IR models perform poorly, showing that tool retrieval is harder than standard document retrieval because lexical overlap is low and the domain shifts toward functional matching. The paper also releases ToolRet-train with `200k+` instances to improve retrievers and raise downstream agent pass rate.

## Problem & Motivation

Most tool-use benchmarks hand the agent a tiny pre-filtered tool subset, which hides the real retrieval problem that arises when practical toolsets contain tens of thousands of APIs, code functions, and app actions. The authors argue that this simplification overestimates agent competence, because retrieval errors directly reduce the pass rate of downstream tool-use LLMs. They therefore target a benchmark that measures realistic tool retrieval difficulty, isolates why standard IR models fail, and provides training data for retrievers that need to map user intent to functionally appropriate tools rather than merely surface-level lexical matches.

## Method

- **Benchmark construction**: aggregate `34` tool-use datasets released between `2023-08` and `2024-12`, then normalize them into retrieval tuples of `(query, target tools)` aligned with BEIR/MTEB-style evaluation.
- **Task sampling**: encode tasks with `NV-Embed-v1`, run `k`-means with `k = min(#queries, #tools in toolset)`, and sample one query per cluster to reduce redundancy while preserving coverage.
- **Toolset merging**: manually inspect documentation and merge overlapping tool inventories across datasets, yielding a final corpus of `43,215` uniquely identified tools.
- **Instruction construction**: ask `3` experts to write `100` seed instructions, then use GPT-4o to generate target-aware instructions by in-context learning; each task is paired with `(q, I)` where `I` states relevance criteria for the target tools.
- **Benchmark composition**: organize ToolRet into three subsets based on tool documentation format: `36,978` Web APIs, `3,794` code functions, and `2,443` customized applications; the benchmark contains `7,615` tasks overall.
- **Evaluation protocol**: report `NDCG@K`, `Recall@K`, `Precision@K`, and completeness `C@K`, where `C@K = 1` iff all target tools appear in the top-`K` results, under both query-only and query-plus-instruction settings.
- **Model families**: benchmark sparse retrieval (`BM25s`), single-task dense retrievers (`contriever`, `gtr`, `colbertv2`, `COLT`), multi-task embedding models (`e5`, `bge`, `gte`, `NV-Embed-v1`, `GritLM`), cross-encoder rerankers, and LLM rerankers such as RankGPT-style setups.
- **Training dataset**: extend ToolRet into ToolRet-train with `200k+` instances from ToolACE, APIGen, and ToolBench; each example contains query `q`, instruction `I`, positives `T`, and `10` hard negatives retrieved by `NV-Embed-v1`.
- **Training objective**: optimize retriever parameters `θ` with the contrastive objective `` `L = -(1/|T|) Σ_{t_i∈T} log( exp(sim(I⊕q,t_i)) / (exp(sim(I⊕q,t_i)) + Σ_{t̂_j∈T̂} exp(sim(I⊕q,t̂_j))) )` ``, where `I ⊕ q` concatenates instruction and query with a special token.
- **Optimization details**: during training, set negative pool size `` `K = 10` `` and learning rate `` `5e-5` ``.

## Key Results

- ToolRet is substantially harder than conventional IR benchmarks: average target count is `2.17` per query versus `1.00` on NQ/MS MARCO and lexical overlap (ROUGE-L) is only `0.06` versus `0.31` on NQ and `0.34` on MS MARCO.
- In the query-only setting, even the strongest embedding baseline `NV-Embed-v1` reaches only `NDCG@10 = 33.83` and `C@10 = 32.12`; the best overall reranker `bge-reranker-v2-gemma` reaches `NDCG@10 = 35.51`, `Recall@10 = 49.43`, and `C@10 = 34.14`.
- The paper's qualitative summary is accurate numerically: all tested retrievers stay below roughly `52%` `Recall@10` and below `35%` `C@10` on the main benchmark analysis.
- Instructions help consistently: in the query-plus-instruction setting, `gte-Qwen2-1.5B-inst` reaches `NDCG@10 = 45.96` and `C@10 = 45.39`, while `bge-reranker-v2-gemma` improves to `NDCG@10 = 47.52`, `Recall@10 = 61.20`, and `C@10 = 48.90`.
- Generated instructions are reasonably reliable: `90.1%` are relevant to the query, `92.3%` describe target-tool features, `89.2%` cover all target tools, hallucination is absent in `94.1%` of cases, and expert agreement reaches `κ = 0.743`.
- Retrieval quality materially affects downstream agents: on ToolBench-G1, GPT-3.5 drops to a `50.60` pass rate with retrieved tools from `bge-large`, compared with oracle tools at `62.00`, a decrease of `11.40`.

## Limitations

- The benchmark is English-only and limited to text retrieval; multilingual and non-text tool retrieval are not studied.
- The paper does not thoroughly test prompt sensitivity, even though instruction wording materially changes LLM behavior.
- Merging tools from many datasets introduces one-to-many ambiguity: multiple tools can be functionally similar, but only the original dataset's target tools are treated as gold.
- The evaluation mainly studies retrieval-then-calling rather than interleaved retrieval-and-calling loops, so adaptive multi-step agent interaction is left open.
- ToolRet-train is proposed and shown to help, but the paper does not fully close the gap to oracle toolsets or solve fine-grained functional discrimination.
