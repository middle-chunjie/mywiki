## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature. Chain of Code (CoC) proposes a reasoning framework in which a language model first writes code-like intermediate steps and then executes them with a hybrid runtime: executable lines are run by Python, while undefined or non-executable semantic operations are delegated back to the model as an "LMulator" that predicts the next program state. This lets the method retain the precision of code execution for arithmetic or symbolic subproblems while still handling semantic judgments that are hard to formalize as runnable functions. On BIG-Bench Hard, CoC reaches `84%` with `text-davinci-003`, improving over direct prompting (`55%`) and Chain-of-Thought (`72%`), and the paper argues that the same interleaving pattern extends beyond language benchmarks to robotics tasks requiring API calls plus commonsense reasoning.

## Problem & Motivation

Chain-of-Thought is effective for semantic reasoning but often weak on exact arithmetic, symbolic manipulation, and tasks where deterministic execution would help. Prior code-based prompting methods improve numeric reasoning, but they assume every intermediate step can be rendered as executable code, which breaks down for semantic predicates such as sarcasm detection, fruit classification, or commonsense judgments. The paper's core motivation is to combine both regimes instead of choosing one: use code as the structured reasoning substrate and as an interface to external tools, while allowing a language model to "fill in" the semantic gaps when the interpreter cannot execute a line. This targets mixed semantic-algorithmic tasks and positions CoC as a more general reasoner than either pure natural-language reasoning or pure program execution.

## Method

- **Two-stage procedure**: CoC first performs **generation**, where the LM writes structured code or pseudocode for the query, then **execution**, where each line is processed either by Python or by an LM-based emulator.
- **Interweaved executor**: for each line `l_t`, if Python can execute it, the runtime updates the program state directly; otherwise the LMulator receives the question, prior code, and previous state trace, and predicts the next state `s_t`.
- **Program state tracking**: execution maintains explicit variable bindings such as `state_t = {var = value}` so that interpreter-executed lines and LM-emulated lines share one evolving memory.
- **Fallback rule**: undefined helpers such as `is_fruit(obj)` or `is_sarcastic(text)` are not forced into runnable code; instead, their outputs are simulated by the LM and fed back into the state before execution continues.
- **Implementation detail**: the paper's main implementation uses Python `try/except` logic line by line, storing state as strings that are parsed into Python built-in objects where possible.
- **Answer extraction**: the final prediction is normally read from a variable named `answer`; for irrecoverable failures, the LM can still emit a final `A: ...` answer string.
- **Evaluation setup**: main experiments use `text-davinci-003` with `k = 3` few-shot demonstrations on `23` BIG-Bench Hard tasks, plus cross-task prompting with `k = 3` examples from different task families; GSM8K is included as an additional arithmetic benchmark.
- **Ablations**: the paper compares full interweaving against `Python-only`, `LM-only`, `LM-state`, and two simpler `try Python except LM` variants to isolate the value of interpreter use and explicit state traces.
- **Instruction-tuned transfer**: extra experiments test `gpt-3.5-turbo` and `gpt-4` in zero-shot and coerced few-shot settings to see whether CoC still helps outside completion-style prompting.

## Key Results

- On BIG-Bench Hard single-task prompting with `text-davinci-003`, CoC reaches `84%`, versus `55%` for direct prompting and `72%` for Chain-of-Thought; human average is `68%`.
- On BIG-Bench Hard cross-task prompting, CoC reaches `61%`, ahead of direct prompting at `50%` and CoT at `55%`.
- On PaLM 2-S code variant, CoC improves single-task BBH from `49%` direct / `61%` CoT to `78%`.
- Ablation results on BBH single-task show `Interweave 84%`, `try Python except LM state 82%`, `try Python except LM 80%`, `Python 48%`, `LM state 63%`, and `LM 57%`.
- On algorithmic BBH tasks, CoC averages `95%`, compared with `71%` for CoT and `41%` for direct prompting; on NLP-tagged tasks it achieves `74%`, matching or slightly exceeding CoT.
- Representative per-task gains are large on symbolic or arithmetic tasks: `100%` on Multi-Step Arithmetic, `100%` on Tracking Shuffled Objects, and `96%` on Object Counting.
- On GSM8K single-task prompting, CoC achieves `71%`, versus `16%` direct and `63%` CoT.
- When adapted to instruction-tuned models with few-shot prompting, CoC reaches `91%` on BBH with `gpt-4`, compared with `88%` for CoT and `69%` for direct prompting.

## Limitations

- The method increases inference cost because it performs both code generation and stepwise execution, with additional context consumed by state traces.
- Benefits are uneven across task types; some language-heavy tasks such as `Ruin Names` remain weak or even degrade relative to simpler baselines.
- The implementation uses a relatively crude state representation based on strings and Python built-in data types, so arbitrary custom objects are not naturally supported during LM-side execution.
- The approach depends on reliable switching between executable and non-executable lines; simpler Python-only baselines collapse on many non-algorithmic tasks.
- Running LLM-generated code raises safety concerns, which the paper flags explicitly as a deployment risk requiring safeguards.
- The main headline experiments rely heavily on few-shot prompting with completion-style models, so real-world gains with newer chat models may depend on prompt engineering.
