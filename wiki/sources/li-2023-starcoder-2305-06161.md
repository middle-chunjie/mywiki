## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

StarCoder and StarCoderBase are `15.5B`-parameter open-access code large language models released by the BigCode community, designed to combine competitive coding performance with unusually transparent data governance. StarCoderBase is trained on `1T` tokens from a cleaned subset of The Stack plus GitHub issues, commits, and notebooks; StarCoder is then fine-tuned on `35B` Python tokens. Architecturally the models pair a decoder-only Transformer with `8K` context, `MQA`, `FIM` training, learned positional embeddings, and FlashAttention. Beyond benchmark gains, the paper contributes an end-to-end release pipeline: opt-out-aware data curation, PII redaction via a dedicated encoder model, benchmark decontamination, and attribution tooling based on membership checking plus BM25 search.

## Problem & Motivation

The paper targets a gap between strong proprietary code models and weaker or less transparent open alternatives. The authors argue that competitive code LLMs should not be trained on opaque, legally ambiguous corpora without mechanisms for opt-out, attribution, privacy protection, or independent auditing. They therefore pursue a joint technical and governance goal: build an open-access code model that is strong on Python and multilingual coding tasks while documenting how the data was curated, how sensitive information was removed, and how users can inspect possible training-set overlap in generated code.

## Method

- **Training corpus**: start from The Stack v1.2 and restrict to permissively licensed repositories; after opt-outs (`44` people), filtering, and source balancing, the final mixture spans `80+` programming languages plus GitHub issues, git commits, and Jupyter notebooks, with JSON/YAML/CSS down-weighted to `1 GB / 1 GB / 3 GB`.
- **Filtering and deduplication**: select `86` languages from `358`; apply extension-specific heuristics such as HTML visible-text ratio `>= 20%`, XML detection in first `100` chars, YAML/JSON size filters, and near-deduplication using `5`-gram MinHash + LSH at Jaccard similarity `0.7`.
- **Data formatting**: code examples use sentinel metadata tokens such as `<reponame>`, `<filename>`, and `<ghstars>`; each metadata field is prefixed independently with probability `0.2`. FIM is applied at character level with rate `0.5`, using `PSM` with probability `0.5` and `SPMv2` with probability `0.5`.
- **Tokenizer**: byte-level [[byte-pair-encoding]] with vocabulary size `|V| = 49,152`, including sentinel tokens; pre-tokenization adds digit splitting and the GPT-2 regex splitter.
- **Model architecture**: [[decoder-only-transformer]] with [[multi-query-attention]], learned absolute positional embeddings, and FlashAttention; hyperparameters are `40` layers, hidden size `6144`, intermediate size `24576`, `48` attention heads, max positions `8192`, and `≈15.5B` parameters.
- **Pretraining**: train StarCoderBase for `250k` iterations with global batch size `4M` tokens, Adam (`β1 = 0.9`, `β2 = 0.95`, `ε = 1e-8`), weight decay `0.1`, and cosine decay from `3e-4` to `3e-5` after `2000` warmup steps.
- **Fine-tuning**: initialize from StarCoderBase and continue on `35B` Python tokens for `8500` steps (roughly `2` epochs), using learning rate `5e-5 -> 5e-6` after `1000` warmup steps.
- **Systems setup**: training runs on `512` NVIDIA A100 `80 GB` GPUs across `64` nodes with tensor parallelism `4`, pipeline parallelism `4`, data parallelism `32`, micro-batch size `1`, gradient accumulation `16`, and BF16 training.
- **PII pipeline**: train a `≈125M`-parameter StarEncoder with MLM + NSP, then fine-tune it as a `6`-class NER model over a `12,000`-file annotated dataset; add pseudo-labeling and rule-based post-processing to improve redaction quality before applying it to `815 GB` of data.
- **Attribution tooling**: release a two-stage provenance workflow with a Bloom-filter membership check over `50`-character strings and an Elasticsearch `7.17` BM25 index using lowercase + ASCII folding and `3`-gram tokenization.

## Key Results

- **Python coding**: StarCoder reaches `33.6` pass@1 on HumanEval and `52.7` on MBPP, versus StarCoderBase at `30.4 / 49.0`; it slightly exceeds OpenAI `code-cushman-001` on HumanEval (`33.6` vs `33.5`) and clearly beats it on MBPP (`52.7` vs `45.9`).
- **Data-science coding**: on DS-1000, StarCoder scores `26.0` overall in completion mode and `25.4` in insertion mode, outperforming StarCoderBase (`23.8 / 24.0`) and all reported baselines across the library-specific breakdowns.
- **Open-domain coding**: on ODEX, StarCoder reports overall pass@1 of `44.7` (English), `37.6` (Spanish), `44.2` (Japanese), and `50.4` (Russian), with substantially smaller open-vs-closed domain gaps than prior open models.
- **PII redaction**: pseudo-labeling lifts key detection F1 from `56.66%` to `70.41%`, password F1 from `73.39%` to `81.57%`, and name F1 from `89.19%` to `91.59%`.
- **Long context**: using the full `8K` window lowers perplexity relative to `2K` on every tested language, e.g. C++ `2.01 -> 1.79`, JavaScript `1.98 -> 1.68`, and PHP `1.73 -> 1.43`.
- **Natural-language side evaluations**: StarCoderBase gets `21.5%` on GSM8K with PAL vs `8.4%` with CoT, `34.2%` 5-shot accuracy on MMLU, and overall StereoSet stereotype score `55.53` with ICAT `76.65`.

## Limitations

- The paper is from `2023`, so both the baseline set and the legal/safety framing predate newer code LLMs, stronger long-context methods, and later open-model release practices.
- Several evaluations are English-centric, and low-resource languages can behave unstably during training; the paper explicitly notes late-training degradation for `R`.
- Some auxiliary evaluations may still be partially contaminated because their datasets are sourced from public GitHub repositories; the paper itself flags this risk for return-type prediction and docstring generation.
- The PII pipeline excludes some categories from final redaction decisions, notably IDs and usernames, and can still make false positives or false negatives depending on language and context.
- Attribution tools only search the retained training subset; the Bloom-filter check can produce false positives, and neither tool distinguishes boilerplate from protected code.
- OpenRAIL-M constrains downstream use but does not eliminate model capability risks: the model can still emit insecure code, PII, toxic text, or malicious artifacts.
