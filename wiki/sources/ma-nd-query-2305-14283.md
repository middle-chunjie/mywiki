## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

This paper proposes Rewrite-Retrieve-Read for retrieval-augmented LLMs, adding an explicit query rewriting stage before web retrieval to bridge the gap between user input and the knowledge actually needed by a frozen reader. The framework supports both prompting an LLM to emit search queries and training a smaller T5 rewriter with warm-up supervision plus PPO from downstream reader feedback. Retrieval uses Bing snippets or URL text filtered by BM25, while the readers are ChatGPT and Vicuna-13B. Across HotpotQA, AmbigNQ, PopQA, and MMLU, rewritten queries usually improve both retrieval hit rate and end-task accuracy, showing that query adaptation is a distinct leverage point in black-box retrieval-augmented pipelines.

## Problem & Motivation

Standard retrieve-then-read systems usually pass the original question directly to the retriever, even though the raw input is often a poor search query. The authors argue that there is a structural gap between the surface form of the task input and the knowledge needed for retrieval, especially when the reader is a black-box LLM that cannot be jointly trained with the retriever. They therefore target query rewriting as an independent adaptation layer: clarify the information need before retrieval, use an up-to-date web search engine instead of a maintained dense index, and optionally train a small rewriter to align the query with a frozen retriever-reader stack.

## Method

- **Pipeline**: given task input `x`, the system first generates a rewritten query `\tilde{x}`, retrieves context `doc`, then predicts `\hat{y}` from `[doc, x]` in a Rewrite-Retrieve-Read pipeline.
- **Frozen rewriter**: an LLM is prompted with few-shot demonstrations to produce zero, one, or multiple search queries; the prompt format is task-specific but keeps the reader prompt fixed across settings.
- **Trainable rewriter**: a small sequence-to-sequence model `G_\theta` initialized from `[[t5]]` takes over rewriting; warm-up uses supervised next-token training with `\mathcal{L}_{warm} = -\sum_t \log p_\theta(\hat{\tilde{x}}_t \mid \tilde{x}_{<t}, x)`.
- **RL formulation**: rewriting is modeled as an MDP where state `s_t = [x, \hat{\tilde{x}}_{<t}]`, action `a_t` is the next token, and an episode ends at EOS; rewards are computed only after retrieval and reading.
- **PPO optimization**: the policy is updated with clipped PPO, ratio `k_{t,\theta} = p_\theta(a_t \mid s_t) / p_{\theta'}(a_t \mid s_t)`, advantage from GAE, and reward `R(s_t, a_t) = R_{lm}(\hat{\tilde{x}}, y) - \beta KL(\pi_\theta \| \pi_0)`.
- **Task rewards**: for open-domain QA, `R_{lm} = EM + \lambda_f F_1 + \lambda_h Hit`, where `Hit = 1` if the answer appears in retrieved context and `-1` otherwise; for MMLU, `R_{lm} = EM`.
- **Retriever and reader**: the retriever is Bing with either returned snippets or full URL text re-ranked by `[[bm25]]`; the frozen readers are `[[chatgpt]]` (`gpt-3.5-turbo`) and `[[vicuna-13b]]`.
- **Training hyperparameters**: warm-up uses `lr = 3e-5`, batch sizes `{16, 20}`, epochs `{6, 8, 12}`; RL sampling uses `5120` steps, `10` threads, `512` steps per thread; PPO training uses `lr = 2e-6`, batch sizes `{8, 16}`, epochs `{2, 3, 4}`, `KL_target = 0.2`, `K_\beta = 0.1`, `\beta_0 = 0.001`, and `4`-beam decoding.

## Key Results

- **HotpotQA**: direct prompting scores `32.36` EM / `43.05` F1, retrieve-then-read drops to `30.47` / `41.34`, LLM rewriting recovers to `32.80` / `43.85`, and the trainable rewriter reaches the best `34.38` / `45.97`.
- **AmbigNQ**: direct prompting scores `42.10` EM / `53.05` F1, retrieve-then-read improves to `45.80` / `58.50`, LLM rewriting reaches `46.40` / `58.74`, and the trainable rewriter reaches `47.80` / `60.71`.
- **PopQA**: direct prompting scores `41.94` EM / `44.61` F1, retrieve-then-read reaches `43.20` / `47.53`, LLM rewriting peaks at `46.00` / `49.74`, and the trainable rewriter is slightly lower at `45.72` / `49.51`.
- **Retrieval quality on AmbigNQ**: BM25-based retrieval outperforms snippets, and trainable rewriting raises hit ratio from `76.4%` under retrieve-then-read to `82.2%`, with EM/F1 increasing from `45.80/58.50` to `47.80/60.71`.
- **MMLU**: with ChatGPT as reader, rewriting improves STEM from `58.8` to `63.5` EM and Other from `69.0` to `72.6`; with Vicuna-13B, the trainable rewriter raises Humanities from `39.8` to `43.2` and Other from `50.2` to `59.3`.

## Limitations

- The trainable rewriter introduces task-specific training, so it sacrifices some direct-transfer generality relative to pure few-shot prompting.
- The framework is deliberately one-turn; unlike agent-style multi-hop systems, it does not iteratively search, read, and reformulate queries.
- Web search is convenient and up-to-date, but it is noisy, commercially gated, and less controllable than curated dense-retrieval corpora.
- On PopQA, the trainable rewriter remains slightly worse than the frozen LLM rewriter, suggesting imperfect distillation and difficulty learning when retrieval should be skipped.
