## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature in LLM-based robot planning.

Demo2Code is a framework that generates executable robot task code from a combination of natural language instructions and state-based demonstrations, using a two-stage extended [[chain-of-thought-prompting]] pipeline. Stage 1 applies recursive summarization to compress demonstrations — first individually, then jointly — into a compact latent task specification. Stage 2 recursively expands this specification into hierarchical, composable Python code by iteratively defining undefined helper functions until only primitive API calls remain. Evaluated on a tabletop manipulation simulator, the novel Robotouille cooking simulator, and the EPIC-Kitchens egocentric video dataset, Demo2Code achieves the highest unit-test pass rates across all benchmarks compared to language-only and demo-only baselines.

## Problem & Motivation

Existing LLM-based robot planners (e.g., [[code-as-policies]]) translate natural language instructions into code but cannot leverage demonstrations. Naively concatenating demonstrations into an LLM prompt is intractable for long-horizon tasks due to context-length limits. Simultaneously, generating complex, control-flow-rich code in a single step is error-prone. The key insight is that demonstrations and code share a latent task specification — a compact language description of the task — that can serve as a structured intermediate representation connecting the two modalities.

## Method

**Two-stage extended chain-of-thought:**

- **Stage 1 — Recursive Summarization (`summarize(demos)`):**
  - Each demonstration (a sequence of predicate-state transitions in an MDP `<S, A, T, R>`) is independently summarized until the LLM deems it sufficiently compact (e.g., predicate changes → low-level actions → high-level subtasks).
  - All compressed demonstrations are then jointly summarized into a single latent task specification, formatted as pseudocode-style natural language prefixed with "Specifically: …"
  - Intermediate reasoning (e.g., aggregating subtasks into an ordered list) encourages the LLM to identify control loops and repeated patterns.

- **Stage 2 — Recursive Code Expansion (`expand_code(code)`):**
  - Given the task specification as a docstring and imported action/perception API libraries, the LLM generates high-level task code that may call undefined helper functions (e.g., `cook_obj_atLoc`).
  - Each undefined function is recursively expanded in subsequent calls until the base case is reached: all functions use only imported API primitives.
  - Encourages composable, reusable code with loops and conditionals.

**Evaluation metrics:**
- Execution Success Rate (0/1 code runs without error)
- Unit Test Pass Rate (all subgoals achieved, constraints satisfied via state-transition inspection)
- Code BLEU (similarity to oracle Spec2Code output)

**Model:** `gpt-3.5-turbo-16k` at temperature `0` for all experiments.

**Robotouille:** Novel open-source cooking simulator built on PDDLGym with procedurally generated scenarios; `10` scenarios per task-preference pair; average training horizon `12.7` states vs. testing horizon `31.3` states.

## Key Results

- **Tabletop Manipulation:** Demo2Code overall unit-test pass rate `0.88` vs. Lang2Code `0.19` and DemoNoLang2Code `0.81`; execution success `0.99` across all methods.
- **Robotouille (Table 2):** Demo2Code unit-test pass rate `0.59` (overall), execution success `0.79` — highest among all baselines; Lang2Code `0.29`/`0.64`, DemoNoLang2Code `0.38`/`0.49`.
- **EPIC-Kitchens (Table 3):** Demo2Code passes unit tests for 5/7 users vs. Lang2Code 3/7 and DemoNoLang2Code 2/7.
- **Chain-of-thought ablation:** No CoT yields near-zero pass rate for long-horizon tasks; full chain significantly outperforms 1-step or 2-step variants.
- **Noise robustness:** Removing 10% of predicates or states does not degrade — and sometimes improves — performance on long-horizon tasks, as removed predicates are rarely key information and reducing length helps the LLM.
- Demo2Code generalizes to unseen ingredients (cheese, chicken, tomato), unseen task structures, and generates for-loops for the longest task (70-state burger assembly).

## Limitations

- Context length: concatenating sufficiently summarized demonstrations from extremely long tasks (e.g., day-long cooking) may still exceed maximum context length; incremental chunked specification building is left as future work.
- Primitive API assumption: Stage 2 assumes all low-level action primitives are pre-provided; the system cannot automatically extend its skill library.
- LLM hallucination: no closed-loop feedback mechanism; incorporating self-supervised verification (reprompting on failure) remains an open challenge.
- Evaluation mismatch: task-code planners measure complexity via horizon length and control-flow structure, while classical planners use LTL-based complexity — the two evaluation paradigms are not directly comparable.
- EPIC-Kitchens evaluation relies on human annotators due to the absence of a matching simulator, limiting scalability.
