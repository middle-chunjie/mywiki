## Summary

Memory-R1 proposes a reinforcement-learning-based memory stack for long-horizon LLM agents. The system decomposes persistent dialogue reasoning into a Memory Manager that updates an external memory bank with `ADD`, `UPDATE`, `DELETE`, or `NOOP`, and an Answer Agent that distills retrieved memories before answering. Both modules are optimized with outcome-driven RL, specifically PPO or GRPO, using exact-match answer correctness as reward rather than dense labels for intermediate memory actions. On LoCoMo, the method reports new state-of-the-art performance with only `152` training QA pairs, and the same pipeline transfers without retraining to MSC and LongMemEval. The paper's main claim is that adaptive memory control and selective memory use should be learned jointly with downstream task success rather than scripted with heuristics.

## Problem & Motivation

LLM agents are effectively stateless once relevant evidence falls outside the context window, which makes multi-session dialogue reasoning brittle. Prior memory-augmented pipelines usually bolt on an external store and heuristic retrieval, but they do not learn when to preserve, merge, delete, or ignore facts, and they often force the model to reason over noisy retrieved context. Memory-R1 targets this gap by treating both memory control and memory utilization as reinforcement-learning problems tied directly to final answer correctness.

## Method

- **Two-agent pipeline**: a Memory Manager maintains an external memory bank during dialogue, while an Answer Agent consumes retrieved memories to answer questions; the manager chooses among `ADD`, `UPDATE`, `DELETE`, and `NOOP`.
- **Manager policy**: given extracted information `x` and retrieved old memories `M_old`, the manager samples `(o, m') ~ π_θ(. | x, M_old)`, where `o` is the operation and `m'` is the updated memory content.
- **PPO objective**: the manager and answer agent can both be trained with clipped PPO, `J(θ) = E[min(ρ_θ A, clip(ρ_θ, 1 - ε, 1 + ε) A)]`, using downstream answer quality as the scalar signal.
- **GRPO objective**: the alternative optimizer samples `G` candidates and uses group-relative normalization, `J(θ) = E[(1 / G) Σ_i ρ_θ^(i) A_i - β D_KL(π_θ || π_ref)]`, with `A_i = (r_i - mean(r)) / std(r)`.
- **Reward design**: both modules are trained with outcome-based reward `R_answer = EM(y_pred, y_gold)`, avoiding manual supervision for memory operations or memory distillation decisions.
- **Answer-time memory use**: for each question the pipeline retrieves `60` candidate memories via RAG, then the Answer Agent applies a memory distillation policy before generating the final answer `y ~ π_ans(. | q, M_ret)`.
- **Training data construction**: LoCoMo is split `152 / 81 / 1307` for train/validation/test QA; the appendix describes building temporal memory snapshots from prior turns and pairing them with dialogue turns or QA tuples for the two RL stages.
- **Implementation details**: experiments use `4 x NVIDIA H100 80GB` GPUs for most settings (`8` for Qwen-2.5-14B), total batch size `128`, micro-batch size `2` per GPU, max prompt length `4096`, max response length `2048`, actor LR `1e-6`, critic LR `1e-5`, training temperature `1.0`, and evaluation temperature `0`.

## Key Results

- On LoCoMo with LLaMA-3.1-8B-Instruct, `Memory-R1-GRPO` reaches overall `F1 = 45.02`, `BLEU-1 = 37.51`, and `Judge = 62.74`, which the paper summarizes as relative gains of `48%`, `69%`, and `37%` over Mem0.
- On Qwen-2.5-7B-Instruct, `Memory-R1-GRPO` reaches overall `F1 = 43.14`, `BLEU-1 = 36.44`, and `Judge = 61.51`, exceeding Mem0 by `57.3%`, `41.5%`, and `33.8%`, respectively.
- The method is data-efficient: it is trained with only `152` LoCoMo QA pairs, yet the same policy is reported to generalize zero-shot to `3` benchmarks: LoCoMo, MSC, and LongMemEval.
- Ablations show each component matters: RL-tuned memory management reaches `F1 = 24.91`, `BLEU-1 = 33.05`, `Judge = 59.91`; memory distillation raises answer-agent performance from `34.37 / 40.95 / 60.14` to `37.51 / 45.02 / 62.74`.

## Limitations

- The evaluation is dialogue-centric; the paper does not test whether the same RL-based memory design transfers to multimodal or non-conversational agent settings.
- The Memory Manager and Answer Agent are trained separately to handle sparse rewards, so the framework is not end-to-end and may miss coordination gains available to joint optimization.
- Reward choice matters: judge-based rewards improve `Judge` score but produce longer answers that underperform on `F1` and `BLEU-1`, indicating metric sensitivity in the optimization target.
- The appendix is slightly inconsistent about history construction for manager training: the prose mentions preceding `24` turns, while Algorithm 1 states previous `50` turns.
