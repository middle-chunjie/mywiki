## Summary

WebSailor proposes a full post-training pipeline for open-source [[web-agent]] systems that must solve high-uncertainty [[information-seeking]] tasks on the open web. The paper argues that BrowseComp-style performance depends less on standard multi-hop QA skill than on strong [[uncertainty-reduction]] behavior: exploring, pruning, cross-checking, and synthesizing evidence under sparse guidance. To train this capability, the authors generate hard questions with graph-based structured sampling and [[information-obfuscation]], reconstruct concise reasoning from expert traces, apply a small [[rejection-sampling-fine-tuning]] cold start, and then optimize with DUPO, an efficient agent RL algorithm. Across Qwen-2.5 backbones from 3B to 72B, WebSailor establishes a new open-source state of the art on BrowseComp-en/zh and remains competitive on GAIA, Xbench-DeepSearch, and SimpleQA.

## Problem & Motivation

Existing open-source agents perform poorly on complex web benchmarks because most training data emphasizes either low-uncertainty lookup tasks or fixed-path multi-hop QA. The paper frames BrowseComp-style problems as high-uncertainty, hard-to-reduce tasks where the agent must search a large unstructured evidence space, decide which branches to pursue, and converge within a limited number of tool calls. The authors therefore target a stronger form of web reasoning that goes beyond direct answer retrieval and instead learns robust exploration, evidence integration, and trajectory planning under sparse rewards.

## Method

- **Agent formulation**: the system uses the [[react]] framework with trajectories `H_T = (tau_0, a_0, o_0, ..., tau_T, a_T)`, where each round contains a thought, a tool action, and an observation. The action space has `search`, `visit`, and `final answer`.
- **Tool interface**: `search` can issue multiple queries and returns top-10 title/snippet/URL results per query; `visit` fetches webpage content and summarizes it for a specified goal. The appendix states that `visit` uses Jina retrieval plus Qwen-2.5-72B as the summary model.
- **SailorFog-QA data synthesis**: the authors start from rare entities in [[wikidata]], expand random-walk-style knowledge graphs over the web, sample subgraphs, and construct Level-3 questions whose answers require non-linear exploration rather than a fixed reasoning chain.
- **Information obfuscation**: question generation deliberately hides exact clues by converting precise facts into vague descriptors such as approximate dates, partial names, or qualitative attributes, increasing initial ambiguity while preserving answerability.
- **Reasoning reconstruction**: given an expert action-observation trace, WebSailor discards verbose native thoughts and reconstructs concise stepwise thoughts with `hat(tau)_t ~ pi*(tau | H_(t-1), a_t, o_t)`, producing short-CoT supervision that preserves strategy without copying the expert's style.
- **RFT cold start**: the supervised warm-start retains only correct trajectories, filters out examples longer than `32k` tokens, and keeps cases with `> 5` tool calls. Observation tokens are masked out of the loss so training focuses on thought and action generation.
- **DUPO reinforcement learning**: Duplicating Sampling Policy Optimization keeps only cases with non-zero reward variance inside a rollout group and fills batches by duplicating informative samples from the same batch. The paper reports `2-3x` speedup over DAPO-style dynamic sampling.
- **Policy objective**: DUPO uses a clipped token-level objective with group-relative advantage `A_hat_(i,t) = (R_i - mean(R))/std(R)` and a rule-based reward `R_i = 0.1 * R_i^format + 0.9 * R_i^answer`.
- **Training hyperparameters**: SFT uses batch size `32`, learning rate `5e-6`, minimum learning rate `1e-10`, weight decay `0.1`, and warmup-plus-cosine decay. RL uses rollout group size `8`, temperature `1.0`, `top_p = 1.0`, batch size `128`, mini-batch size `32`, learning rate `1e-6`, and at most `50` RL steps.

## Key Results

- On BrowseComp-en / BrowseComp-zh / Xbench-DeepSearch / GAIA, `WebSailor-72B` reaches `12.0 / 30.1 / 55.0 / 55.4`, the strongest reported open-source numbers in the paper.
- `WebSailor-32B` scores `10.5 / 25.5 / 53.3 / 53.2`, while `WebSailor-7B` still reaches `6.7 / 14.2 / 34.3 / 37.9`, outperforming several larger open-source baselines.
- Against prior open-source agents, the margin on BrowseComp-en is large: `WebDancer-32B` gets `2.5`, `WebThinker-RL` gets `2.8`, and `R1-Searcher-7B` gets `0.4`, versus `12.0` for `WebSailor-72B`.
- The synthetic training set is materially harder than WebDancer-QA: with browsing tools, `o4-mini` scores `47.3` on SailorFog-QA vs `90.2` on WebDancer-QA, and `DeepSeek-R1` scores `38.9` vs `84.4`.
- The cold start is small but important: the paper reports that just over `2k` high-quality RFT examples are enough to bootstrap RL effectively, while direct RL converges worse on hard benchmarks.
- DUPO improves RL efficiency by roughly `2-3x` relative to DAPO-style dynamic sampling, which matters because long-horizon agent rollouts are the main training bottleneck.

## Limitations

- The training pipeline drops trajectories longer than `32k` tokens, which likely caps performance on even harder long-horizon tasks.
- WebSailor can over-think, issuing multi-step tool calls for relatively simple questions instead of stopping early.
- RL is only run for `50` steps because synchronous agent RL is still slow, so the final policy may be under-optimized.
- Performance can still degrade as inference length grows, indicating unresolved context-management issues.
- Even after closing much of the open-source gap, the system remains behind DeepResearch on the hardest reported settings, especially BrowseComp-en.
