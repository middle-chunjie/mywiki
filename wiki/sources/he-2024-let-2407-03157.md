## Summary

The paper studies real-time code editing for code LLM assistants, where a user modifies existing code and expects the model to immediately repredict the next token or line without fully rebuilding the post-edit KV cache. The authors identify temporal confusion as the main failure mode of naively splicing freshly encoded edits into the old cache under RoPE-based models. They propose Positional Integrity Encoding (PIE), which updates only the positional component of suffix keys through a single matrix transform while reusing cached values. On RepoBench-C-8k with DeepSeek-Coder `1.3B`, `6.7B`, and `33B`, PIE preserves nearly the same code-completion accuracy as full recomputation while cutting KV-cache update latency by more than `85%`, making interactive code editing substantially cheaper.

## Problem & Motivation

Interactive coding assistants operate in a dynamic setting rather than the usual static decoding regime: developers insert, delete, and revise lines while expecting the model to continue completion from the edited state. Full recomputation of the KV cache after every light edit is accurate but too expensive for long contexts. A naive fast path that only re-encodes the edited span breaks positional consistency because old suffix tokens still occupy pre-edit RoPE positions, so the model attends to duplicated or shifted indices. The paper argues that for many code edits the semantic effect on the suffix is small, so fixing positional integrity alone may recover most of the benefit of exact recomputation.

## Method

- The edited context replaces original tokens `w_i ... w_j` with a new subsequence `a_1 ... a_m`. PIE keeps prefix cache entries `K_[1:i]` and `V_[1:i]` unchanged, freshly encodes only the edited span, and initially forms `K^edit = Concat(K_[1:i], K^edit_[i+1:i+m], K_[j+1:n])` and `V^edit = Concat(V_[1:i], V^edit_[i+1:i+m], V_[j+1:n])`.
- The structural error comes from RoPE-based positional indexing. Attention is written as `z_ij = x_i^T W_q^T R_(j-i) W_k x_j` with `R_(j-i) = R_i^T R_j`, so positional information is embedded inside the cached keys through the rotary matrix.
- For each suffix key at original position `j' in [j+1, n]` and layer `l`, PIE removes the stale RoPE position and reapplies the correct post-edit position via `k^{edit,l}_{j'} = R_(i+m+j'-j) R^(-1)_(j') k^l_{j'}`.
- Using the group property of RoPE, the update simplifies to a single shift transform: `k^{edit,l}_{j'} = R_(i+m-j) k^l_{j'}`. The same rotation is shared by all suffix positions, so the cache edit reduces to one matrix multiplication per cached key block rather than layerwise recomputation of the whole suffix.
- Values are not structurally misaligned by RoPE and are therefore reused unchanged for the suffix. The final cache is `K^edit = Concat(K_[1:i], K^edit_[i+1:i+m], K^edit_[j+1:n])` and `V^edit = Concat(V_[1:i], V^edit_[i+1:i+m], V_[j+1:n])`.
- Experiments use DeepSeek-Coder `1.3B`, `6.7B`, and `33B`; RepoBench-C-8k tasks for insertion, deletion, and multi-place edition; greedy decoding of `64` tokens; batch size `1`; and `3` random seeds. Metrics are Exact Match, Edit Similarity, and KV-cache update time.

## Key Results

- On insertion, PIE nearly matches full recomputation while drastically reducing latency. For Python with DeepSeek-Coder `33B`, XF-F Exact Match is `35.77` for PIE vs `35.75` for full recomputation, while update time drops from `2194 ms` to `134 ms`.
- On deletion, the same pattern holds. For Java with DeepSeek-Coder `6.7B`, IF scores are `50.88` EM / `80.52` ES for PIE vs `50.86` / `80.53` for full recomputation, with update time reduced from `578 ms` to `42 ms`.
- On the hardest edit setting combining insertion and deletion, average latency reductions are `87.9%` / `88.2%` for `1.3B`, `92.4%` / `92.8%` for `6.7B`, and `94.8%` / `95.2%` for `33B` on Python / Java.
- Representation fidelity stays extremely close to exact recomputation: cosine similarity between PIE-edited keys and full-recomputed keys is reported around `1.0` across layers, and token-distribution KL divergence stays below `0.0002` over `64` generated tokens.
- In appendix code-generation tests on HumanEval / HumanEval-X with DeepSeek-Coder `6.7B`, PIE reaches `43.9` / `49.7` Pass@1 versus `49.4` / `50.3` for full recomputation, substantially outperforming conflict fast encoding.

## Limitations

- The method is specialized to RoPE-style positional encoding; the derivation does not directly apply to models using different positional schemes.
- The paper explicitly assumes the semantic effect of small edits is often negligible in code. That assumption may fail in dialogue or natural-language settings where earlier edits strongly change later meaning.
- Evaluation focuses on code completion benchmarks and synthesized insertion/deletion/edit scenarios on RepoBench-C-8k, so broader generalization to editing-heavy IDE workflows is not established.
- The reported latency measures only KV-cache update cost, not end-to-end user-perceived editing latency including downstream decoding, retrieval, or tool orchestration.
