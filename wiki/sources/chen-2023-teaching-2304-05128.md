## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

SELF-DEBUGGING is a few-shot prompting framework that teaches a code-focused [[large-language-model]] to iteratively repair its own outputs through a generation-explanation-feedback loop. Starting from one or more sampled programs, the method optionally executes code, asks the model to explain the current solution or trace its behavior, and feeds back either correctness judgments or unit-test failures to elicit a revised program. Across [[text-to-sql-generation]], [[code-translation]], and [[text-to-python-generation]], the paper shows that this training-free procedure can outperform prompting-only reranking baselines, improve Codex from `81.3` to `84.1` on Spider with explanations, reach `92.5` on TransCoder with unit-test plus explanation feedback, and markedly improve sample efficiency by reusing failed candidates instead of discarding them.

## Problem & Motivation

Large language models are strong at code generation, but hard problems often fail on the first try. Prior repair and reranking methods either require extra training, verifier models, or large candidate pools. The paper asks whether a pretrained model can debug its own output with no additional finetuning by inspecting execution results and verbalizing what its code is doing. The motivation is pragmatic: humans often recover from near-miss solutions by explaining code, checking test outputs, and iteratively revising rather than restarting from scratch.

## Method

- **Overall loop**: each SELF-DEBUGGING turn has Generation, Explanation, and Feedback steps. The loop stops when feedback says the prediction is correct or when the maximum number of turns reaches `10`.
- **Execution-based initialization**: when multiple candidates are sampled, the paper first performs execution-based selection over programs with non-error execution results, choosing the prediction with the most frequent execution output. With unit tests `\{(i_k, o_k)\}_{k=1}^K`, a program `P` passes when `P(i_k) = o_k` for all `k`.
- **Sampling setup**: one-program initialization uses greedy decoding with temperature `\tau = 0`; multi-sample initialization uses `\tau = 0.7`. All explanation, feedback, and revised-program generations use greedy decoding.
- **Feedback types**: the framework studies simple correctness feedback, unit-test feedback (UT), code-explanation feedback (Expl), and execution-trace feedback (Trace). The trace is model-generated natural-language simulation rather than access to internal execution states.
- **Text-to-SQL instantiation**: for [[text-to-sql-generation]] on [[spider]], the prompt first infers the required return type, then executes the SQL and includes the returned table in the explanation prompt. If the table has more than `2` rows, only the first `2` rows are shown. The model compares the question description and SQL explanation to decide whether to revise.
- **Code translation instantiation**: for [[code-translation]] on [[transcoder]], the task is `C++` to Python translation on `560` test problems with `10` unit tests each. SELF-DEBUGGING is triggered only when the predicted Python program fails the available tests.
- **Text-to-Python instantiation**: on [[mbpp]], the benchmark contains `500` Python problems with `3` unit tests each. Following prior work, the prompt exposes the first unit test and keeps the remaining `2` hidden for full evaluation.
- **Models evaluated**: experiments cover [[code-davinci-002]], GPT-3.5, [[gpt-4]], and [[starcoder]] (`15.5B` parameters). The paper emphasizes that successful debugging usually ends within `3` turns even though `10` are allowed.

## Key Results

- Spider dev: Codex baseline is `81.3`, simple feedback stays at `81.3`, and explanation-based SELF-DEBUGGING reaches `84.1`, outperforming prompting-only baselines Coder-Reviewer (`74.5`) and MBR-Exec (`75.2`), and even exceeding LEVER (`81.9`), which uses extra training.
- Hardest Spider examples: the paper reports a `9%` accuracy gain on extra-hard SQL problems, with error fixes concentrated in wrong `WHERE` clauses (`25.7%`), missing `DISTINCT` (`17.1%`), and wrong `JOIN` clauses (`14.3%`).
- TransCoder: Codex improves from `80.4` to `92.5` with UT + explanation; GPT-4 rises from `77.3` to `90.4`; GPT-3.5 reaches `92.7`; StarCoder improves from `70.0` to `76.6`.
- MBPP: Codex improves from `61.4` to `70.8` with trace feedback and to `69.8` with UT + explanation; GPT-4 rises from `72.8` to `80.6`; StarCoder improves from `47.2` to `53.2`.
- Sample efficiency: on Spider, greedy decoding plus SELF-DEBUGGING matches the baseline that uses `16` samples, and SELF-DEBUGGING from `8` initial samples outperforms the baseline that uses `32` samples. The abstract further claims it can match or beat baselines using more than `10x` as many candidates.
- No-execution ablation: removing unit-test execution weakens the gains substantially; for example, Codex on TransCoder reaches only `83.9` without execution versus `92.5` with UT + explanation.

## Limitations

- The strongest gains depend on executable feedback. Without unit-test execution, improvements shrink sharply and can even regress, as on MBPP with Codex simple feedback (`61.4` to `57.6`).
- The study is prompt- and model-specific: SQL results favor Codex over GPT-3.5 and GPT-4, suggesting the method is sensitive to few-shot prompt following and task specialization.
- The evaluation covers three code benchmarks, but not broader agentic software engineering settings, long-horizon repositories, or newer code LLMs released after 2023.
- Iterative debugging adds extra generation and execution cost per problem, yet the paper does not provide a detailed latency or dollar-cost analysis relative to large-sample reranking baselines.
