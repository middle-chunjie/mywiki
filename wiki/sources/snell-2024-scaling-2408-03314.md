## Summary

The paper studies how to allocate inference-time compute for large language model reasoning instead of treating extra sampling as a uniform best-of-`N` budget. On MATH with a PaLM 2-S* base model, the authors analyze two axes: search against a dense process reward model and iterative answer revision that changes the proposal distribution itself. They show that the best strategy depends strongly on prompt difficulty: easy questions favor sequential refinement, while harder ones benefit more from wider search. A prompt-adaptive, compute-optimal policy based on difficulty bins improves test-time compute efficiency by roughly `2x-4x` over standard baselines, and in FLOPs-matched settings can outperform scaling model parameters by about `14x` on easier problems.

## Problem & Motivation

The paper asks how an LLM should spend a fixed but non-trivial test-time compute budget on difficult reasoning prompts. Prior work reported mixed results for self-correction, best-of-`N`, and verifier-guided search, making it unclear whether inference-time compute is a dependable scaling axis. The authors argue that different prompts require different allocation policies: some benefit from local refinement of partially correct solutions, while others require broader exploration. Understanding this allocation problem matters both for deploying smaller models more effectively and for evaluating whether added inference compute can substitute for some pretraining compute.

## Method

- **Task setting**: evaluate on the MATH benchmark with `12k` train and `500` test problems using a PaLM `2-S*` base model; difficulty is defined relative to the base model rather than human labels.
- **Unified abstraction**: frame test-time scaling as changing either the proposal distribution or the verifier used to rank candidate solutions.
- **Compute-optimal policy**: for prompt `q` and compute budget `N`, define `theta_q^*(N) = argmax_theta E_{y ~ Target(theta, N, q)} [1_{y = y*(q)}]`, then approximate it by selecting the best strategy per difficulty bin.
- **Difficulty estimation**: bin prompts into `5` levels using pass@1 from `2048` samples per question; a deployable approximation replaces oracle correctness with the average verifier score over the same `2048` samples.
- **PRM training**: train a [[process-reward-model]] as a binary classifier with soft Monte Carlo returns using BCE `-(y log y_hat + (1-y) log (1-y_hat))`; use `16` samples/question, `16` rollouts/step, AdamW with `lr = 3e-5`, `batch_size = 128`, `dropout = 0.05`, betas `(0.9, 0.95)`, and a `10%` held-out validation split.
- **PRM scoring and search**: aggregate step scores with the PRM's final-step value, then apply best-of-`N` weighted selection across answers. Search compares best-of-`N`, beam search with width `sqrt(N)` or `4`, and lookahead search with `k in {1, 3}`; beam expansion is capped at `40` rounds and lookahead cost is counted as `N * (k + 1)`.
- **Revision model**: build training trajectories by sampling `64` answers/question, pairing correct answers with up to `4` correlated incorrect answers chosen partly by character edit distance, then fine-tune with AdamW using `lr = 1e-5`, `batch_size = 128`, `dropout = 0.0`, betas `(0.9, 0.95)`.
- **Revision inference**: sample sequential revisions while truncating context to the most recent `4` attempts; because about `38%` of correct answers can be revised back into incorrect ones, final selection uses either a verifier or majority voting.
- **FLOPs-matched comparison**: compare test-time compute to parameter scaling with `X = 6 N D_pretrain` and `Y = 2 N D_inference`; matching a parameter scaling factor `M` requires multiplying small-model inference compute by `M + 3 (D_pretrain / D_inference) (M - 1)`.

## Key Results

- Compute-optimal search nearly matches PRM best-of-`N` with up to `4x` less test-time compute, e.g. about `16` versus `64` generations in low-budget regimes.
- For verifier search, beam search is best at low budgets and on medium-hard questions, but degrades on easy questions as budgets grow to `64-256` generations.
- For revisions, prompt-adaptive sequential/parallel allocation can beat parallel best-of-`N` with up to `4x` less compute, e.g. `64` versus `256` samples.
- Sequential revision chains improve pass@1 over steps even beyond the `4` previous-answer contexts seen during training.
- In FLOPs-matched comparisons against a model with about `14x` more parameters, compute-optimal test-time scaling wins on easier or intermediate questions at low inference-load ratios `R = 0.16` and sometimes `R = 0.79`, but loses on the hardest bins and high-load `R = 22`.

## Limitations

- The empirical study is narrowly centered on mathematical reasoning with MATH, so transfer to other reasoning, agentic, or open-ended NLP settings is unproven.
- The practical difficulty estimator is expensive because it uses `2048` samples/question; the reported gains do not fully internalize this deployment cost.
- Hard difficulty-bin questions show little benefit from current search and revision methods, suggesting limited capability extension beyond the base model's competence frontier.
- The main experiments rely on a specific PaLM 2-S* setup plus learned verifiers, so the exact exchange rate between pretraining and test-time compute may shift for other model families.
