## Summary

⚠ Possibly outdated: published 2021; re-verify against recent literature.

CoSQA introduces a human-annotated dataset of 20,604 real Bing web-query/Python-function pairs for query-code matching and pairs it with CoCLR, a contrastive training recipe built on a siamese CodeBERT encoder. Unlike prior resources derived mainly from documentation or Stack Overflow, CoSQA targets real search intent and labels each candidate pair with at least three annotators after heuristic query filtering and model-based candidate mining. The model concatenates query and code embeddings with their difference and element-wise product for binary relevance prediction, then augments training with in-batch negatives and simple query rewrites. On CodeXGLUE WebQueryTest, training on CoSQA improves CodeBERT accuracy by 5.1 points, and CoCLR adds another 10.5 points; code-search MRR rises from 54.41 to 64.66.

## Problem & Motivation

Existing code-search corpora are either large but weakly supervised, using documentation or Stack Overflow text as pseudo queries, or genuinely query-driven but too small to train deep models. The paper argues that real web queries have a different distribution from documentation and forum questions, so semantic matching between user intent and code is under-served by existing data. CoSQA is designed to fill that gap with a larger human-annotated web-query dataset, and CoCLR is proposed to further improve query-code matching by cheaply creating additional positive and negative training signals.

## Method

- **Query curation**: start from Microsoft Bing logs containing `python`; remove non-code-search intents with `100+` heuristic keywords over categories such as debugging, conceptual questions, tool usage, and programming knowledge. On a `250`-query human test set, the filter reaches `F1 = 67.65` and `accuracy = 82.40`.
- **Code collection**: use complete Python functions with documentation from [[codesearchnet]]; discard functions with non-English docs or malformed special tokens. The paper prefers full functions because they are self-contained, syntactically valid, and easier to pair with documentation.
- **Candidate mining**: retrieve high-confidence query-code pairs with a CodeBERT matcher fine-tuned on `148K` StaQC question-code pairs. Use cosine similarity on pooled `[CLS]` embeddings, keep only the top-1 code per query with similarity `> 0.5`, and cap each code's frequency at `10`.
- **Annotation protocol**: apply two-stage labeling for intent and answer correctness; keep only queries with code-search intent and pairs with at least `3` annotators after removing low-agreement items via Krippendorff-style agreement filtering.
- **Base encoder**: encode queries and code separately with a siamese [[codebert]] model, yielding `q_i = CodeBERT(q_i)` and `c_i = CodeBERT(c_i)` from pooled `[CLS]` states.
- **Matching head**: build a relation vector `r^(i,i) = tanh(W_1 [q_i, c_i, q_i - c_i, q_i ⊙ c_i])`, then predict relevance with `s^(i,i) = sigmoid(W_2 r^(i,i))` and binary cross-entropy `L_b`.
- **CoCLR objective**: optimize `L = L_b + L_ib + L_qr`, where in-batch augmentation uses `L_ib = -(1/(n-1)) Σ_{j != i} log(1 - s^(i,j))` and query-rewritten augmentation creates positive pairs by randomly deleting, copying, or switching query words.
- **Training setup**: use AdamW with batch size `32`; for code question answering set `lr = 1e-5` and `warmup = 0.1`, and for code search set `lr = 1e-6`. Evaluation uses CoSQA splits `20,000/604` for QA train/validation and `19,604/500/500` for code-search train/validation/test on a `16 GB` NVIDIA Tesla V100.

## Key Results

- Query-intent filtering reaches `F1 = 67.65` and `accuracy = 82.40` on a `250`-query human-labeled set before large-scale annotation.
- Final data statistics: `20,604` labeled query-code pairs, `6,267` distinct codes, average query length `6.60`, average code length `71.51`, and mean Krippendorff's alpha `0.63`.
- Code question answering on CodeXGLUE WebQueryTest improves from `47.80` with CodeBERT trained on CodeSearchNet alone to `52.87` with CoSQA training, then to `63.38` with CoCLR.
- Code search MRR improves from `51.29` with CodeBERT trained on CodeSearchNet to `54.41` after adding CoSQA, then to `64.66` with CoCLR.
- In ablations, query rewriting alone yields at best `55.66` MRR with word switching, in-batch augmentation alone reaches `63.51`, and combining in-batch augmentation with switched rewrites gives the best `64.66`.
- Removing documentation from code hurts search more than removing headers or bodies: MRR drops from `64.66` to `58.54`, versus `62.01` without headers and `59.11` without bodies.

## Limitations

- The dataset is restricted to Python; the paper explicitly leaves multilingual extension as future work.
- Query filtering still relies on heuristics with only moderate `F1`, so some search-intent errors likely remain upstream of annotation.
- CoCLR uses simple lexical rewrite operations rather than semantically controlled paraphrasing, which may introduce noisy positives.
- The strongest gains depend heavily on natural-language documentation, so conclusions may transfer poorly to undocumented or weakly documented code.
- Empirical evaluation is limited to code search and code question answering with a preprint-era benchmark stack rather than broader downstream coding tasks.
