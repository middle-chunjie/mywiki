## Summary

PAG (Planning-Ahead Generation) addresses a fundamental limitation of constrained beam search in generative retrieval: relevant document identifiers are frequently pruned during decoding because prefix scores are computed greedily without lookahead. The paper proposes a hybrid DocID design — a set-based identifier built from lexical tokens (decoded simultaneously in a single forward pass) and a sequential identifier derived from residual quantization of dense document representations (decoded autoregressively). A planning-ahead constrained beam search algorithm augments each prefix's cumulative score with the maximum simultaneous relevance score among documents whose sequential DocID shares that prefix, providing a document-level prior that prevents early pruning. Three-stage training first learns set-based and sequential decoders in parallel, then jointly fine-tunes a unified model. On MSMARCO Dev, PAG improves MRR@10 by 15.6% relative over RIPOR while using a 10× smaller beam size, yielding a 22× speedup in query latency on a single A100 GPU.

## Problem & Motivation

Constrained beam search in generative retrieval is a greedy local search: at each step only the top-k prefix scores survive. Unlike natural language generation, each relevant document has exactly one valid DocID, so if any prefix of that DocID falls below the beam threshold it is irrecoverably lost. Empirical analysis on RIPOR shows that even with beam size 1,000 the model fails to match brute-force decoding performance (scoring every document without pruning), meaning many relevant DocIDs are pruned at intermediate steps. This gap motivates a decoding algorithm that incorporates global document-level relevance signals to guide prefix selection rather than relying solely on next-token probabilities.

## Method

**DocID Construction**

- **Sequential DocID** `c^d = [c_1^d, …, c_L^d]` (`L = 8`, vocabulary `V = 2048`): the generative model M (T5-base) is used as a dense encoder; document representations are quantized via Faiss residual quantization (RQ) to produce integer token sequences. Scoring: `s(c^d; q) = Σ_i E_i[c_i^d] · h_i^d`, where `h_i^d = Decoder(Encoder(q), q, c^d_{<i})`.
- **Set-based DocID** `t^d = {t_1^d, …, t_m^d}` (`m = 64`): the model is used as a sparse encoder. A query or document representation `h^q ∈ R^{V_T}` is computed via `h^q = MaxPool(log(1 + ReLU(E_simul · Q^T)))`. The top-m tokens by weight in the document's sparse representation form the set-based DocID. Simultaneous relevance score: `s_simul(q, d) = Σ_{i=1}^{m} h^q[t_i^d]`.

**Planning-Ahead Constrained Beam Search**

1. Pre-compute `s_simul(q, d)` for all `d ∈ C` in a single forward pass; select top-n documents to form D.
2. Build prefix-to-max-score dictionary T: for each possible prefix `c*_{≤i}`, `T[c*_{≤i}] = max_{d ∈ D, c^d_{≤i} = c*_{≤i}} s_simul(q, d)`.
3. Modified scoring for autoregressive expansion: `s'(c^p_{≤i}; q) = max_{d ∈ D_{c^p_{≤i}}} s_simul(q, d) + s(c^p_{≤i}; q)`, where `D_{c^p_{≤i}}` is the subset of D sharing prefix `c^p_{≤i}`.
4. Beam of size `k = 100` selects the top-k prefixes by `s'(·)` at each step; final ranking uses `s(·)` (sequential score only) to avoid bias toward set-based signals.

**Three-Stage Training Pipeline**

- **Stage 1 (set-based)**: pre-train M as a SPLADE-style sparse encoder using MarginMSE + FLOPs regularization to learn sparse document representations; extract top-m tokens as set DocIDs; fine-tune generative retrieval model for set-DocID prediction with self-negative mining (model M^set).
- **Stage 2 (sequential)**: dense encoder pre-training with MarginMSE + hard negatives; seq2seq pre-training using 10 pseudo-queries per document (doc2query, M^s2s); prefix-oriented fine-tuning with multi-objective MarginMSE loss over sampled prefix lengths `S_L = {4, 8}`, weights `α_i = {0.5, 1.0}` (model M^seq). Stages 1 and 2 run in parallel.
- **Stage 3 (unified)**: initialize by averaging weights of M^set and M^seq; joint optimization combining `L_set + L_seq`; feed query text to decoder in addition to encoder output; trained for 120 epochs with Adam (lr=5e-4, warmup ratio 4.5%, 8×A100-40GB).

**Index memory**: 3.27 GB (vs 25.3 GB for single-vector dense retrieval models — 7.7× savings).

## Key Results

- **MSMARCO Dev** (6,980 queries, 8.8M passages): MRR@10 = 0.385 (+15.6% rel. over RIPOR 0.333), Recall@10 = 0.670 (+19.2% rel.); beam size 100 vs RIPOR's beam 1,000.
- **TREC-DL 2019**: NDCG@10 = 0.705 (+12.3% rel. over RIPOR 0.628); Recall@10 = 0.267.
- **TREC-DL 2020**: NDCG@10 = 0.700 (+10.9% rel. over RIPOR 0.631); Recall@10 = 0.236.
- **Latency** (single A100-80GB): PAG 25 ms / query (seq. dec.) vs RIPOR ~250 ms at beam 1,000 — **22× speedup**.
- **vs. dense retrieval** (for reference): +11.9% MRR@10 over TAS-B, +14.9% over TCT-ColBERT, +4.1% over RocketQA on MSMARCO Dev, while using 7.7× less index memory.
- **Ablation**: removing simultaneous scoring (−10.3% MRR@10, −17.7% Recall@10); using set-based DocIDs alone (MRR@10 = 0.303); joint training outperforms post-hoc linear interpolation of M^set + M^seq by +6.9% MRR@10 / +13.0% Recall@10.
- Beam size sensitivity: PAG with beam 10 achieves MRR@10 = 0.379; increasing to 100 gives 0.385 (1.6% gain), much less sensitive than RIPOR.

## Limitations

- The simultaneous decoding brute-force scores all `|C|` documents (O(|C|·m) FLOPs); may become slower than sequential decoding for billion-scale corpora — the paper acknowledges this and suggests future use of ANN approximation.
- Index memory (3.27 GB) is still larger than pure semantic-DocID systems like RIPOR (1.06 GB), because set-based DocIDs add 2.77 GB.
- About 14% of MSMARCO Dev queries and 21% of TREC-DL 19/20 queries show performance degradation from joint scoring, likely for queries better suited to lexical matching alone.
- The approach inherits RIPOR's sequential DocID design (RQ-based); alternative DocID schemes (e.g., title-based, n-gram-based) are not explored.
- Evaluated only on English passage retrieval; generalization to other languages or document retrieval is not studied.
