## Summary

This paper introduces SWE-Gym, a training environment for [[software-engineering-agent]] research built from real GitHub issues, executable repository snapshots, and unit-test-based reward signals. The released benchmark contains `2,438` validated Python task instances from `11` repositories, plus a `230`-instance [[swe-gym-lite]] subset for cheaper experimentation. Using successful [[agent-trajectory]] data collected in SWE-Gym, the authors fine-tune Qwen-2.5-Coder-Instruct agents and train verifiers for [[inference-time-scaling]]. The environment yields large gains over zero-shot open-weight agents: OpenHands with a 32B model rises from `7.0%` to `20.6%` on SWE-Bench Verified after training, and verifier reranking lifts performance further to `32.0%`. The main contribution is therefore not just a dataset, but an executable post-training substrate for repository-level coding agents.

## Problem & Motivation

Software engineering agents had been improving mostly through scaffolding and prompting, but the field lacked a reusable training environment with executable runtimes, reproducible dependencies, and reliable reward signals. Existing resources either provide only final patches, focus on synthetic or isolated programming tasks, or omit executable verification. The paper targets this bottleneck by building an environment where a model can interact with a real repository, receive unit-test feedback, and produce training data for both policy improvement and verifier learning. The goal is to make learning-based SWE agents viable without depending on proprietary frontier models or heavy prompt engineering.

## Method

- **Repository filtering and raw pool**: start from Python repositories created before `2022-07-01` with `>500` stars, `>500` PRs, `>100` contributors, and at least `300` lines of code; this yields `358` repositories and `64,689` raw issue-derived instances (`SWE-Gym Raw`).
- **Executable environment construction**: adapt SWE-Bench extraction/versioning scripts, version instances by repository release information, then manually configure dependencies from files such as `requirements.txt`, CI scripts, and docs; validation keeps only instances where the human gold patch improves test outcomes.
- **Final environment**: retain `2,438` unit-test-validated tasks from `11` repositories and release per-instance Docker images totaling about `6 TB`; define [[swe-gym-lite]] as a `230`-task subset filtered toward single-file, better-specified, cheaper tasks.
- **Agent setting**: study two [[software-engineering-agent]] scaffolds, [[openhands]] CodeAct for general-purpose long-horizon interaction and [[moatlesstools]] for specialized workflows; base models are [[qwen-2-5-coder-instruct]] `7B`, `14B`, and `32B`.
- **Policy improvement**: use [[rejection-sampling-fine-tuning]] on successful [[agent-trajectory]] samples. For OpenHands, collect `491` successful trajectories from `gpt-4o-2024-08-06` and Claude-3.5-Sonnet variants; successful traces average about `39.9` messages and `18,578.5` tokens.
- **OpenHands training hyperparameters**: full-parameter fine-tuning with `lr = 1e-4`, `epochs <= 5`, `global_batch_size = 8`, and `max_context_length = 32768`; evaluation is bounded by `100` interaction turns or the `32k` context window.
- **MoatlessTools self-improvement**: run iterative on-policy sampling with `30` rollouts per task at `t = 1.0`, cap accepted successes at `2` trajectories per instance to reduce easy-task bias, then fine-tune again; the `32B` run uses LoRA with `rank = 64`, `lr = 5e-4`, `epochs = 5`, and batch size `8`.
- **Verifier training**: train an [[outcome-supervised-reward-model]] over task context plus trajectory. For OpenHands, the verifier scores `tau = [o_1, a_1, ..., o_n, a_n]` and predicts `<YES>`/`<NO>`; reward is `r = exp(l_y) / (exp(l_y) + exp(l_n))`.
- **Verifier data**: combine `443` off-policy successes and `875` on-policy successes, sample matched failures, and fine-tune on `2,636` total trajectories; at inference, sample `k` candidate trajectories and rank them for [[inference-time-scaling]].

## Key Results

- SWE-Gym contains `2,438` validated tasks from `11` repositories; SWE-Gym Lite contains `230` tasks. SWE-Gym Raw has `64,689` issue-derived instances from `358` repositories.
- Dataset difficulty is non-trivial: average codebase size is `971.2` non-test files and `340,675.0` non-test LOC; average gold patch edits `69.8` lines across `2.5` files, versus `32.8` lines and `1.7` files for SWE-Bench test.
- OpenHands + Qwen-2.5-Coder-Instruct-32B on SWE-Bench Verified improves from `7.0%` zero-shot to `20.6%` after training on `491` successful trajectories (`+13.6` absolute points).
- OpenHands + 32B on SWE-Bench Lite improves from `3.0%` to `15.3%` (`+12.3` points); 14B rises from `4.0%` to `16.4%` on Verified, and 7B rises from `1.8%` to `10.6%`.
- On Verified, the 32B OpenHands model reduces stuck-in-loop rate from `29.4%` to `23.8%`; 7B drops from `39.6%` to `21.0%`.
- MoatlessTools reaches higher zero-shot Lite performance than OpenHands for open-weight models: `19.0%` with 32B and `7.0%` with 7B; after two self-improvement iterations, the 7B model reaches `10.0%` and the 32B model `19.7%`.
- Verifier reranking substantially improves inference-time performance: OpenHands 32B rises from `20.6@1` to `29.8@8` Best@`8`, and to `32.0@16` on SWE-Bench Verified; Pass@`16` reaches `42.8%`.
- Relative to public open-weight baselines on SWE-Bench Verified, the paper reports `32.0%` for the 32B agent+verifier system, compared with `28.8%` for Lingma SWE-GPT 72B.

## Limitations

- SWE-Gym covers only Python and only `11` repositories after filtering, so language and repository diversity remain limited.
- Building executable environments required roughly `200` human annotation hours plus `10,000` CPU core hours, which makes extension expensive.
- The OpenHands self-improvement experiment is negative: mixing `868` on-policy successes with prior data drops SWE-Bench Lite resolution from `15.3%` to `8.7%`.
- Verifier quality is still imperfect: on Verified, Best@`16 = 32.0%` remains well below Pass@`16 = 42.8%`, leaving substantial headroom.
- The environment uses pre-built Docker images totaling about `6 TB`, which raises storage and deployment costs.
- The paper argues scaling has not saturated, but current evidence is limited to `491` successful training trajectories and relatively small open-weight model families.
