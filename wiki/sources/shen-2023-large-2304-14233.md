## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

This paper proposes LameR, a zero-shot large-scale retrieval pipeline that replaces learned dense retrievers with a large language model plus BM25. The key idea is to first retrieve a small set of candidate passages for a query, then prompt the LLM with both the query and those candidates so it can generate answer-like passages that better match the target collection's domain, intent, and lexical patterns. The generated answers are concatenated back with the original query and passed to BM25 for final retrieval. Across TREC DL19/20 and low-resource BEIR tasks, the method substantially improves over vanilla BM25 and HyDE, showing that candidate-grounded LLM query augmentation plus transparent lexical matching can be a strong zero-shot retrieval recipe.

## Problem & Motivation

The paper studies zero-shot large-scale retrieval, where no labeled query-document pairs are available for the target collection. Prior self-supervised dense retrievers such as Contriever are often weaker than BM25 in this setting, and simple LLM-based augmentation like HyDE can still be bottlenecked by the downstream retriever or generate out-of-domain answers from short, ambiguous queries. The authors aim to make the LLM aware of in-collection patterns before answer generation and to avoid the mismatch between expressive natural-language augmentations and weak latent retrievers.

## Method

- **Retrieval backbone**: use BM25 as the non-parametric retriever with default Pyserini hyperparameters `k1 = 0.9` and `b = 0.4`, scoring documents by `Rel_BM25(d, q) = Σ_{t in q} IDF(t) * TF(t,d) * (k1 + 1) / (TF(t,d) + k1 * (1 - b + b * len(d) / avgdl))`.
- **Candidate retrieval**: for query `q`, first retrieve a small candidate set `C^q = Retriever(q, D, M)` with `M = 10` in the main setting; the candidates are intended to expose domain, answer format, and in-collection lexical patterns.
- **Candidate-prompted answer generation**: prompt the LLM with the query and enumerated candidate passages, asking it to write a correct answering passage even though most candidates may be wrong; the paper uses `gpt-3.5-turbo` by default and also evaluates `gpt-4`.
- **Multiple generations**: sample `N = 5` answers `A^q = {a_1^q, ..., a_N^q}` from the LLM to reduce vocabulary mismatch and provide multiple plausible lexical realizations of relevant content.
- **Query augmentation**: build the augmented query by concatenation, `q_bar = Concat(q, a_1^q, q, a_2^q, ..., q, a_N^q)`, then run final retrieval `D_hat^{q_bar} = Retriever(q_bar, D, K)` with `K = 1000` for evaluation.
- **Why BM25 instead of a dense encoder**: the authors argue that lexical retrieval takes the LLM outputs literally, avoiding the representational bottleneck of a self-supervised dense retriever that may not faithfully encode the generated augmentations.
- **Experimental setup**: queries and passages are truncated to `128` tokens; evaluation covers MS MARCO passage retrieval via TREC DL19/20 and six low-resource BEIR tasks; metrics include `MAP`, `nDCG@10`, and `Recall@1000`.

## Key Results

- On TREC DL19, LameR reaches `MAP = 47.2`, `nDCG@10 = 69.1`, and `R@1k = 89.9`, outperforming BM25 (`50.6` nDCG@10) and HyDE (`61.3` nDCG@10).
- On TREC DL20, LameR reaches `MAP = 45.6`, `nDCG@10 = 64.8`, and `R@1k = 88.7`, again beating BM25 (`48.0`) and HyDE (`57.9`) on `nDCG@10`.
- On BEIR low-resource tasks, LameR is best among zero-shot baselines on four of six datasets, including Scifact (`73.5`), TREC-COVID (`75.8`), DBPedia (`39.0`), and TREC-NEWS (`50.3`) in `nDCG@10`.
- Hyperparameter analysis shows gains saturate around `M = 10` retrieved candidates and `N = 5` generated answers, which become the paper's default trade-off between effectiveness and overhead.
- Oracle analysis shows headroom remains large: LameR-oracle reaches `84.0` `nDCG@10` on DL19, while a naive second round drops to `68.1`, suggesting error reinforcement rather than iterative improvement.
- Replacing `gpt-3.5-turbo` with `gpt-4` lifts DL20 `nDCG@10` from `64.8` to `65.9`, slightly surpassing even the fully supervised DPR baseline (`65.3`).

## Limitations

- The method is instruction-sensitive, so prompt wording and model choice can materially affect answer generation quality.
- LLM inference dominates compute cost even though the BM25 retrieval stages remain efficient in latency and index size.
- Performance is uneven across datasets: ArguAna drops to `40.2` `nDCG@10`, below HyDE's `46.6`, and FiQA remains weak at `25.8`, which the authors attribute to long inputs and limited financial knowledge.
- The large oracle gap indicates that retrieval quality is still constrained by the quality of demonstration candidates; naive multi-round application can amplify errors instead of correcting them.
