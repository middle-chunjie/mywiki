## Summary

The paper studies how to run decoder-only large language models continuously on streams that far exceed their pretraining context length without paying the full cost of dense attention or repeated cache recomputation. Its core observation is that many autoregressive transformers allocate substantial attention mass to the first few tokens even when those tokens are semantically irrelevant; the authors call these tokens attention sinks. StreamingLLM exploits this by keeping a tiny set of initial sink-token KV states together with a rolling recent-token cache, which stabilizes the attention distribution and preserves perplexity on very long inputs. Across Llama-2, MPT, Falcon, and Pythia, the method supports stable language modeling beyond 4 million tokens and yields substantial decode-time speedups over sliding-window recomputation.

## Problem & Motivation

Streaming deployments such as multi-round dialogue need stable generation over effectively unbounded token streams, but standard autoregressive decoding caches all previous KV states, so memory and latency grow with sequence length. Simple window attention fixes the memory growth by keeping only recent tokens, yet the model collapses as soon as the earliest tokens are evicted. The paper asks whether a pretrained LLM with a finite attention window can be turned into an efficient infinite-stream model without fine-tuning, and argues that the missing piece is not more context per se but preserving the special initial tokens that anchor the model's attention distribution.

## Method

- **Attention-sink diagnosis**: the paper explains window-attention failure through the softmax denominator, where `softmax(x)_i = e^{x_i} / \sum_j e^{x_j}` forces some attention mass to be assigned even when no prior token is strongly relevant. Initial tokens become globally visible repositories for this excess mass.
- **StreamingLLM cache layout**: keep a small sink block plus a rolling recent window, written as cache configuration `x + y`, where `x` is the number of preserved initial tokens and `y` is the number of recent cached tokens. The default deployment uses `x = 4`.
- **Empirical sink choice**: replacing the first four original tokens with four newline tokens still restores perplexity, indicating that absolute position rather than token semantics drives the sink effect.
- **Position handling inside cache**: token positions are reassigned contiguously within the current cache. If cached source positions are `[0, 1, 2, 3, 6, 7, 8]`, decoding the next token uses cache-local positions `[0, 1, 2, 3, 4, 5, 6, 7]` rather than original text positions.
- **RoPE integration**: for models with [[rotary-positional-embedding]], keys are cached before rotary transformation and re-rotated for the rolling segment at each decode step so that positional offsets stay consistent inside the cache.
- **ALiBi integration**: for models with [[alibi]], StreamingLLM applies a contiguous linear bias inside the cache instead of the bias implied by original-text gaps.
- **Evaluation setup**: long-text perplexity is measured on PG19 with `y = 2048` for Llama-2 and `y = 1024` for Falcon, MPT, and Pythia; StreamingLLM is compared against dense attention, pure [[window-attention]], and [[sliding-window-recomputation]].
- **Pretraining variant**: the paper also trains `160M`-parameter models from scratch under three settings: Vanilla attention, `SoftMax_1(x)_i = e^{x_i} / (1 + \sum_j e^{x_j})`, and a learnable [[sink-token]] prepended to every sequence. Training uses the deduplicated The Pile, batch size `256`, `143000` steps, and `8 x A6000` GPUs.

## Key Results

- On the first `65K`-token PG19 book with Llama-2-13B, pure window attention `0 + 1024` reaches perplexity `5158.07`, while reintroducing four initial tokens `4 + 1020` restores perplexity to `5.40`; using four newline tokens still gives `5.60`.
- On `400K` PG19 tokens, Llama-2-7B drops from perplexity `3359.95` under `0 + 4096` to `9.59` under `4 + 4092`; MPT-7B improves from `460.29` to `14.99`, Falcon-7B from `17.90` to `12.12`, and Pythia-12B from `21.62` to `12.09`.
- StreamingLLM matches the perplexity of the sliding-window recomputation oracle on `20K`-token texts and remains stable on concatenated PG19 streams beyond `4 million` tokens across Llama-2 `[7B, 13B, 70B]`, MPT `[7B, 30B]`, Falcon `[7B, 40B]`, and Pythia `[2.8B, 6.9B, 12B]`.
- In decode-time efficiency tests on a single NVIDIA A6000, StreamingLLM achieves up to `22.2x` per-token speedup over sliding-window recomputation while keeping a similar memory footprint.
- In streaming QA over concatenated ARC examples, Llama-2-13B-Chat reaches `80.89` on ARC-E and `65.61` on ARC-C with StreamingLLM, versus `0.25` and `0.34` for pure window attention; Llama-2-70B-Chat reaches `91.37` and `80.20`.
- In the `160M` pretraining study, a learnable sink token stabilizes streaming with only `1 + 1023` cached tokens at perplexity `18.01`, compared with `18.49` for vanilla and `19.90` for the zero-sink `SoftMax_1` variant.

## Limitations

- The method does not actually extend the model's usable long-term memory; it only stabilizes generation over long streams while the model still depends on information remaining inside the current cache.
- Performance degrades once the relevant query-answer distance exceeds cache capacity; StreamEval accuracy eventually falls to `0` when the required token distance is larger than the configured cache.
- For long-document QA and summarization, a tiny sink budget can underperform truncation baselines: on LongBench, `4 + 3496` is weaker than truncation on several tasks until many initial tokens are preserved.
- The implementation relies on relative-position handling that is compatible with RoPE or ALiBi and does not solve broader context-utilization weaknesses of current LLMs.
