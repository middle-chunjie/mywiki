## Summary

ToolTree proposes a training-free tool-planning framework for [[llm-agent]]s that recasts multi-tool decision making as executable [[monte-carlo-tree-search]] over tool trajectories rather than greedy next-step selection. Its core idea is to combine a fast pre-execution prior `r_pre(s, a)` with a grounded post-execution reward `r_post(s, a)`, then use them for both UCT-guided search and bidirectional pruning. This lets the planner explore alternative tool sequences, cut implausible branches before calls, and stop expanding branches that fail after real execution. Across closed-set benchmarks (GTA, m&m) and open-set benchmarks ([[toolbench]], [[restbench]]), ToolTree improves both accuracy and accuracy-per-second, with roughly 10% average gains over prior planning baselines while staying more compute-efficient than comparable search methods.

## Problem & Motivation

Existing LLM tool-use systems often plan greedily: they pick the next tool that looks locally best and commit to a single trajectory. That causes brittle error propagation when early tool choices are wrong and wastes opportunities to explore better branches. Search-based alternatives alleviate myopia, but tool planning makes the search space explode because each step depends on tool choice, argument construction, and updated execution state. The paper therefore targets a middle ground: a search procedure that remains forward-looking and recovery-capable like tree search, but uses lightweight predictive signals and explicit pruning so tool planning stays practical under fixed rollout budgets and large tool libraries.

## Method

- **Planning formulation**: model tool planning as sequential decision making over tool library `T_lib = {t_1, ..., t_m}`, state space `S`, action space `A`, and reward `R`, seeking a policy `pi : S -> A` that maximizes `E[R(s*) | pi, T_lib]` over executable tool trajectories.
- **Selection rule**: ToolTree augments UCT with a pre-execution prior, selecting actions by `UCT(s, a) = Q(s, a) + lambda * r_pre(s, a) * sqrt(ln N(s) / N(s, a))`, where `Q(s, a)` is the running mean of grounded rewards and `r_pre(s, a) in [0, 1]` estimates usefulness before calling the tool.
- **Expansion and pre-pruning**: from a leaf state `s_t`, expand only admissible actions whose schemas match the current context and whose predictive score passes `r_pre(s_t, a) >= tau_pre`; the retained set is further limited to `top-K` by `r_pre`, which shrinks branching before expensive tool execution.
- **Execution layer**: each selected child executes the actual tool/API call and appends `(a, o_{t+1})` to context `C_{t+1}`. ToolTree caches identical `(action, args)` calls within a rollout and records typed error tokens for persistent failures so downstream scoring and pruning stay explicit.
- **Post-evaluation and backpropagation**: after execution, an LLM judge assigns `r_post(s_t, a) = J(C_t, a, o_{t+1}) in [0, 1]`. The search updates `N(s, a) <- N(s, a) + 1` and `Q(s, a) <- Q(s, a) + (r_post(s_t, a) - Q(s, a)) / N(s, a)` along the traversed path.
- **Bidirectional pruning**: ToolTree prunes branches twice, first with `tau_pre` before execution and then with `tau_post` after grounded feedback. In experiments it uses `tau_pre = 0.3` and `tau_post = 0.4`, preventing budget from being spent on branches that are either obviously implausible or empirically unproductive.
- **Search budget and stopping**: the planner uses exploration constant `lambda = 1.4`, maximum rollouts `R_max = 60`, and early stopping when the best `Q` improves by less than `1e-3` for `10` consecutive rollouts. Open-set runs also use fixed API retrieval with `K = 20`.
- **Evaluation regimes**: closed-set planning uses GTA and [[m-and-ms]] with typed APIs and reports tool/argument/planning/execution F1; open-set planning uses [[toolbench]] and [[restbench]] with pass rate and win rate over retrieved APIs.

## Key Results

- On GTA with GPT-4o, ToolTree reaches `66.95` average score, beating the LATS/MCTS-style baseline (`64.78`) by `+2.17` and zero-shot (`57.78`) by `+9.17`.
- On m&m with GPT-4o, ToolTree scores `88.61` average, above LATS (`86.45`) and zero-shot (`80.58`); on GTA with GPT-4o-mini it also leads with `55.89` average.
- On ToolBench with GPT-4o, ToolTree achieves `69.04` average (`76.81` pass / `61.27` win), about `+2.49` over LATS (`66.55` average).
- On RestBench-TMDB with GPT-4o, ToolTree posts `74.50` average (`72.40` pass / `75.59` win), versus `71.35` for LATS; on RestBench-Spotify it reaches `71.36` average.
- Ablation shows the full system gives `76.44` accuracy at `18.2k` token cost; removing post-evaluation drops accuracy to `68.94`, while removing both evaluations drops it to `66.70` and increases token cost to `24.3k`.
- Scaling analyses show the strongest efficiency gains in the `16-64` step regime, and tool-library stress tests from `14` to `10,014` tools reduce average F1 only from `55.89` to `54.27` (`-1.62%` relative).

## Limitations

- The method still depends on an LLM judge for both `r_pre` and `r_post`; appendix analysis shows non-trivial judge error rates (`25.8%` for GPT-4o and `39.4%` for GPT-4o-mini), so planner quality is partly bottlenecked by evaluator quality.
- Efficiency gains come from pruning and caching, but ToolTree is still slower than greedy planners such as ReAct or zero-shot when wall-clock latency matters more than final accuracy.
- Most evidence is benchmark-based and uses fixed retrieval, shared tool schemas, and curated tool libraries; real production agents may face noisier schemas, unstable APIs, and harder long-horizon credit assignment.
- The paper is an arXiv preprint rather than a peer-reviewed conference version, and some broader claims, such as cross-domain generality over many tool ecosystems, rely on benchmark transfer rather than deployment studies.
