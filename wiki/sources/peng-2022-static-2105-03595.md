## Summary

The paper introduces HiTYPER, a hybrid Python type inference framework that combines rule-based static analysis with neural recommendations instead of trusting either source alone. Its core representation is a type dependency graph (TDG) over variable occurrences and typing expressions, which lets forward inference rules propagate sound types and backward rejection rules invalidate neural suggestions that violate typing constraints. To improve coverage on unresolved variables, HiTYPER queries a deep model only for hot type slots and corrects explicitly invalid user-defined type guesses via similarity against imported types. On Typilus and ManyTypes4Py benchmarks, the system improves overall exact-match accuracy over strong neural baselines while preserving static correctness and substantially improving rare-type and return-value inference.

## Problem & Motivation

Type inference for Python is difficult because purely static approaches are precise only when enough syntactic constraints are present, while neural models are more flexible but may suggest type assignments that violate typing rules or collapse on rare, long-tail user-defined types. The paper argues that these two families are complementary: static analysis offers correctness and frequency-insensitive reasoning, whereas deep models provide useful hints for underconstrained variables such as arguments and returns. The goal is therefore to build a framework that can integrate neural recommendations into a sound static inference loop instead of treating them as final predictions.

## Method

- **Type representation**: HiTYPER formalizes Python types as `` `theta ::= gamma | alpha[theta, ..., theta] | u | None | type` ``, separating built-in types, generic container types, and user-defined types so later rules can reason about them uniformly.
- **Type dependency graph (TDG)**: for each function it builds a directed graph `` `G = (N, E)` `` whose nodes are symbol nodes, expression nodes, branch nodes, and merge nodes. Each variable occurrence gets its own symbol node, similar to SSA, so dynamic reassignments can be tracked precisely.
- **Import analysis**: before graph construction, the system resolves local and global imports to collect user-defined classes and overloaded operators, which expands the available candidate set beyond built-in Python types.
- **Forward static inference**: expression nodes store typing judgments of the form `` `pi |- e : theta` ``. HiTYPER traverses the TDG forward and fires rules only when the premises of an operation are satisfied, including special handling for calls, class instantiation, container literals, slicing, and comprehensions.
- **Backward type rejection**: instead of accepting neural suggestions directly, the framework checks them against valid-type sets such as `` `tilde(theta)` `` and cross-input constraints like type agreement in `Add`. Rejected candidates are propagated backward through the TDG so downstream mistakes are removed before another forward pass.
- **Hot type slots**: among unresolved nodes, HiTYPER extracts a minimal dominating set of influential blank slots using the semi-NCA dominator algorithm, then queries neural models only for those slots because they unlock inference for many dependent variables.
- **Neural recommendation and correction**: the system can consume Top-1/3/5 predictions from Typilus or Type4Py. If a predicted user-defined type is not imported or defined locally, HiTYPER replaces it with the most similar valid type using Word2Vec embeddings trained with `` `d = 256` ``, context window `` `10` ``, skip-gram, and BPE subtokenization for OOV names.
- **Iteration**: static inference, type rejection, and neural recommendation repeat until the TDG reaches a fixed point or a maximum iteration limit, ensuring the final accepted assignments are those derivable after validation.

## Key Results

- On ManyTypes4Py, HiTYPER reaches overall Top-1 exact match `0.69`, beating Type4Py's `0.62`; Top-1 match-to-parametric improves from `0.66` to `0.77`.
- On Typilus's dataset, HiTYPER reaches overall Top-1 exact match `0.61` versus Typilus `0.54`; Top-5 exact match improves from `0.65` to `0.72`.
- Return-value inference improves most: on ManyTypes4Py Top-1 exact match rises from `0.49` to `0.60`, and on Typilus's dataset from `0.41` to `0.57`.
- For rare types, HiTYPER achieves Top-1 exact match `0.51` on ManyTypes4Py and `0.43` on Typilus's dataset, exceeding the corresponding DL baselines by more than `30%` relative.
- For user-defined types, HiTYPER raises Top-1 exact match from `0.29` to `0.49` over Type4Py and from `0.32` to `0.47` over Typilus.
- Considering only the static component, HiTYPER infers `3663` correct annotations on ManyTypes4Py and `3444` on Typilus's dataset, compared with Pyre Infer's `1275` and `1027`, and Pytype's `777` and `552`.
- In real GitHub projects, the system found `7` wrong human annotations across `6` repositories; `2` submitted fixes were approved by repository owners.

## Limitations

- The method is Python-specific and depends on Python typing rules, import resolution, and CPython/mypy conventions; the paper does not show portability to other dynamic languages.
- Around `10%` of outputs still disagree with human annotations, with subtype/generalization cases highlighted as a common failure mode even when the inferred type passes `mypy`.
- External APIs without explicit in-project implementations are not deeply inferred; the approach instead relies on typedef resources or neural predictions for those cases.
- Similarity-based correction for user-defined types is heuristic and depends on name similarity, Word2Vec training data, and BPE segmentation, which may be brittle for obscure naming schemes.
