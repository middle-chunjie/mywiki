## Summary

Repoformer proposes a selective retrieval framework for repository-level code completion that avoids cross-file retrieval when it is unnecessary or harmful. The core idea is to fine-tune a code language model to make a self-assessment decision after seeing left and right in-file context, emitting a special retrieval token only when cross-file context is likely to improve completion quality. The model is trained with self-supervision derived from performance differences between retrieval-free and retrieval-augmented generation, and is implemented as an extension of fill-in-the-middle prompting. Across RepoEval, CrossCodeEval, and the new CrossCodeLongEval benchmark, Repoformer improves completion quality over always-retrieve baselines while also reducing latency, showing that repository-level RAG should be adaptive rather than unconditional.

## Problem & Motivation

Existing repository-level code completion systems usually perform retrieval augmentation for every query, assuming cross-file context is always helpful. The paper shows this assumption is weak: for many completion instances, retrieval does not help and sometimes degrades predictions by injecting irrelevant code. Always retrieving also adds non-trivial latency, especially when retrieval cost is comparable to generation cost or worse. The paper therefore targets a more practical objective: learn a code LM that can decide whether repository-level retrieval is likely to improve the answer, abstain when in-file context is sufficient, and remain robust when retrieved context is noisy.

## Method

- **Task formulation**: each instance is `(X_l, X_r, Y, F)`, where `X_l` and `X_r` are left/right in-file context, `Y` is the missing code span, and `F` is the set of other repository files.
- **Base RAG setup**: an in-repository retriever `R` retrieves cross-file contexts `CC = {cc_1, ..., cc_k}` from `F`, and a code LM `M` generates `Y_hat` from `X_l`, `X_r`, and optional `CC`.
- **Retriever configuration**: the main setup uses Jaccard similarity with top-`k = 10` retrievals plus fragment alignment; index chunk size is `20` lines for line/API/chunk completion and `50` lines for function completion, with stride `chunk_size / 2`.
- **Prompting and truncation**: StarCoder-style fill-in-the-middle prompting includes both `X_l` and `X_r`; the paper left-truncates `X_l` to `1024` tokens, right-truncates `X_r` to `512` tokens, and truncates `CC` to `512` tokens.
- **Self-selective retrieval**: after a new token `<eof>`, the LM predicts either `<cc>` to trigger retrieval or an empty action `phi` to abstain. Inference supports greedy selection or threshold selection on `p(<cc> | X_l, X_r)`.
- **FIM-based implementation**: the whole decision-and-generation pipeline is cast as a single left-to-right fill-in-the-middle pass, allowing the same LM to decide retrieval and complete code in one framework.
- **Self-supervised labeling**: training labels are generated by comparing retrieval-free vs retrieval-augmented completion quality and setting `label = ES(Y_hat_RAG, Y) - ES(Y_hat_base, Y) > T`.
- **Losses**: the self-evaluation loss is `L_eval = -log p_M(<cc> | X_l, X_r)`, the generation loss is `L_gen = -log p_M(Y | X_l, X_r, CC)` when `label = true` and `-log p_M(Y | X_l, X_r)` otherwise, and the total objective is `lambda * L_eval + L_gen`.
- **Training data**: the authors filter `18k` Python repositories from The Stack, then create `240k` chunk-completion and `120k` function-completion instances; positive retrieval labels account for `30.18%` of chunk instances and `35.16%` of function instances.
- **Fine-tuning setup**: REPOFORMER trains StarCoderBase `1B/3B/7B/16B` with `lambda = 1.0`, maximum sequence length `2048`, learning rate `2e-5`, batch size `512`, `50` warmup steps, linear decay, and `2` epochs on `8 x A100 40GB`.

## Key Results

- Retrieval is often unnecessary: on RepoEval function/API completion, retrieved context improves only about `20%` of instances, leaves performance unchanged on more than `60%`, and harms roughly another `20%`.
- On RepoEval line completion with `3B` models, StarCoderBase ES improves from `72.12` (no retrieval) and `76.68` (always retrieve) to `79.02` with REPOFORMER SelectiveT.
- On RepoEval API completion with `3B` models, ES rises from `69.02` (no retrieval) and `72.62` (always retrieve) to `74.96` with REPOFORMER SelectiveT.
- On RepoEval function completion with `16B` models, unit-test pass rate improves from `34.73` (no retrieval) and `42.86` (always retrieve) to `44.18` with REPOFORMER SelectiveT.
- On CrossCodeLongEval chunk completion, `16B` REPOFORMER SelectiveT reaches `75.50` ES vs `71.90` for always retrieving; on long-form functions it reaches `58.93` ES vs `58.06`.
- Selective retrieval reduces latency: for `1B` RepoEval API completion, SelectiveG uses retrieval on only `18%` of instances for `69%` speedup, while SelectiveT uses retrieval on `61%` of instances and still yields `28%` speedup with better ES than always retrieve.
- A threshold around `0.4` can keep accuracy near the always-retrieve baseline while cutting latency by about `50%`, and abstention precision exceeds `0.8` on all tasks except RepoEval function completion (`0.78`).
- As a plug-in policy for larger generators, REPOFORMER-1B improves both latency and accuracy, e.g. StarCoderBase-16B API ES from `74.50` to `74.84` with `24%` speedup.

## Limitations

- The training label is based on edit similarity rather than semantic correctness, which the authors note likely weakens calibration for long-form function completion and unit-test-based evaluation.
- The main training and evaluation discussion centers on Python; multilingual transfer is promising but mostly deferred to appendix experiments.
- The retrieval setup is primarily single-iteration RAG with Jaccard-based sparse retrieval, so the paper does not fully explore more complex iterative or production retrieval stacks.
- RepoEval function-completion repositories are intentionally small for testing, which compresses retrieval overhead and may understate selective-RAG latency gains in larger repositories.
- The method learns a mostly uniform retrieval policy across repositories rather than repository-personalized policies conditioned on duplication structure or codebase organization.
