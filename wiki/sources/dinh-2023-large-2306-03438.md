## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

This paper defines buggy-code completion, where a code model must finish a partial program whose prefix may contain a latent semantic error, and shows that current code LLMs handle this setting very poorly. The authors formalize a potential bug as a small edit to a reference prefix that makes at least one otherwise valid completion fail, build two Python benchmarks covering synthetic operator flips and realistic rejected contest submissions, and evaluate CODEGEN plus INCODER under execution-based pass@k metrics. Across both datasets, pass@1 collapses from strong reference-prefix baselines to near-zero under buggy prefixes, and post-hoc mitigation via prefix removal, rewriting, or repair narrows but does not close the gap. The paper argues that practical code assistants must treat buggy draft code as noisy intent rather than blindly trusted context.

## Problem & Motivation

Existing code-completion evaluations usually assume the provided prefix is clean, but real development workflows often involve partially written and potentially incorrect code. That mismatch matters because a completion model may over-trust a buggy prefix and continue an invalid implementation rather than infer the user's intended program. The paper therefore asks whether high-performing code LLMs remain useful when the prompt contains semantic mistakes that are subtle enough to avoid syntax errors yet severe enough to make the completed program fail. The motivation is practical IDE assistance: a robust assistant should either repair or discount suspicious draft code instead of letting the bug poison completion quality.

## Method

- **Task definition**: buggy-code completion (`bCC`) takes a specification `h` and buggy prefix `s'`, and seeks a complete program `t` satisfying `h`; the output is not forced to preserve `s'` verbatim if deviating from the buggy prefix is necessary.
- **Potential bug formalization**: starting from a reference prefix `s` with some valid completion `c`, a small edit `e` yields `s'` such that `t' = s' :: c` fails the task, whereas `t = s :: c` succeeds.
- **buggy-HumanEval benchmark**: constructed from HumanEval by flipping semantic operators in canonical Python solutions using `ast` mutations such as `+ -> -`, `!= -> ==`, or `>= -> <`; keep only altered programs that fail tests, then split at lines `l <= i < L` to create buggy/reference prefix pairs. Final size: `1896` instances from `107` problems.
- **buggy-FixEval benchmark**: derived from FixEval and CodeNet by pairing rejected and accepted submissions from the same user, keeping only syntax-valid rejected programs that fail tests, and constraining character edit distance between prefixes to be `<= 20` after ignoring whitespace/comments. Final size: `292` pairs from `94` easy contest problems after manual filtering.
- **Naive completion baseline**: prompt the model with the problem statement plus partial code and sample `n = 100` completions; for buggy-HumanEval the prompt is an unfinished function with docstring, while buggy-FixEval uses a file-level triple-quoted problem statement followed by partial program text.
- **Evaluated models and decoding**: CODEGEN-350M/2B/16B-MONO use temperature sampling with `T = 0.6`; INCODER-1B/6B use top-`p = 0.95` with `T = 0.2`; output caps range from `200` to `600` tokens`.
- **Removal-then-completion**: drop the buggy partial code entirely and complete from the specification plus minimal scaffold, sacrificing useful prefix information in exchange for bug-free input.
- **Completion-then-rewriting**: first complete naively, then run a repair model (RealiT) over the full buggy program; this is especially aligned with operator and literal misuse patterns.
- **Rewriting-then-completion**: detect the most suspicious line before completion by scoring each token with a likelihood gap `score_token = p(x*) - p(x)` between the argmax token `x*` and observed token `x`, then aggregate non-zero token scores per line via `max` or `avg`; rewrite the selected line with INCODER-6B infilling and only then complete.
- **Evaluation**: execution-based `pass@k = 1 - C(n-c, k) / C(n, k)` with `n = 100` and `k in {1, 10, 100}`, macro-averaged by problem so heavily mutated items do not dominate.

## Key Results

- Naive completion breaks sharply under buggy prefixes: on buggy-HumanEval, reference-prefix pass@1 of `41.1%`-`54.9%` falls to `0.5%`-`3.1%`; on buggy-FixEval, `24.1%`-`37.8%` falls to `1.2%`-`4.3%`.
- CODEGEN-2B-MONO drops from `54.9%` to `3.1%` on buggy-HumanEval, which is worse than completing with no partial code at all (`9.3%`).
- All mitigation methods help relative to naive buggy-prefix completion. For CODEGEN-2B, buggy-HumanEval improves from `3.1%` to `9.3%` with removal-then-completion, `24.9%` with rewriting-then-completion, and `23.6%` with completion-then-rewriting.
- On buggy-FixEval, the best small-model result for CODEGEN-2B is removal-then-completion at `8.6%`, versus `7.2%` for rewriting-then-completion and `4.7%` for completion-then-rewriting, showing realistic bugs are harder than synthetic operator flips.
- Larger models still fail badly in the buggy setting: CODEGEN-16B-MONO goes from `68.9%` reference-prefix pass@1 on buggy-HumanEval to `4.4%` with naive buggy-prefix completion; mitigation only recovers to `24.3%` at best.
- Error analysis on CODEGEN-2B-MONO shows the dominant completion remains unchanged between clean and buggy prefixes in `90%` of failed instances and `93%` of affected problems, indicating systematic insensitivity to small semantic edits.
- Likelihood-based line localization reaches about `82%` accuracy on buggy-HumanEval and `53%` on buggy-FixEval; `max` aggregation works better on synthetic single-token bugs, while `avg` slightly wins on realistic bugs.
- Thresholding can partially balance repair aggressiveness: for rewriting-then-completion on buggy-FixEval with CODEGEN-2B, threshold `0.9` yields `37.0%` pass@1 on reference prefixes and `7.2%` on buggy prefixes, versus `8.6%`/`8.6%` when always rewriting at threshold `0`.

## Limitations

- The proposed fixes remain far below reference-prefix completion, so the paper diagnoses the problem more convincingly than it solves it.
- buggy-HumanEval uses synthetic single-operator mutations, which are controlled but simpler than many real development bugs.
- buggy-FixEval is more realistic, yet still tied to competitive-programming submissions with available tests, so transfer to broader software engineering settings is uncertain.
- Post-hoc mitigation can hurt normal completion when no bug exists, creating a calibration trade-off between aggressiveness and preservation.
- The study focuses on Python benchmarks and a small set of public Code-LLMs, so conclusions about other languages and newer code models need re-validation.
