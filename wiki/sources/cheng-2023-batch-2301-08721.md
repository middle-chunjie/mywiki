## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

This paper proposes batch prompting, a prompting scheme for hosted large language model APIs that packs `b` inference samples into one request while sharing the same few-shot demonstrations. Instead of issuing `N` separate calls for `N` test instances, the method issues roughly `N / b` calls and uses indexed question-answer slots so the batched outputs remain aligned and parseable. Under few-shot in-context learning, the authors argue token efficiency improves almost inverse-linearly with `b` because shared prompt tokens dominate total cost. Across 10 datasets and multiple OpenAI models, batch prompting cuts token and wall-clock cost by up to `5x` with six-sample batches while usually matching, and sometimes improving on, standard prompting accuracy.

## Problem & Motivation

LLM API inference is expensive when a system must answer large numbers of requests or benchmark thousands of evaluation examples. Standard few-shot prompting repeats the same demonstrations for every single test example, so prompt tokens, API overhead, and request-per-minute limits dominate both monetary and latency costs. The paper targets this inefficiency by asking whether one can reuse shared prompt context across multiple inference examples without changing the model, API, or training setup.

## Method

- **Problem setup**: given `N` test samples, standard few-shot prompting makes `N` separate API calls with `K` demonstrations plus one inference sample per call.
- **Batch prompting format**: group the `K` in-context exemplars into `K / b` batches and place `b` test questions in one prompt, formatting them as indexed pairs such as `Q[1] ... Q[b]` followed by aligned outputs `A[1] ... A[b]`.
- **Position identifiers**: each in-batch sample receives an explicit index token, which both helps the model keep question-answer correspondence and simplifies deterministic output parsing.
- **Token analysis**: the paper defines token efficiency as the fraction of tokens spent on generated outputs; under its simplified few-shot accounting, `η_standard = 1 / (K + 1)` and `η_batch = b / (K + b)`, so efficiency improves almost inverse-linearly with `b` when shared prompt tokens dominate.
- **Time analysis**: batch prompting reduces the number of API requests from `N` to `N / b`; the paper notes longer decoded outputs increase model-side generation cost, but argues API overhead and rate limits dominate practical latency in current hosted settings.
- **Main experimental setup**: use `K = 12` manually selected few-shot exemplars with Chain-of-Thought reasoning, temperature `0`, and batch sizes `b ∈ {2, 3, 4, 6}`.
- **Models**: the main study uses Codex (`code-davinci-002`), with transfer experiments on GPT-3 (`text-davinci-003`), GPT-3.5 (`gpt-3.5-turbo`), and GPT-4 (`gpt-4`).
- **Tasks and analyses**: evaluate 10 datasets spanning commonsense QA, arithmetic reasoning, and NLI/NLU; further analyze sample-grouping strategies, long-input behavior on WikiTQ, and compatibility with end-to-end, Chain-of-Thought, and program-based reasoning.

## Key Results

- **Cost reduction**: batch prompting yields up to `5x` lower token and wall-clock cost per sample at `b = 6`, with nearly inverse-linear savings as batch size increases.
- **Codex accuracy is usually preserved**: CSQA `77.2 -> 77.4`, GSM8K `55.7 -> 58.7`, MultiArith `97.5 -> 98.7`, and MNLI `65.3 -> 65.7` under the best batch configuration.
- **Some tasks degrade**: StrategyQA drops `73.3 -> 71.0`, SVAMP `83.7 -> 81.3`, AQuA `46.1 -> 42.1`, RTE `76.9 -> 74.7`, and SST-5 `51.3 -> 49.7`.
- **Transfer to newer chat models**: with GPT-3.5, batch prompting improves CSQA `72.9 -> 75.4`, GSM8K `71.0 -> 76.7`, AddSub `89.3 -> 92.0`, and RTE `77.6 -> 81.6`.
- **GPT-4 remains competitive but not uniformly better**: batch prompting improves CSQA `84.0 -> 86.0` and preserves AddSub at `99.0`, while GSM8K falls `96.0 -> 93.0`, SVAMP `98.0 -> 95.0`, and RTE `92.0 -> 90.0`.
- **Batch size tradeoff**: `b = 3` or `b = 4` usually offers a better cost-performance balance than `b = 6`, which often incurs the sharpest accuracy loss.

## Limitations

- The method is best when shared input tokens are much longer than generated outputs, so it is not a good fit for heavy-output tasks such as story generation.
- Performance can decline on harder tasks or longer inputs; the paper highlights notable drops on AQuA and on long-table WikiTQ settings.
- Similarity-based and diversity-based test-sample grouping do not outperform random batching in the reported experiments.
- Reported wall-clock gains depend on current API overhead and request-rate limits remaining a dominant bottleneck; if backend decoding cost becomes dominant, time savings shrink.
- The study does not provide a reliable predictive rule for when batch prompting will help or hurt, so users still need pilot experiments before large-scale deployment.
