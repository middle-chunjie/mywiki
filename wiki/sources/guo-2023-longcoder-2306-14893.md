## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

LongCoder proposes a sparse decoder-style code language model for long-context code completion, targeting file-level inputs that exceed the effective window of standard Transformers. The model combines sliding-window attention with two globally accessible mechanisms: bridge tokens, which summarize local spans for long-range access, and memory tokens, which expose imports and class/function definitions identified from code structure. The paper also introduces the Long Code Completion benchmark (LCC), whose examples are roughly `5×` longer than prior public completion sets. Across LCC, CodeXGLUE, and RepoBench XF-R, LongCoder outperforms prior `~124M–150M` baselines while keeping inference cost close to other sparse models, showing that structured sparse attention can recover useful global code dependencies without paying full quadratic attention cost.

## Problem & Motivation

Prior code completion benchmarks mostly evaluate short contexts, even though real GitHub source files are much longer: the paper reports an average of `1,305` source tokens per Python file and `2,090` tokens after tokenization, with `41%` / `24%` of files exceeding `1,024` / `2,048` tokens. Vanilla Transformer attention scales as `O(n^2)` and becomes expensive on such inputs, while truncating context loses imports, definitions, and other globally scoped statements that are often required for correct completion. The paper therefore asks how to keep file-level context available during autoregressive completion while retaining near-linear inference cost.

## Method

- **Base architecture**: LongCoder is a sparse pre-trained Transformer for autoregressive code completion. It keeps the standard causal attention computation `Attention(Q, K, V) = softmax(QK^T / sqrt(d_k) + M)V`, but replaces the dense mask with sparse patterns tailored to long code.
- **Window attention**: each token attends only within a left context window `w`, with mask `M_window[i,j] = 0` if `i - j <= w`, else `-inf`. This yields `O(n × w)` complexity and exploits the empirical observation that a window of `256` covers more than `90%` of attention mass in CodeGPT.
- **Bridge attention**: the model inserts `m` bridge tokens across the sequence. A bridge token attends to its preceding chunk of roughly `ceil(n / m)` tokens, and all later tokens can attend to that bridge token. This reduces long-range access to at most `2` hops while keeping complexity `O(m × n) ≈ O(n)`.
- **Global attention / memory tokens**: using `tree-sitter`, the model parses code into an [[abstract-syntax-tree]] and marks line-feed positions belonging to imports and class/function/structure definitions as globally accessible positions `G`, with mask `M_global[i,j] = 0` if `j in G` and `i >= j`. This encodes code heuristics about globally scoped statements.
- **Unified sparse mask**: the final attention mask is `M = max(M_window, M_bridge, M_global)`, combining local context, learned span summaries, and syntax-guided global access.
- **Training/inference setup**: non-sparse baselines use maximum context `512`, while sparse models use `4,096`. To keep the local comparison fair, LongCoder sets `w = 512`, bridge-token budget `m = 16`, and global-token budget `k = 64`.
- **Initialization and objective**: LongCoder is initialized from UniXcoder parameters and pre-trained on CodeSearchNet with the same [[next-token-prediction]] objective as the compared baselines.
- **Fine-tuning details**: Adam optimizer, batch size `16`, learning rate `2e-4`, early stopping, and up to `10` epochs. During inference, it retains a cache of at most `592` past key/value tokens to control memory use.

## Key Results

- **LCC main benchmark**: LongCoder (`150M` params) achieves Python `17.88` EM / `55.07` Edit Sim, Java `26.42` / `61.21`, and C# `31.34` / `64.37`, outperforming Longformer and BigBird while using `211M` inference memory and `812 ms` runtime per example.
- **Relative gains over other sparse baselines**: compared with Longformer, LongCoder improves EM by `+1.09`, `+1.62`, and `+1.59` on Python/Java/C#, and improves Edit Sim by `+4.00`, `+5.18`, and `+6.14`.
- **CodeXGLUE short-context benchmark**: LongCoder still reaches SOTA on shorter inputs, with PY150 `43.77` EM / `73.37` Edit Sim and JavaCorpus `33.13` / `67.32`, slightly above UniXcoder.
- **RepoBench XF-R**: with cross-file context, LongCoder reaches Python `21.4` EM / `59.7` Edit Sim and Java `19.7` / `60.1`, again beating UniXcoder (`18.0` / `53.4`, `16.5` / `50.0`).
- **Ablations**: removing memory tokens drops average EM by about `1%`; removing bridge tokens drops Edit Sim by about `3%`; keeping only in-window context nearly collapses performance back toward UniXcoder.
- **Dataset scale**: LCC uses `100k` train, `10k` dev, and `10k` test examples per language, with average test context lengths `1993.3` (Python), `1841.4` (Java), and `1970.5` (C#).

## Limitations

- The paper explicitly notes that LongCoder is small relative to deployment-scale code models such as OpenAI Codex (`12B` in the comparison table), so it does not test whether the sparse design still dominates at much larger scale.
- Pre-training is limited to CodeSearchNet rather than a much larger code corpus, so some of the observed gap could change under stronger large-scale pretraining.
- Benchmark reliability is imperfect because LCC and several baseline datasets are sourced from GitHub, creating a real [[data-contamination]] risk for heavily pretrained code models.
- The method mainly addresses file-level context; repository-level or multi-file completion is discussed as future work rather than directly solved.
- The global memory heuristic depends on syntax parsing and line-level markers for imports/definitions, which may miss other semantically important long-range dependencies.
