## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

The paper proposes CodeScore, an execution-aware code evaluation metric learned from large language models rather than computed by direct matching or full program execution. Its core idea is to train UniCE on generated code paired with reference code and/or natural-language requirements so the model predicts continuous `PassRatio` and binary `Executability`, approximating functional correctness under three input regimes: Ref-only, NL-only, and Ref&NL. On APPS-Eval, MBPP-Eval, and HE-Eval, CodeScore substantially improves correlation with execution-derived ground truth over BLEU, CodeBLEU, BERTScore, COMET, and CodeBERTScore, while being orders of magnitude cheaper than large-scale execution-based evaluation. The work is notable because it reframes code evaluation as learned execution modeling rather than surface-form comparison.

## Problem & Motivation

Match-based code evaluation metrics such as BLEU, Accuracy, and CodeBLEU mainly compare lexical or structural overlap with a reference, but code quality is fundamentally about whether the generated program behaves correctly. This mismatch is severe when multiple implementations are functionally equivalent yet syntactically different. Pure execution-based evaluation is closer to the target notion of correctness, but it depends on extensive test cases, incurs high CPU and I/O cost, and raises security concerns when executing model outputs. The paper also argues that Ref-only evaluation is too narrow because some realistic settings provide only natural-language requirements or benefit from using both requirements and references. The goal is therefore to learn a metric that tracks functional correctness, generalizes across input formats, and avoids repeated costly execution at inference time.

## Method

- **Target signals**: UniCE learns a continuous `PassRatio = (1 / |C_p|) Σ_{c ∈ C_p} 𝕀{Eval(g_p, I_{p,c}) = O_{p,c}}` and a binary `Executability ∈ {0, 1}` for generated code.
- **Input formats**: the model supports three unified evaluation inputs, `g + r`, `g + n`, and `g + r + n`, where `g` is generated code, `r` reference code, and `n` the natural-language requirement.
- **Outputs**: given unified input `x`, UniCE predicts `(CodeScore, Exec) = UniCE(x)` with `CodeScore ∈ [0,1]` and `Exec` estimating whether the code is executable on all provided inputs.
- **Loss design**: training minimizes `L = L_C + L_E`, where `L_C = (CodeScore - PassRatio)^2` and `L_E = -log p(Exec | Executability)`. This couples regression on functional quality with classification of compile/runtime viability.
- **Unified multi-task training**: a shared objective `L^Uni = L^Ref + L^NL + L^{Ref+NL}` trains the same model across Ref-only, NL-only, and Ref&NL settings instead of fitting separate evaluators.
- **Representation learning**: UniCE uses a code-pretrained LLM to encode the sequence, pools hidden states with layer-wise attention `e_t = γ Σ_{k=1}^l e_t^k h^k`, and represents the full input with the first-token embedding `e_first` rather than mean pooling.
- **Prediction head**: `e_first` is fed to a 3-layer feed-forward network with Tanh activations and output widths `3072 -> 1024 -> 2`.
- **Base model and formatting**: the implementation uses [[code-understanding]] model UniXcoder with input template `"[CLS] g [SEP] r [SEP] n [SEP]"`, truncation length `1024`, Adam optimizer, learning rate `0.001`, and `5` epochs.
- **Training data**: UniCE is trained on APPS-Eval, with optional MBPP-Eval fine-tuning in specific experiments, and reported results are averaged across `5` runs on a single `Tesla A100-PCIe-40G`.

## Key Results

- On APPS-Eval, CodeScore with Ref&NL input and `L^Uni` reaches `τ = 0.5275`, `r_s = 0.7040`, `r_p = 0.7210`, and `MAE = 0.1044`, improving over the best non-CodeScore metric by `40.56%`, `55.07%`, and `58.87%` on the three correlation coefficients.
- On MBPP-Eval, CodeScore with `L^Uni` reaches `r_s = 0.6027` in the Ref&NL setting after fine-tuning; on HE-Eval it reaches `r_s = 0.6597` zero-shot, outperforming other match-based and embedding-based metrics in both settings.
- On APPS-Eval, CodeScore inference at `44.2×` normalized runtime is far cheaper than execution-based evaluation with `181` tests per task at `20.7k×`; analogous execution baselines on MBPP-Eval and HE-Eval cost `28.7k×` and `21.7k×`.
- The Exec component achieves above `90%` on precision, F1, and accuracy even in zero-shot transfer to HE-Eval, indicating that executability can be learned robustly.
- Human evaluation on `100` HE-Eval samples gives CodeScore `3.4 ± 0.3`, compared with `2.2 ± 0.7` for CodeBERTScore, `2.1 ± 0.5` for CodeBLEU, and `1.3 ± 0.4` for BERTScore.
- Dataset construction is sizable: APPS-Eval has `267,162/33,395/33,395` train/dev/test examples, MBPP-Eval has `15,679/3,000/3,000`, and HE-Eval has `4,221` test examples.

## Limitations

- The approach still requires collecting generated code, references, NL descriptions, and sufficiently rich test cases to produce training labels, so it is not free from data construction cost.
- The paper explicitly notes that CodeScore is mainly validated for function-level Python code, leaving broader languages, larger program scopes, and more realistic dependency settings for future work.
- Although much cheaper than exhaustive execution-based evaluation, CodeScore still adds model inference cost and latency compared with lightweight match-based metrics.
- The quality claims depend on execution-derived supervision from extended test cases; if those tests are weak or systematically biased, the learned evaluator can inherit that limitation.
