## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

This paper proposes Iter-RetGen, a retrieval-augmented large language model workflow that alternates whole-output generation and retrieval across iterations instead of interleaving retrieval actions inside a single reasoning trace. The key idea is that an earlier model output, even if imperfect, exposes missing subgoals and can therefore act as a better retrieval context for the next round. Using `text-davinci-003` with 3-shot chain-of-thought prompting and dense retrieval, the method improves over prior retrieval-augmented baselines on multi-hop question answering and commonsense reasoning while remaining competitive on fact verification. The authors also show that first-round generations can supervise retriever adaptation via reranker-to-retriever distillation, further improving accuracy and reducing the number of retrieval-generation iterations needed.

## Problem & Motivation

One-shot retrieval often fails when the original question underspecifies the latent sub-questions needed to collect supporting evidence, especially in multi-hop QA and other tasks with complex information needs. Prior methods such as ReAct and Self-Ask address this by interleaving retrieval with partial generation, but that structure constrains generation, prevents the model from jointly processing all retrieved evidence, and increases both API and retrieval overhead. The paper targets the semantic gap between the user question and the evidence actually needed for a correct answer, asking whether a model's own intermediate output can serve as a better retrieval cue without forcing the decoder into a rigid action-observation format.

## Method

- **Iterative loop**: given question `q`, corpus `D`, and `T` iterations, Iter-RetGen retrieves with `q` in the first round and with `y_(t-1) || q` afterwards, then generates `y_t = M(prompt(D_(y_(t-1)||q), q))` for `1 <= t <= T`; the final answer is `y_T`.
- **Generation-augmented retrieval**: the previous output `y_(t-1)` is treated as a specific context that can bridge semantic gaps between the original question and relevant passages, rather than issuing explicit sub-queries or retrieval actions inside the decoding process.
- **Retrieval-augmented generation**: each iteration uses chain-of-thought prompting with all retrieved passages prepended to the question, so the model reasons over the full retrieved set at once instead of consuming evidence piecemeal.
- **Dense retriever**: passage ranking uses inner-product similarity `s_theta(q,d) = <E(q; theta_q), E(d; theta_d)>`; experiments retrieve top-`k = 5` paragraphs per query from Wikipedia corpora aligned to each benchmark.
- **Retriever adaptation**: a reranker `s_phi(q,d)` with access to `y_1 || q` provides soft targets, and the retriever query encoder is optimized with `theta_q* = arg min KL(P_phi(.|y_1,q), P_theta(.|q))`, where both distributions are temperature-scaled over the retrieved set.
- **Prompting / inference**: the main backend is `text-davinci-003` under a `3`-shot setup with greedy decoding; ReAct and Self-Ask are allowed up to `5` retrieval interactions, while Iter-RetGen varies `T` from `1` to `7`.
- **Adaptation hyperparameters**: retriever distillation uses TART as reranker, trains on `9,000` questions with `1,000` validation questions for HotPotQA and Feverous, uses batch size `32`, learning rate `1e-5`, and stops within `1,000` steps at the lowest distillation loss.

## Key Results

- **HotPotQA**: Iter-RetGen improves `Acc†` from `64.8` for Self-Ask to `71.2` at `T = 2` and `73.4` at `T = 4`; answer recall of retrieved paragraphs rises from `49.5` in iteration `1` to `66.1` in iteration `2`.
- **2WikiMultiHopQA**: Self-Ask reaches `55.9 Acc†`, while Iter-RetGen reaches `58.1` at `T = 2` and peaks at `59.4` at `T = 6`; answer recall jumps from `29.0` to `45.2` between the first two iterations.
- **MuSiQue / Bamboogle**: on MuSiQue the method is competitive with Self-Ask (`41.0` vs. `42.9 Acc†` at `T = 2`), while on Bamboogle it improves from `54.8` to `59.2` at `T = 2` and `60.8` at `T = 7`.
- **Feverous / StrategyQA**: Iter-RetGen is competitive on Feverous (`68.8 Acc†` at `T = 2` vs. `70.7` for Self-Ask) and stronger on StrategyQA (`73.0` at `T = 2`, `74.1` at `T = 7`, versus `70.2` for Self-Ask).
- **Efficiency**: Iter-RetGen with `T = 2` uses only `2` LLM API calls and `10` retrieved paragraphs total, versus roughly `3.0-3.2` API calls and `14.8-16.0` retrieved paragraphs for Self-Ask across datasets.
- **Retriever adaptation / evaluation**: distilling retrieval with `y_1` lifts HotPotQA `Acc†` from `71.2` to `75.7` and Feverous from `68.8` to `69.5`; when EM and `Acc†` disagree on sampled QA outputs, `Acc†` is judged correct `97.5%` of the time.

## Limitations

- The study relies on a fixed black-box LLM (`text-davinci-003`), so some gains may depend on the promptability and failure modes of that particular model rather than the iteration pattern alone.
- Performance usually improves most at the second iteration and then plateaus, so extra rounds add cost with diminishing returns.
- Error analysis shows that `65%` of sampled HotPotQA failures are retrieval-related, and `76.9%` of those retrieval failures stem from completely wrong first-round reasoning that misleads the second-round retrieval.
- The paper does not evaluate long-form generation, where finer-grained retrieval may be necessary, and it leaves prompt-specific or gradient-based optimization of the generation policy for future work.
