## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

ClassEval introduces a manually constructed benchmark for class-level Python code generation, targeting a harder setting than function-level suites such as HumanEval and MBPP. The benchmark contains 100 tasks and 412 methods, each specified by a class skeleton and evaluated with both method-level and class-level tests, created through roughly 500 person-hours of manual work to reduce data leakage and improve test sufficiency. Using ClassEval, the authors compare 11 large language models under holistic, incremental, and compositional generation strategies. Results show a sharp drop from method-level to class-level coding performance, with GPT-4 and GPT-3.5 remaining clearly ahead, while most open models benefit more from method-by-method generation than from generating the whole class in one shot.

## Problem & Motivation

Existing code-generation benchmarks mostly evaluate standalone function or statement synthesis, which underestimates the difficulty of realistic software tasks where methods share fields, call one another, and must respect class-level interfaces. The authors argue that method-level performance therefore fails to capture whether an LLM can generate larger, interdependent code artifacts. ClassEval is proposed to close this gap with manually designed class-level tasks, richer specifications, and stronger tests that can expose failures in dependency handling, long-context understanding, and multi-method consistency.

## Method

- **Benchmark target**: construct the first manual benchmark for class-level Python code generation, covering `100` classes and `412` methods with approximately `500` person-hours of annotation effort.
- **Task sourcing**: tasks are drawn from three channels: revisiting existing benchmarks, manually exploring `PyPI`, and author brainstorming; excluded categories include network programming, GUI design, data visualization, system programming, and concurrent programming.
- **Class skeleton format**: each task provides a structured class skeleton with mandatory elements such as class name, class description, method signatures, and method-level functional descriptions, plus optional items such as imports, constructor, parameter/return descriptions, and example I/O.
- **Construction pipeline**: four manual stages are used: task selection, class skeleton construction, test construction, and canonical solution construction, with double-checking and disagreement resolution across participants.
- **Dependency design**: the benchmark intentionally includes diverse within-class dependencies, including library dependency, field dependency, method dependency, and standalone methods, rather than collections of independent functions.
- **Testing protocol**: each task has both method-level tests and class-level tests; method-level tests require at least `5` cases per method, class-level tests must invoke every method at least once, and all tests are constrained to run within `5` seconds using `unittest`.
- **Studied models**: `11` LLMs are evaluated, including GPT-4, GPT-3.5, WizardCoder, Instruct-StarCoder, Instruct-CodeGen, CodeGeeX, InCoder, PolyCoder, SantaCoder, Vicuna, and ChatGLM.
- **Generation strategies**: the study compares holistic generation (whole class at once), incremental generation (method-by-method with previously generated bodies), and compositional generation (method-by-method with signatures only).
- **Prompting setup**: instruction-following models receive a system prompt plus strategy-specific task instructions; non-instruction models are prompted with code context only.
- **Metrics**: correctness is measured with `Pass@k = E[1 - C(n-c, k) / C(n, k)]`, using `n = 5` samples and `k in {1, 3, 5}`; dependency fidelity is measured with `DEP(M) = Σ_i G_i(M) / Σ_i S_i(M)` and `DEP(F) = Σ_i G_i(F) / Σ_i S_i(F)`.
- **Implementation details**: nucleus sampling uses `temperature = 0.2` with default `top_p`; greedy decoding uses `temperature = 0`; all models are evaluated under a maximum context window of `2048` tokens on `8 x A800-80G` GPUs for open models.

## Key Results

- **Benchmark scale**: ClassEval contains `100` classes and `412` methods, with average code length `45.7` lines and average class-skeleton length `259.3` tokens, versus `67.7` for HumanEval and `14.5` for MBPP.
- **Test sufficiency**: canonical-solution coverage reaches `99.7%` statement coverage and `98.2%` branch coverage, with averages of `8.0` method-level tests and `33.1` class-level tests per task.
- **Dependency profile**: `314` methods (`76.2%`) are class-level dependent; specifically, `269` methods (`65.5%`) use field dependencies, `107` (`26.0%`) use method dependencies, `89` (`21.7%`) use library dependencies, and only `58` (`14.1%`) are standalone.
- **Best overall models**: with nucleus sampling, GPT-4 achieves class-level `Pass@1/3/5 = 37.6/41.3/42.0%` and method-level `62.8/67.4/68.5%`; GPT-3.5 follows with class-level `29.6/34.9/36.0%` and method-level `50.4/59.0/61.1%`.
- **Gap from function-level benchmarks**: on greedy `Pass@1`, GPT-4 drops from `85.4%` on HumanEval methods to `62.5%` method-level and `37.0%` class-level on ClassEval; GPT-3.5 drops from `68.9%` to `52.5%` method-level and `27.0%` class-level.
- **Strategy effect**: holistic generation is best only for GPT-4 and GPT-3.5, improving class-level `Pass@5` by roughly `6-9%` and `4-14%` respectively over method-by-method strategies; other models generally do better with incremental or compositional generation.

## Limitations

- The benchmark is limited to `100` tasks and only Python, so conclusions may not generalize to larger benchmark scales, other programming languages, or more complex multi-file settings.
- Manual construction reduces leakage risk but still introduces human subjectivity in task design, prompt selection, and canonical-solution choices.
- The evaluation context is capped at `2048` tokens for all models, which may understate the capability of models with larger native context windows.
- Only models released since `2022` and mostly between `1B` and `20B` parameters are included, so the comparison does not cover the full model landscape.
- The study shows dependency-generation weaknesses but does not provide finer-grained mechanistic analysis beyond aggregate error types such as `AttributeError`, `TypeError`, and occasional `KeyError`.
