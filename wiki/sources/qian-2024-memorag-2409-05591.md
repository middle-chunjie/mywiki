## Summary

MemoRAG reframes long-context processing as memory-enhanced retrieval rather than either direct full-context generation or conventional query-to-passage RAG. The core idea is to build a compact global memory of the entire context with a lightweight memory model, then use that memory to draft answer-like clues that are more semantically aligned with the evidence than the original query. Those clues drive retrieval over the raw long context, and a stronger generator produces the final response from retrieved evidence. The paper combines a KV-compression-based memory architecture with staged training and reports broad gains over full-context long LLMs, standard dense-retrieval RAG, and advanced RAG baselines on LongBench, InfiniteBench, and a diverse UltraDomain benchmark.

## Problem & Motivation

Standard long-context processing faces a poor tradeoff: directly feeding very long inputs into LLMs is expensive in inference time and GPU memory, while conventional [[retrieval-augmented-generation]] assumes a query whose wording is already close to the answer-bearing evidence and a knowledge base that is easy to chunk and index. The paper argues that many long-context tasks violate both assumptions. Complex QA requires integrating dispersed evidence, while summarization and other high-level tasks may not provide a directly searchable query at all. MemoRAG is motivated by a human-like pipeline: form a global memory of the document first, recall task-relevant clues from memory, then retrieve precise supporting details from the full context before generating the final answer.

## Method

- **Overall pipeline**: MemoRAG replaces standard RAG retrieval `E = Γ(q, C)` with memory-guided retrieval `Y = Θ(q, E | θ)`, `E = Γ(y, C)`, `y = Θ_mem(q | θ_mem)`, where `y` is a draft answer or clue generated from compact memory rather than from the raw long context.
- **Dual-system architecture**: a light but long-range memory model forms global memory over the whole context, while a heavier generator produces the final answer from the retrieved evidence; by default the paper uses the memory model's base LLM as generator for parameter efficiency, but the main experiments use `Phi-3-mini-128K-instruct` to avoid truncation.
- **Compact global memory**: after each context window of length `l`, the model inserts `k` memory tokens with compression ratio `β = l / k`, where `β ∈ {4, 8, 16, 32, 64}` in experiments. Instead of retaining the full raw-token KV states, it accumulates only memory-token KV caches, yielding approximate `β×` GPU-memory reduction.
- **Memory-token attention**: the model introduces separate memory-token projections `W_Q^m`, `W_K^m`, `W_V^m`; attention is computed over combined regular-token and memory-token keys/values, and only memory-token caches are preserved across windows.
- **Scalability claim**: because the preserved memory state is compressed, a `128K`-context model can in principle process up to `8M` tokens when `β = 64`, even though performance declines gradually as compression becomes more aggressive.
- **Training stage 1, pretraining**: only the newly initialized memory-token parameters are trained while the base LLM stays frozen. The loss is next-token prediction conditioned on accumulated memory, `L_pre = -Σ_t log P(x_t | x_cache^m, x_{1:t-1})`. Appendix A reports `2B` pretraining tokens, context window `2048`, `1` epoch, batch size `8`, and learning rate `5e-5`.
- **Training stage 2, supervised clue learning**: the memory model is then optimized to generate task-specific clues with `L_SFT = -Σ_t log P(y_t | x_cache^m, q)`. The SFT set contains `17,116` samples of length `4K-64K`; training runs for `2` epochs with batch size `8` and learning rate `1e-5`.
- **Training stage 3, RLGF**: reinforcement learning with generation feedback ranks clue sets by their contribution to end-to-end answer quality, then optimizes a preference-style loss `L_RLGF = Σ_(y+, y-) max(0, 1 - R(y+) + R(y-))`. The paper builds `2,000` preference instances from the SFT data.
- **Retriever/generator configuration**: in the main setup, MemoRAG uses [[bge-m3]] as retriever with top-`3` hits, chunks the context with maximum length `512`, trains the memory model on `Mistral-7B-Instruct-v0.2-32K`, and evaluates on `8` `NVIDIA A800-80G` GPUs.

## Key Results

- On the main benchmark average, MemoRAG reaches `40.2`, outperforming Full-context inference (`35.0`), MInference (`33.3`), SelfExtend (`30.1`), BGE-M3 RAG (`29.7`), GraphRAG (`29.4`), and RQ-RAG (`30.1`).
- On multi-hop QA, MemoRAG improves `2Wiki` from `38.1` (Full) to `54.1` and `HotpotQA` from `48.1` (Full) / `42.1` (BGE-M3) to `54.8`.
- On non-QA and long-book tasks, MemoRAG improves `MultiNews` from `24.9` to `26.3`, `GovReport` from `32.6` to `32.9`, `En.SUM` from `13.0` to `15.7`, and `En.QA` from `15.2` to `22.9`.
- On specialized UltraDomain tasks, MemoRAG raises `Fin` from `47.8` (Full) to `51.5`, `Legal` from `46.5` (Full) / `39.6` (GraphRAG) to `51.0`, and `Misc` from `48.7` to `55.6`.
- In the legal case study, MemoRAG improves answer F1 from `0.36` for standard RAG to `0.83` by generating clue phrases such as the definition and deadline content of the "Outside Date."
- For `128K` contexts, both MemoRAG and standard RAG stay under `60 GiB` GPU memory in the efficiency analysis, while long-context full-input LLMs consume substantially more memory because of the large raw-token KV cache.

## Limitations

- MemoRAG depends on a specialized memory module with new trainable parameters and multi-stage optimization; Appendix A notes that the newly initialized parameters can reach about `1.1B` for a `Qwen2-7B` backbone, so this is not a zero-cost drop-in modification.
- Its indexing stage is slower than standard RAG because it must first form global memory before retrieval, even though it is still faster than direct long-context prefilling and GraphRAG.
- Retrieval remains chunk-based and approximate: the main setup uses `512`-token chunks, top-`3` retrieval, and dense retrieval over the raw context, so failures in clue generation or chunk retrieval can still propagate to final answers.
- Performance falls as the compression ratio `β` increases; the paper reports graceful degradation and stabilization around `β = 32`, but higher compression still removes semantic detail.
- The empirical evidence is broad but benchmark-centric; many gains are shown on curated long-context evaluations rather than interactive production systems with noisy user queries and continuously changing corpora.
