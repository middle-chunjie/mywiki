## Summary

The paper introduces TOOLMAKER, an agentic framework that converts scientific papers plus their linked code repositories into executable tools that downstream LLM agents can call. Instead of synthesizing small tools from scratch, TOOLMAKER installs a repository, snapshots the resulting environment, explores the codebase, plans an implementation, and then iteratively debugs a generated Python wrapper through a closed-loop self-correction cycle. To evaluate this setup, the authors release TM-BENCH, a benchmark of 15 realistic scientific tasks that require dependency installation, repository exploration, and reusable tool construction rather than single-shot code generation. TOOLMAKER solves `12/15` tasks and substantially outperforms an OpenHands baseline, suggesting that explicit environment management and iterative execution feedback are decisive for autonomous tool creation.

## Problem & Motivation

Existing LLM agents are bottlenecked by the tools humans pre-build for them, which limits applicability in domains such as life sciences and medicine where many tasks require highly specialized software. Prior tool-creation work mostly builds simple tools from scratch and typically lacks direct operating-system interaction, making it poorly matched to real scientific workflows that depend on external repositories, heavyweight dependencies, models, and configuration logic. The paper argues that the open-science trend of releasing code alongside papers creates an opportunity: instead of inventing tools de novo, an agent should autonomously install, adapt, and expose existing research software as reusable tool interfaces. This reframes tool creation as repository-grounded software integration under execution feedback.

## Method

- **Tool definition**: each task is specified by a minimal interface containing a short task description, a GitHub repository URL, and required input arguments with example values; the target artifact is an executable Python function plus an environment definition.
- **State formalization**: the workflow state is `s = (h, e)` where `h` is conversation history and `e` is environment state; components operate over `S = H x E` and return messages, observations, or updated states.
- **Component types**: TOOLMAKER decomposes execution into LLM calls, environment interactions, and higher-level agents; LLM-only steps update `h`, while environment actions map `e -> (e', o)`.
- **Action space**: the environment toolset is `A = {RUN-BASH-COMMAND, LIST-DIRECTORY, READ-FILE, WRITE-FILE, BROWSE, GOOGLE-DRIVE-LIST-FOLDER, GOOGLE-DRIVE-DOWNLOAD-FILE, RUN-IMPLEMENTATION}` so the agent can inspect files, install dependencies, and execute candidate tools.
- **Two-stage workflow**: TOOLMAKER first performs environment setup, then tool implementation. The setup stage installs the repository from a clean `python:3.12` image and records all write actions so they can be replayed as a bash script or Dockerfile.
- **Repository setup**: an `INSTALL_REPOSITORY` agent clones the repo, reads documentation, downloads non-dataset dependencies such as model weights, and produces a reproducible installed snapshot `e_bar`.
- **Implementation stage**: a separate `EXPLORE` agent examines the installed repo, then a `PLAN` step creates high-level pseudocode and an `IMPLEMENT` step writes the first candidate tool. The paper states that generic LLM calls use `gpt-4o-2024-08-06`, while planning and implementation use `o1-mini-2024-09-12`.
- **Closed-loop self-correction**: after implementation, TOOLMAKER restores the installed snapshot `e_bar`, restores conversation state `h_bar`, runs the candidate tool on the example invocation, asks the model to assess success, and if needed invokes `DIAGNOSE_ERROR`, `REIMPLEMENT`, and `SUMMARISE`; summaries are accumulated in `sigma` across iterations.
- **Execution environment**: the environment is a Docker container controlled via an HTTP server; Docker checkpointing is used to roll back to prior states, which makes iterative debugging reproducible and isolates the host system.
- **Benchmark design**: TM-BENCH contains `15` tasks spanning pathology, radiology, omics, NLP, tabular learning, and 3D vision, with `42` held-out test invocations and `124` unit tests. Tasks are manually vetted to ensure each repository can in fact be installed and used.

## Key Results

- TOOLMAKER achieves `12/15` correct tools (`80%`), versus `3/15` (`20%`) for [[openhands]] on TM-BENCH.
- At the invocation level, TOOLMAKER passes `37/42` test invocations and `116/124` unit tests; OpenHands passes `9/42` invocations and `31/124` tests.
- Average tool-creation cost is `$0.94` with `21.8` actions for TOOLMAKER, compared with `$0.15` and `7.5` actions for OpenHands, indicating that the stronger method spends more compute but converts it into substantially better correctness.
- On `stamp_train_classification_model`, TOOLMAKER needs `9` self-correcting iterations and `33` actions to discover and implement the multi-step pipeline of feature extraction followed by classifier training.
- Adding task-specific paper summaries lowers TOOLMAKER's average cost/actions to `$0.71` and `18.1`, but slightly reduces end performance to `11/15` tools, `34/42` invocations, and `113/124` tests.
- Replacing `gpt-4o`/`o1-mini` with `o3-mini` drops TOOLMAKER to `9/15` correct tools and OpenHands to `1/15`, showing the method is sensitive to backbone quality.
- The partially failed `esm_fold_prediction` case still passes `2/3` test invocations; the failure is traced to a masked-token pattern absent from the example invocation, suggesting benchmark examples materially affect synthesis success.

## Limitations

- The framework assumes repositories are reasonably well structured, documented, and installable; poorly maintained open-source projects remain a major failure mode.
- TM-BENCH is manually curated so that all tasks are solvable in principle, which may make it cleaner than arbitrary real-world repository selection.
- Passing `124` unit tests does not guarantee robustness on unseen edge cases, especially in high-stakes scientific or clinical deployments.
- TOOLMAKER covers in-silico tool creation, not the physical experimentation bottlenecks that still constrain fully autonomous scientific discovery.
- Reproducibility can still break if upstream repositories are deleted, renamed, or force-pushed despite commit pinning.
- The authors explicitly note misuse risks in life sciences, including the possibility of automating harmful biochemical tooling without sufficient oversight.
