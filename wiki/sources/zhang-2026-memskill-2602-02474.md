## Summary

MemSkill reframes LLM agent memory operations as a learnable and evolvable set of *memory skills* rather than static hand-coded procedures (add/update/delete/skip). The system comprises three components: a lightweight MLP *controller* that selects a Top-K subset of skills from a shared *skill bank* conditioned on the current interaction span and retrieved memories; an LLM-based *executor* that generates skill-guided memory updates in one pass; and a *designer* that periodically mines hard-case failures, clusters them, and uses an LLM to refine existing skills or propose new ones. Training alternates between controller optimization via PPO and skill-bank evolution via the designer, forming a closed-loop that improves both skill selection and the skill set itself. Experiments on LoCoMo, LongMemEval, HotpotQA, and ALFWorld consistently outperform strong memory baselines across conversational and embodied settings.

## Problem & Motivation

Most LLM agent memory systems rely on fixed, hand-designed operation primitives (e.g., add/update/delete/skip) and heuristic modules that decide what to store and how to revise it. These static designs bake in strong human assumptions, scale poorly as interaction histories grow, and are brittle under distribution shift. The paper argues that memory extraction itself should be a learnable abstraction—structured, reusable behaviors shaped by interaction data—rather than a fixed procedural workflow.

Three properties are identified for an ideal memory system: (i) minimal reliance on human priors, with memory behaviors shaped by data; (ii) support for larger extraction granularity (span-level rather than per-turn); and (iii) skill-conditioned, compositional memory construction via a selectable set of reusable skills applied in one generation step.

## Method

**Skill Bank:**
- Shared across all interaction traces; each skill contains a short *description* (used for selection/representation) and detailed *content* (instructions for the executor).
- Initialized with four primitives: `Insert`, `Update`, `Delete`, `Skip`.
- Expanded and refined by the designer over training.

**Controller (skill-selection policy):**
- Lightweight MLP initialized with PPO.
- At each span step `t`, encodes state `h_t = f_ctx(x_t, M_t)` from the current text span `x_t` and retrieved memories `M_t`; encodes each skill via `u_i = f_skill(desc(s_i))`.
- Scores skills by dot product: `z_{t,i} = h_t^T u_i`; selects ordered Top-K set without replacement via Gumbel-Top-K sampling.
- Variable-size skill bank compatibility: scoring is over embeddings, not a fixed action head, so the controller adapts as the bank grows.
- Joint log-prob of the Top-K without-replacement selection: `log π_θ(A_t|s_t) = Σ_{j=1}^K [log p_θ(a_{t,j}) - log(1 - Σ_{ℓ<j} p_θ(a_{t,ℓ}))]`.
- PPO objective uses clipped surrogate loss `L_policy`, value function loss `L_value`, and entropy bonus `H(θ)`.

**Executor (skill-conditioned memory extraction):**
- Fixed LLM (LLaMA-3.3-70B-Instruct or Qwen3-Next-80B-A3B-Instruct, API access).
- Conditioned on current span `x_t`, retrieved memories `M_t`, and selected skills `A_t`.
- Produces structured memory updates applied to the trace-specific *memory bank*.
- Processes interaction at *span level* (default chunk size `512` tokens) rather than turn by turn, reducing LLM calls.

**Designer (skill evolution):**
- Maintains a sliding hard-case buffer; each case records query, retrieved memories, model prediction, reference answer, and failure count.
- Difficulty score: `d(q) = (1 - r(q)) · c(q)`, where `r(q)` is task reward and `c(q)` is cumulative failure count.
- Representative cases mined via KMeans clustering over query semantics for diversity.
- Two-stage update: (1) LLM identifies missing/mis-specified memory behaviors from hard cases; (2) proposes edits to existing skills and new skills.
- Rollback to best-performing snapshot if an evolution step regresses; early stopping if multiple consecutive cycles fail to improve stabilized tail reward `r̄_tail = mean(r_t) for t ∈ [3L/4, L]`.
- Post-evolution exploration phase biases controller logits toward new skills for `T_explore = 50` steps with linearly decaying threshold `τ_t = τ_0 · (1 - t/T_explore)`, `τ_0 = 0.3`.

**Evaluation-time settings:**
- `K = 7` for LoCoMo/LongMemEval; `K = 5` for ALFWorld; span size `512` tokens.
- Encoder for state and skill representations: Qwen3-Embedding-0.6B (shared).
- Memory retriever: Contriever (top-20 items).
- LLM judge: `openai/gpt-oss-120b`.
- Hardware: NVIDIA A6000 GPUs.

## Key Results

- **LoCoMo** (LLaMA): MemSkill achieves F1 `38.78`, L-J `50.96`; best L-J among all methods.
- **LoCoMo** (Qwen, transfer): L-J `52.07`, best overall.
- **LongMemEval** (LLaMA, transfer from LoCoMo): F1 `31.65`, L-J `59.41`, best among all methods despite zero LongMemEval-specific training.
- **ALFWorld** (LLaMA): success rate `47.86` (Seen) / `47.01` (Unseen); best SR in both splits.
- **ALFWorld** (Qwen, transfer): SR `60.00` (Seen) / `64.18` (Unseen).
- **HotpotQA distribution shift** (50/100/200 docs, LLaMA): MemSkill outperforms MemoryOS and A-MEM across all context lengths; gains widen at `200` docs.
- **Ablation on LoCoMo** (L-J, LLaMA/Qwen): default `50.96`/`52.07`; w/o controller `45.86`/`41.24`; w/o designer `44.11`/`34.71`; refine-only `44.90`/`46.97`. Disabling the designer causes the largest drop, especially under Qwen.

## Limitations

- Controller training requires downstream evaluation queries to generate reward, meaning labeled or simulatable tasks are needed; fully unsupervised deployment is not addressed.
- Skill bank evolution relies on LLM-generated skill proposals; the quality of proposed skills depends on the capability of the designer LLM and may be hard to control in low-resource settings.
- Span-level processing uses a fixed default chunk size (`512`); optimal chunk size may vary across domains and is not learned.
- Experiments use a restricted skill-count budget (`at most 3 edits per evolution round`), which may limit adaptation in highly diverse tasks.
- ALFWorld training uses offline expert trajectories; the approach has not been validated for fully online settings with environment interaction during training.
- Transfer is evaluated on two LLMs (LLaMA and Qwen); generalization to smaller or non-English models is unexplored.
