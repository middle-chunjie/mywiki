## Summary

⚠ Possibly outdated: published 2022; re-verify against recent literature.

DS-1000 introduces a `n = 1000` problem benchmark for data-science code generation built from `451` unique StackOverflow questions across seven Python libraries. The paper's main contribution is not a new model but a benchmark construction pipeline that combines naturalistic prompts, executable contexts, multi-criteria evaluation, and explicit anti-memorization perturbations. Problems are manually rewritten so they can be executed and scored by both functional tests and surface-form constraints, yielding a reported sample-level false discovery rate of `1.8%` on accepted Codex-002 outputs. As a benchmarking result, Codex-002 in insertion mode reaches `43.3%` pass@1, leaving substantial headroom and establishing DS-1000 as an early realistic stress test for code-oriented LLMs.

## Problem & Motivation

Existing code-generation benchmarks either focus on competition-style problems with reliable tests or on natural data-science tasks scored by weak surface-form metrics. The authors argue that realistic data-science assistance requires both natural prompts and reliable execution-based evaluation. They therefore target everyday StackOverflow-style questions involving libraries such as NumPy, Pandas, SciPy, TensorFlow, PyTorch, Scikit-learn, and Matplotlib, while also addressing the contamination risk that models may simply memorize public solutions from pretraining corpora.

## Method

- **Problem sourcing**: scrape StackOverflow posts under seven library tags, filter to questions with `vote >= 1`, `views >= 1000`, and an accepted answer, then calibrate ranking by problem age.
- **Selection and curation**: sample an initial pool of `4500` problems, score them for usefulness, clarity, difficulty, presence of examples, and evaluability, and retain content derived from `451` unique source problems.
- **Executable rewriting**: convert each natural problem into an executable prompt by adding imports, defining context variables, specifying expected output variables, and editing reference solutions to run correctly in that context.
- **Evaluation design**: implement multi-criteria checking with functional tests plus targeted surface-form constraints, e.g. rejecting forbidden APIs or inefficient loop-based solutions when the prompt requires vectorized code.
- **Memorization defense**: apply surface perturbations, semantic perturbations, and harder rewrites so benchmark items differ from original web problems while preserving or intentionally controlling difficulty.
- **Quality assurance**: require at least `3` expert annotators per problem/evaluation pair, red-team the metric against known-bad predictions, and estimate reliability from `2800` reviewed model samples.
- **Environment and benchmarking**: fix execution to Python `3.7.10`-compatible library versions, evaluate pass@1 from `40` samples per problem with `temperature = 0.2`, `top_p = 0.95`, and `max_tokens = 1024`, and compare completion versus insertion prompting.

## Key Results

- DS-1000 contains `1000` benchmark problems derived from `451` unique StackOverflow questions spanning `7` Python data-science libraries.
- Construction required about `1200` annotator hours from five authors familiar with Python and data science.
- The evaluation reports sample-level false discovery rate `1.8%` and false omission rate `0.5%`; problem-level false positive and false negative percentages are both `5.7%`.
- Dataset statistics: average `1.6` test cases per problem, `19.4%` of problems include surface-form constraints, average prompt length `140` words, and average reference solution length `3.6` lines.
- On numpy-100, Codex-002 drops from `72.5%` to `40.6%` after perturbation, supporting the paper's contamination concern.
- On DS-1000, best overall performance is Codex-002 insertion with `43.3%` pass@1 versus `39.2%` for the same model in completion mode, a `4.1%` absolute gain.
- Smaller baselines lag substantially: CodeGen-6B reaches `8.4%` overall and InCoder-6B reaches `7.4%` in completion mode.

## Limitations

The benchmark is expensive to build because high-quality evaluation still depends on substantial expert labor rather than scalable automatic test synthesis. Coverage is limited to seven Python libraries and a Python `3.7.10`-compatible software stack, so results may not transfer cleanly to newer library versions or broader programming settings. The average of `1.6` test cases per problem is strong for this domain but still sparse relative to formal program verification. Finally, the benchmark mainly studies 2022-era public code models, so its difficulty profile and anti-memorization design should be re-validated against more recent code LLMs.
