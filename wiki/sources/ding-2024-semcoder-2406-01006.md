## Summary

SemCoder trains a `6.7B` code language model to reason about programs through multiple semantic views instead of treating code generation as pure next-token prediction over static source text. The paper introduces monologue reasoning, where the model verbalizes approximate semantics, abstract properties, and forward/backward execution behavior, and pairs this with a new executable Python corpus, PYX, plus a debugging/self-refinement dataset, PyX-R. Starting from DeepSeekCoder-6.7B, the resulting model substantially improves execution reasoning and remains competitive on code generation: SEMCODER-S reaches `79.3` HumanEval, `63.6` CRUXEval-I, and `63.9` CRUXEval-O, beating GPT-3.5-turbo on the execution benchmarks reported here. The core claim is that explicit semantic supervision bridges static code tokens and dynamic runtime behavior.

## Problem & Motivation

Existing code LMs are mostly trained on static source text and docstrings, so they can autocomplete or synthesize code but often fail to model what the program is actually doing during execution. The paper argues that strong performance on debugging, repair, and execution reasoning requires a richer notion of program semantics: high-level intent, invariant-like properties, line-level effects, and input/output behavior. SEMCODER is motivated by the observation that human developers reason this way when they explain code to themselves during debugging, so the authors train the model to imitate that verbal semantic analysis.

## Method

- **Base model and variants**: initialize from `DeepSeekCoder-6.7B`; train `SEMCODER` on PYX and `SEMCODER-S` on an extended mixture of `PYX + Evol-Instruct + 4.3k CodeContest` problems with at least one correct LLM-generated solution.
- **Executable training corpus (PYX)**: build a synthetic Python dataset with `32k` executable samples by sampling parsable AST subtrees as seeds, generating NL-to-code pairs, executing candidate solutions, and keeping only programs that run successfully.
- **Operational-semantics filtering**: for execution data, retain only programs without external-resource interaction and without randomness; expand inputs via type-aware mutation plus LLM-based input generation so traces cover diverse behaviors.
- **Semantic decomposition**: supervise four views of program semantics: approximate semantics (task-level description), symbolic semantics (code itself), abstract semantics (properties and constraints), and operational semantics (stepwise execution behavior).
- **Forward monologue**: given source code and input, the model explains executed lines, side effects, variable-state changes, and final output in natural language, following execution order rather than a rigid trace schema.
- **Backward monologue**: given final output, the model reasons backward using abstract constraints when prior states are not uniquely recoverable, then proposes concrete inputs consistent with those constraints.
- **Annotation pipeline**: use a stronger LLM with rejection sampling to generate monologues; accept forward/backward annotations only when predicted outputs or reconstructed behaviors are consistent with execution-based checks.
- **Training objective**: optimize standard next-token prediction over code and monologue tokens jointly, with task prefixes indicating which semantic view to produce; training uses `2` epochs, learning rate `5e-5` decayed to `5e-6`, batch size `512`, max context length `2048`, on `8 x NVIDIA RTX A6000`.
- **Debug/self-refine extension**: construct `PYX-R` from buggy samples that fail tests, plus rubber-duck debugging rationales and patches validated by differential testing; self-refinement fine-tuning uses `2` epochs and learning rate `1e-5`.

## Key Results

- **Code generation**: `SEMCODER-S` reaches `79.3` / `74.4` on HumanEval / HumanEval+, `79.6` / `68.5` on MBPP / MBPP+, and `27.5` on LiveCodeBench-Lite; GPT-3.5-turbo is `76.8`, `70.7`, `82.5`, `69.7`, and `23.9` respectively.
- **Execution reasoning**: `SEMCODER-S` scores `63.6` on CRUXEval-I, `63.9` on CRUXEval-O, and `61.2` on LCB-Exec; base `SEMCODER` scores `62.5`, `65.1`, and `59.7`, all well above the DeepSeekCoder-6.7B checkpoint (`39.5`, `41.2`, `36.1`).
- **Absolute gains over initialization**: compared with DeepSeekCoder-6.7B, semantic-heavy training improves input prediction by `+23.0` points, CRUXEval-O by `+23.9`, and LCB-Exec by `+23.6`.
- **Ablation on reasoning format**: monologue reasoning (`61.8` CRUXEval-I, `63.5` CRUXEval-O, `58.5` LCB-Exec) outperforms scratchpad (`48.8`, `50.6`, `39.9`), NeXT traces (`49.4`, `50.9`, `32.2`), and concise traces (`52.1`, `55.6`, `35.9`).
- **Iterative self-refinement**: in zero-shot mode, `SEMCODER-S` reaches `84.8` / `79.3` on HumanEval / HumanEval+ and `86.8` / `74.3` on MBPP / MBPP+; after PyX-R fine-tuning it reaches `85.4` / `79.3` and `87.0` / `73.5`.

## Limitations

- The authors manually inspected PYX/PyX-R and note that intermediate monologue steps can still be logically flawed even when the final answer or patch is correct.
- Monologue annotation depends on stronger proprietary LLMs such as `GPT-3.5-turbo` and `GPT-4o-mini`, plus rejection sampling, so the supervision pipeline is not self-contained.
- PYX operational-semantics data excludes programs with external resources or randomness, which narrows coverage relative to real-world software tasks.
- The method is evaluated primarily on Python-centric code-generation, execution-reasoning, and self-refinement benchmarks, so transfer to broader languages and tool-using workflows is still uncertain.
