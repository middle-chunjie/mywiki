## Summary

LeDex is a training framework for improving code-oriented large language models on self-debugging rather than relying only on prompting. The paper builds a verified dataset of wrong code, natural-language explanations, and corrected refinements by sampling buggy programs from MBPP, APPS, and CodeContests, then filtering teacher-generated repairs through execution. It trains open models with two supervised formats, direct refinement and explanation-plus-refinement, and then applies PPO with separate rewards for explanation quality and repair quality. Across StarCoder-15B and CodeLlama backbones, the approach substantially improves `pass@1`, `pass@10`, refinement success rate, iterative multi-round repair, and human-rated explanation usefulness, showing that executable feedback plus explicit explanation supervision can materially strengthen open-source code LLMs.

## Problem & Motivation

Open-source code LLMs can generate useful first-pass programs, but their ability to inspect failed code, understand execution feedback, explain the bug, and iteratively repair it remains weak under prompting alone. Prior self-debugging work mainly studied stronger proprietary models or in-context prompting pipelines, which transfer poorly to smaller open models such as StarCoder and CodeLlama. The paper argues that explanation before refinement helps the model reason about why code fails, but high-quality training data for that behavior is scarce. LeDex addresses this by automatically collecting verified explanation-and-repair trajectories and then directly training models to use them.

## Method

- **Data collection**: sample `20` candidate solutions per problem with `3`-shot prompting and `temperature = 1.0` from pretrained StarCoder / CodeLlama on MBPP training (`374` tasks), APPS training (`5,000` tasks), and CodeContests (`6,627` tasks); keep wrong solutions that fail at least one unit test.
- **Verification pipeline**: query a teacher or stronger model with the problem, wrong code, and execution feedback to generate explanations and refinements; retain only refinements that pass all unit tests, treating their paired explanations as correct supervision.
- **SFT formats**: construct chat-style instruction data for two behaviors, direct repair and explanation-then-repair; the wrong initial solution is kept in context, but its tokens are masked from loss so the model is not optimized to emit buggy code.
- **Refinement reward**: score repaired code with CodeBLEU and unit-test pass rate, using `S_cb(r) = (1 / |R_c^w|) Σ CodeBLEU(r, r_c)` and `S_ut(r) = |T_p(r)| / |T|`, then set `R(r) = 5 · (S_cb(r) + S_ut(r)) - 5`.
- **Explanation reward**: compute semantic similarity against verified explanations with `S_ex(e) = (1 / |E_c^w|) Σ CosSim(RoBERTa(e), RoBERTa(e_c))`, then map it to `R(e) = (50 · S_ex(e) - 35) / 3`, so roughly `S_ex = 0.7` becomes neutral reward.
- **RL objective**: run PPO after SFT, assigning explanation reward at the explanation boundary `t = |e|` and code reward at the final repair token `t = T`, while regularizing with KL to the SFT policy; Appendix A.3 uses discount `γ = 0.99`.
- **Training setup**: SFT uses `2` epochs, batch size `128`, AdamW, learning rate `2e-5`, `500` warmup steps, and cosine decay; RL uses PPO via TRL with learning rate `2e-6`, batch size `64`, on `8 × NVIDIA A100 40GB` GPUs.

## Key Results

- Verified training data scale: MBPP yields `9,500` unique samples with `4,794` wrong solutions and `2,203` correct refinements; APPS yields `44,108` / `16,372` / `6,419`; CodeContests yields `51,134` / `19,614` / `5,113`, for `13,735` verified explanation-repair pairs total.
- SFT improves overall performance by up to `+15.92` `pass@1` and `+9.30` `pass@10` over prompting across four benchmarks.
- RL adds up to `+3.54` `pass@1` and `+2.55` `pass@10` beyond SFT, with especially strong gains on tougher MBPP+ / HumanEval+ settings.
- StarCoder-15B refinement success rises from `6.41%` to `16.56%` with SFT and `17.59%` with RL for explanation-plus-refinement; CodeLlama-13B rises from `11.97%` to `20.06%` and then `23.50%`.
- On human-rated explanation quality for StarCoder, GPT-4 scores improve from `1.90` to `2.92` to `3.18`, and developer ratings improve from `1.73` to `2.60` to `2.76` for prompting, SFT, and RL respectively.
- Iterative refinement remains effective across rounds: for CodeLlama-7B, prompting after three rounds still trails SFT after one round in the paper's aggregate plot (`47.53` vs. `56.75`).

## Limitations

- The framework still depends on explanations and repairs collected from a small set of teacher models (`GPT-3.5-Turbo`, `CodeLlama-34B`, and self-bootstrapped `CodeLlama-7B`), so data quality may inherit their biases.
- Prompt diversity is narrow: the study only trains on two feedback styles, direct refinement and explanation-then-refinement, leaving richer debugging dialogues unexplored.
- Explanation quality is rewarded via embedding similarity to verified explanations, which is a proxy and may miss finer-grained notions of causal correctness or pedagogical usefulness.
- Evaluation focuses on benchmarked code-generation and repair tasks with executable tests, so transfer to real-world software engineering workflows is not fully established.
