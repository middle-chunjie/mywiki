## Summary

The paper defines application-driven data analysis over relational databases and introduces DACO, a benchmark plus training resource built with multi-turn code generation. Each example pairs a stakeholder-style query with a findings-and-suggestions answer grounded in interpreter-executed Python. The released dataset contains `440` databases and `1,942` query-answer pairs, plus a `100`-example human-refined test subset. On top of GPT-4-generated weak supervision, the authors train a `6B` CodeGeeX2 model with supervised fine-tuning and then propose Daco-RL, which combines an answer reward, a step-level contribution reward, and a regularization reward against reward hacking. Human evaluation shows Daco-RL beats the SFT baseline in `57.72%` of pairwise comparisons on helpfulness.

## Problem & Motivation

Prior work on table question answering mainly targets fact lookup, entity resolution, arithmetic aggregation, or short-form reasoning over structured data. The authors argue that real stakeholder-facing analysis is harder: a useful answer must decompose a business or operational query into multiple analytical perspectives, execute quantitative checks on the database, interpret the outputs, and finally synthesize actionable findings and suggestions.

This makes annotation expensive and model training difficult. DACO addresses both bottlenecks by using strong LLMs to bootstrap multi-step analysis traces with executable Python, then refining a small subset with humans and training a smaller open model to imitate and improve on those traces under human-preference optimization.

## Method

- **Task formulation**: input is a relational database `\mathcal{D}` and an application-driven query `\mathbf{q}`; output is an answer `\mathbf{a}` organized into two lists, findings and suggestions.
- **Dataset construction**: databases come from Spider and Kaggle; the final released resource contains `440` databases and `1,942` query-answer pairs with splits `Train/Dev/TestA/TestH = 1558/100/284/100`. The human-refined `TestH` subset is built from `100` samples.
- **Query collection**: ChatGPT first proposes stakeholder roles and decisions, then candidate queries are filtered by `6` annotators; about `42%` of machine-generated queries are removed, with Cohen's `\kappa = 0.62`.
- **Automatic annotation pipeline**: GPT-4 performs multi-turn analysis. At turn `i`, it emits Python code `\mathbf{c}_i`, reads execution output `\mathbf{o}_i`, decides whether the analysis is comprehensive, and either continues or emits final answer `\mathbf{a}`. The resulting valid pairs average about `3.3` coding steps.
- **Human refinement**: for each `TestH` query, annotators review `3` GPT-4 analyses, score each bullet as `0/1/2`, merge the best bullets, deduplicate, and minimally rewrite them; borderline bullets are added only when needed to meet the minimum `3`-bullet threshold.
- **Answer reward model**: `R_a` is trained from pairwise preferences over bullet points and scores each answer bullet. Diversity is encouraged with a repetition penalty `\sum_{j=1}^{i-1} Sim(\mathbf{f}_i, \mathbf{f}_j)` using Sentence-BERT cosine similarity.
- **Contribution reward model**: `R_c` provides dense rewards for intermediate code steps by ranking them with `Sim(\mathbf{a}, \mathbf{o}_i)`, where similarity between the final answer and a code output approximates how much step `i` contributes to the eventual analysis.
- **Regularization reward model**: `R_r` is trained against reward-hacking traces sampled from a hacked policy `\pi_{hack}` so the agent is penalized for patterns that exploit `R_c` without improving true helpfulness.
- **RL objective**: a multitask reward model `R_{a+c}` jointly learns answer and contribution rewards, and code steps are optimized with mixed reward `1/2 (R_{a+c} + w_r R_r)`. PPO optimizes the policy with GAE `\hat{A}_t = \sum_{l=0}^{\infty} (\gamma \lambda)^l \delta_{t+l}`, but gradients are computed only on model-generated subsequences `[ \mathbf{c}_1, ..., \mathbf{c}_m, \mathbf{a} ]`.
- **Model and hyperparameters**: the base model is CodeGeeX2-6B. SFT uses `learning_rate = 1e-5`, `total_steps = 600`, `gradient_accumulation = 4`; Daco-RL uses `learning_rate = 2e-6`, `total_steps = 200`, `gamma = 1.0`, `lambda = 1.0`. Inference uses nucleus sampling with `p = 0.9` and `temperature = 1.0`.

## Key Results

- Final resource size: `440` databases and `1,942` query-answer pairs; in `TestH`, median answer length is `397` tokens with median `5` findings and `5` suggestions.
- Human refinement reveals label quality gaps in machine-generated analyses: only `47.4%` of bullets are judged good, `52.2%` borderline, and `0.4%` bad before refinement.
- Code generation materially helps strong proprietary LLMs: GPT-4 improves from `30.81` to `52.00` helpfulness on `TestA` and from `24.00` to `41.88` on `TestH` when code generation is enabled; human pairwise preference over GPT-4 without code generation is `66.41%`.
- Daco-RL improves the `6B` open model over SFT on `TestH`: helpfulness rises from `11.54` to `21.05`, entailment from `4.47` to `5.98`, and human pairwise win rate reaches `57.72%` over SFT.
- Ablations show both dense rewards matter: removing `R_r` drops `TestH` helpfulness to `18.75`, while removing both `R_c` and `R_r` drops it to `8.79`; code informativeness over SFT falls from `57.86` to `44.57`.
- Daco-RL also improves external transfer on InfiAgent-DA, raising accuracy from `12.92` to `14.61`, with especially large gains on summary-statistics questions (`10.80` to `14.86`) and correlation analysis (`14.86` to `21.57`).

## Limitations

- Most supervision is still GPT-4-generated weak supervision; only a `100`-example test subset receives dense human refinement, so training labels may inherit LLM biases and annotation artifacts.
- The contribution reward is heuristic because it approximates code usefulness with answer-output similarity `Sim(\mathbf{a}, \mathbf{o}_i)`, which the authors explicitly note is vulnerable to reward hacking.
- The task scope is restricted to relational tabular databases and findings/suggestions style reports; the paper does not study unstructured evidence, multimodal data, or open-world analysis settings.
- Even after RL, the `6B` open model remains materially below GPT-4 on helpfulness, and Daco-RL trades off some BLEU (`14.60` to `11.80` on `TestH`) while still exhibiting non-zero code error rates (`2.75%` per step).
