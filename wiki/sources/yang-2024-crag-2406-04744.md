## Summary

CRAG introduces a comprehensive benchmark for retrieval-augmented question answering that is explicitly designed to stress real-world RAG failure modes instead of only static fact lookup. The benchmark contains 4,409 English QA pairs across Finance, Sports, Music, Movie, and Open domains, with eight question types spanning simple facts, conditions, aggregation, multi-hop, post-processing, and false-premise queries. It couples the questions with up to 50 full HTML pages from web search, mock knowledge-graph APIs, and a 2.6M-entity mock KG so systems must handle noisy unstructured and structured evidence jointly. The paper shows that strong LLMs and straightforward RAG pipelines remain far from trustworthy on this setup, making CRAG a useful diagnostic benchmark for retrieval quality, grounding, temporal robustness, and hallucination control.

## Problem & Motivation

Existing QA and RAG benchmarks do not jointly capture dynamic facts, long-tail entities, noisy retrieval, structured-plus-unstructured evidence, and evaluation that distinguishes abstention from hallucination. The authors argue that this leaves a mismatch between benchmark success and real deployment requirements for trustworthy question answering. CRAG is proposed to close that gap by making benchmark instances more realistic, diverse, diagnostically sliceable, and maintainable over time.

## Method

- **Task design**: CRAG defines three progressively harder tasks: retrieval summarization with up to `5` web pages, KG+web augmentation with mock APIs, and end-to-end RAG with up to `50` web pages plus APIs, isolating answer synthesis, structured retrieval use, and ranking under larger candidate sets.
- **Dataset scope**: the benchmark contains `4,409` QA pairs over `5` domains and `8` question categories, with `2,425` web-derived questions and `1,984` KG-derived questions.
- **Question construction**: KG questions are generated from `600+` templates over selected entity types and relations, then instantiated with head/torso/tail entities; web questions are written from plausible user intents and grounded in web results.
- **Retrieval substrate**: each example includes up to `50` full HTML pages from Brave Search plus mock KG access; the full benchmark totals roughly `220K` webpages, a mock KG with `2.6M` entities, and `38` mock APIs.
- **Realism controls**: mock KGs include random same-type distractors and hard negatives, while web and KG snapshots are collected concurrently for real-time and fast-changing questions so the benchmark reflects time-sensitive retrieval conditions.
- **Question distribution**: CRAG annotates dynamism (`437` real-time, `564` fast-changing, `1,007` slow-changing, `2,401` static) and question type slices (`1,205` simple, `689` simple-with-condition, `536` comparison, `489` aggregation, `382` multi-hop, `180` post-processing-heavy, `525` false-premise) to support slice analysis.
- **Metrics**: human-eval labels answers as perfect / acceptable / missing / incorrect with scores `1`, `0.5`, `0`, and `-1`; the benchmark truthfulness metric is the average of those scores, explicitly penalizing hallucinations more than abstentions.
- **Auto-eval**: automatic evaluation first checks exact-match answers, then uses ChatGPT `gpt-3.5-turbo-0125` and Llama 3 `70B` to classify remaining outputs as accurate / incorrect / missing with scores `1`, `-1`, `0`.
- **Split and protocol**: data is split into validation / public test / private test at `30% / 30% / 40%`, with the public test used for reported model comparisons and KDD Cup development.
- **Baseline setup**: straightforward RAG baselines use web windows of `1K`, `2K`, or `4K` tokens depending on model size, KG windows of `0.5K`, `1K`, or `2K`, and Llama 3 `8B` in-context entity extraction for API querying.

## Key Results

- Web retrieval recall reaches `85%` when all `50` pages are available, versus about `69%` for the top `5` pages; recall is `93%` for web-derived questions but only `74%` for KG-derived questions.
- On the public test set of `1,335` questions, the best LLM-only baseline (GPT-4 Turbo) reaches `33.5%` accuracy with `20.0` truthfulness, showing strong parametric models remain unreliable without retrieval.
- Straightforward RAG improves raw accuracy only modestly: the best reported setting (Task 3 with GPT-4 Turbo) reaches `43.6%` accuracy, `30.1%` hallucination, `26.3%` missing, and `13.4` truthfulness.
- Task 2 improves over Task 1 for GPT-4 Turbo from `35.9%` to `41.3%` accuracy, indicating KG evidence helps, but hallucination remains high at `25.1%`.
- Human-evaluated industry systems perform better but are still far from solved: Copilot Pro reaches `50.6` equal-weighted truthfulness with `62.6%` perfect answers, while traffic-weighted truthfulness peaks at `60.5`.
- Automatic evaluation tracks human labels well: the two-step evaluator obtains average F1 of `94.7%` with ChatGPT and `98.9%` with Llama 3.

## Limitations

- The benchmark does not directly evaluate first-stage large-scale retrieval over the open web; it starts from provided candidate pages and APIs, so corpus construction quality is factored out.
- At `4.4K` questions, CRAG is much smaller than datasets such as MS MARCO or Natural Questions, trading scale for realism and richer annotations.
- The release is English-only and covers five domains, so multilingual, multimodal, and multi-turn QA are left to future extensions.
- Mock KGs and APIs improve controllability but are still abstractions of production environments; results on CRAG may not transfer perfectly to unrestricted web retrieval.
- Comparisons between straightforward baselines and industry systems are only partially comparable because they differ in retrieval access and use auto-eval versus human-eval.
