## Summary

⚠ Possibly outdated: published 2021; re-verify against recent literature.

This paper proposes gradient cache, a training procedure that lets deep contrastive models use effectively large batches under tight GPU memory budgets without changing the resulting update. Instead of backpropagating through the full encoder graph for the entire batch, it first computes representations without graphs, then computes representation-level gradients from the contrastive loss, caches those gradients, and finally replays encoder backpropagation sub-batch by sub-batch. The key claim is exact equivalence to the large-batch gradient, not an approximation like ordinary gradient accumulation. On DPR-style dense retrieval, the method reproduces large-batch behavior on a single RTX 2080 Ti, improves over small-batch and accumulated baselines, and scales to batch sizes up to `4096` with roughly `20%` extra runtime for representation pre-computation.

## Problem & Motivation

Large-batch contrastive learning benefits from many in-batch negatives, but standard training requires encoding the whole batch and keeping all intermediate activations in memory. That makes strong dense retrieval setups such as DPR hard to reproduce outside multi-GPU environments: the paper notes that a batch size of `4096` had been used in prior work while a single RTX 2080 Ti can fit only about `8` examples. Standard `gradient accumulation` increases the number of optimizer steps per effective batch but does not restore the missing cross-example negatives inside each micro-batch. The paper therefore targets exact large-batch contrastive training under memory-limited hardware, especially for academic settings.

## Method

- **Contrastive objective**: for anchors `S` and targets `T`, the loss is `L = -(1/|S|) sum_i log( exp(f(s_i)^T g(t_{r_i}) / tau) / sum_j exp(f(s_i)^T g(t_j) / tau) )`, with `tau = 1` in the analysis for simplicity.
- **Key factorization**: encoder-parameter gradients decompose as `∂L/∂Θ = sum_i (∂L/∂f(s_i)) (∂f(s_i)/∂Θ)` and `∂L/∂Λ = sum_j (∂L/∂g(t_j)) (∂g(t_j)/∂Λ)`, so once representation-level gradients are known, encoder backpropagation is independent across examples.
- **Representation-gradient formulas**: the normalized similarity is `p_ij = exp(f(s_i)^T g(t_j)) / sum_t exp(f(s_i)^T g(t))`; then `∂L/∂f(s_i) = -(1/|S|)(g(t_{r_i}) - sum_j p_ij g(t_j))`, with the symmetric form for `∂L/∂g(t_j)`.
- **Step 1, graph-less forward**: run the encoder on the full batch without constructing computation graphs and store all representations for `S` and `T`.
- **Step 2, gradient cache construction**: compute the full-batch contrastive loss only over stored representations, backpropagate through that lightweight graph, and cache `u_i = ∂L/∂f(s_i)` and `v_j = ∂L/∂g(t_j)`.
- **Step 3, sub-batch replay**: split the large batch into memory-fitting sub-batches `S_hat` and `T_hat`; for each sub-batch, run a normal encoder forward/backward and inject the cached representation gradients so accumulated parameter gradients equal the original full-batch gradients.
- **Memory profile**: encoder forward-backward memory becomes approximately constant in sub-batch size, while the persistent cache stores only `(|S|d + |T|d)` floating-point values rather than full activations.
- **Multi-GPU variant**: after graph-less forward, use `all-gather` so each device sees all representations, compute local cached gradients from the global loss, then perform ordinary distributed gradient reduction during optimization.
- **Experimental setup**: the method is tested by inserting gradient cache into [[dpr]] training on [[natural-questions]]; the main setting uses batch size `128`, question sub-batch `16`, passage sub-batch `8`, one positive plus one [[bm25]] negative passage per question, on a single RTX 2080 Ti.
- **Extension beyond dot product**: for deep similarity `d_ij = Phi(f(s_i), g(t_j))`, the paper introduces a second cache over `w_ij = ∂L/∂d_ij` and derives a sub-batch update for `∂L/∂Omega`, but this section is presented analytically rather than as a main empirical result.

## Key Results

- On DPR-style retrieval, `Sequential` training reaches Top-5/20/100 of `59.3 / 71.9 / 80.9`, `Accumulation` reaches `64.3 / 77.2 / 84.9`, and `Cache` reaches `68.6 / 79.3 / 86.0`.
- Increasing cached training batch size from `128` to `512` yields `68.3 / 79.9 / 86.6`, improving Top-20 and Top-100 over the main cache setting.
- The reported DPR reference is `78.4` Top-20 and `85.4` Top-100; the paper argues its single-GPU cache training matches or slightly exceeds that reference due to randomness.
- Gradient cache scales update size from `64` up to `4096` examples, whereas the paper reports gradient checkpointing only runs up to batch `64` on the tested hardware.
- Representation pre-computation adds about `20%` runtime overhead, but the authors report practical end-to-end DPR training in `31` hours on one RTX 2080 Ti versus roughly one day on `8` V100 GPUs in the original DPR report.

## Limitations

- Empirical validation is narrow: experiments focus on a DPR-like dense retriever on [[natural-questions]], not on multiple contrastive NLP tasks or model families.
- The method removes activation-memory scaling in the encoder pass but still requires storing representation caches of size `(|S|d + |T|d)` and performing extra forward passes.
- Runtime is not free: the reported gain comes with roughly `20%` additional compute from graph-less representation pre-computation.
- The extension to deep distance functions is derived mathematically, but the paper does not present a main empirical study validating that generalized cache in practice.
- The paper shows equivalence to large-batch updates, but it does not address whether ever-larger batches eventually hurt generalization or optimization stability.
