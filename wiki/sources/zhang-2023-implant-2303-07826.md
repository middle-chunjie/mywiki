## Summary

> ⚠️ Possibly outdated: published 2023 in the rapidly evolving NLP/code intelligence domain.

Source code carries both sequential naturalness and hierarchical structural information; prior sequence-based models fail to capture the latter fully. This paper proposes **HiT (Hierarchy Transformer)**, which encodes complete root-to-leaf paths from concrete syntax trees (CSTs) as token-level *hierarchical embeddings* — split into statement-level global hierarchy and intra-statement local hierarchy — and injects them into a standard Transformer sequence encoder. HiT is evaluated on three tasks (code classification, clone detection, method name prediction) across 8 datasets and consistently outperforms both tree/graph-based structural models and vanilla Transformers at minimal additional parameter cost (~1–5%).

## Problem & Motivation

Sequence-based code models (e.g., Transformer with token sequences) are efficient and exploit "naturalness" of code but cannot reliably capture structural semantics: the same token may carry different meaning depending on its block position (global hierarchy) or its syntactic role within a statement (local hierarchy). Existing hybrid approaches (GREAT, CodeTransformer) only model pairwise token distances on AST/graphs, overlooking the *full* hierarchical impact — particularly the global nesting structure of statements.

## Method

**Architecture — two stacked Transformers:**

- **Hierarchy Encoder**: for each source code token `t_k`, extract its CST root-to-leaf path `n = (n_1,...,n_{l'})` of AST node types. Embed each node type, run a small Transformer, then mean-pool to get a fixed-size path vector `p_k`.
  - Layers: `2`; hidden size: `256`; heads: `8`
- **Sequence Encoder**: concatenate hierarchy vector and token embedding `[p_k || E_k]` for each position, then run a larger Transformer to produce final token representations `H_1,...,H_l`.
  - Layers: `6` (classification/clone); `4` (generation)

**Path decomposition:**
- *Global hierarchy*: root-to-statement node of the path — encodes block nesting depth and structural position of the enclosing statement.
- *Local hierarchy*: statement-node-to-leaf portion — encodes the syntactic role of the token within its statement.

**Downstream heads:**
- Classification: `MeanPooling → 2-layer MLP → softmax`; trained with cross-entropy.
- Generation (method name prediction): Transformer decoder with a pointer network copy mechanism. Copy probability `p_copy = sigmoid(W_5 h_t* + b_2)`; final token probability `P(w) = p_gen * P_v(w) + p_copy * sum_i a_i^t`.

**Implementation:**
- Parser: Tree-sitter (language-agnostic CST extraction)
- Optimizer: `AdamW`, lr `1e-4`, weight decay; vocab size `5000–8000`
- Hardware: Tesla `V100S 32GB`; 5 runs averaged per experiment

## Key Results

- **Code classification** (Project CodeNet — Java250/Python800/C++1000/C++1400 + POJ-104): HiT accuracy 94.81 / 95.97 / 95.05 / 93.27 / 97.08; beats all baselines including ASTNN, TreeCaps, GREAT. On C++1400, improves vanilla Transformer by +25.4%.
- **Clone detection** (POJ-Clone MAP@R): HiT 80.46 vs. Transformer 67.15 (+13.3 absolute); competitive with MISIM-GNN (complex language-specific preprocessing required).
- **Method name prediction** (CSN-Ruby / CSN-Python F1): HiT 29.06 / 35.41; outperforms CodeTransformer by +1.51 / +0.42 F1; outperforms GTNM by +6.93 / +5.28 F1.
- **Variable scope detection** (Python800 / C++1400 accuracy): HiT 80.27 / 76.12 vs. Transformer 77.18 / 63.77 and GREAT 79.39 / 69.38.
- **Parameter overhead**: classification ~4.55M vs. baseline Transformer 4.50M (~1% extra); generation ~36.75M vs. 34.89M (~5.5%).
- **Training efficiency**: on Python800, HiT converges in 122 min vs. Transformer 169 min; CST preprocessing of 144k samples takes <5 min.

## Limitations

- Does not incorporate large-scale pre-training; CodeBERT (125M params, 27× larger) still outperforms on Java250/Python800/POJ-104 for classification despite HiT surpassing on C++1000/C++1400.
- Evaluation restricted to method-level or function-level code snippets; behaviour on file-level or project-level inputs not tested.
- The GTNM comparison for method name prediction uses only local context (project-level context unavailable for CSN-Ruby/Python), potentially disadvantaging GTNM.
- No formal ablation on hierarchy encoder depth or path length truncation; hyperparameter sensitivity not fully characterised.
- Does not address code with deeply recursive or macro-expanded syntax where CST paths become very long.
