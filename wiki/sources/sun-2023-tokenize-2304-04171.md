## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

GENRET studies the weakest part of generative retrieval: how to assign document identifiers that preserve document semantics rather than relying on ad hoc titles or clustering codes. The paper learns docids end to end with a discrete auto-encoding pipeline composed of a document tokenization model `Q(z|d)`, a retrieval model `P(z|q)`, and a reconstruction model `R(d|z)`. Progressive training optimizes one docid position at a time, while constrained clustering improves codebook balance and docid diversity. On NQ320K, GENRET sets a new state of the art and is especially strong on unseen documents, where prior generative retrievers degrade sharply. The method also stays competitive on MS MARCO and BEIR, suggesting that learned semantic docids transfer better than rule-based identifiers.

## Problem & Motivation

Generative retrieval replaces the standard index-then-rank pipeline with direct autoregressive generation of document identifiers, but its effectiveness depends heavily on how those identifiers are defined. Earlier systems use titles, URLs, arbitrary substrings, or hierarchical clustering outputs as docids, which are easy to construct but often fail to preserve full document semantics. This is particularly problematic for unseen or out-of-distribution documents, where fixed rule-based tokenizers generalize poorly. The paper therefore asks whether docids themselves can be learned so that they are short, unique enough for retrieval, reconstructive of the source document, and optimized jointly with the retrieval model.

## Method

- **Shared seq2seq backbone**: GENRET uses a shared T5-Base [[transformer]] for document tokenization `Q(z|d)` and query-to-docid retrieval `P(z|q)`. Given document `d` and prefix `z_{<t}`, the decoder produces timestep state `` `d_t = Decoder(Encoder(d), z_{<t}) in R^D` ``.
- **Codebook-based docid prediction**: at each step `t`, the model introduces a codebook `` `E_t in R^(K x D)` `` with `K = 512` code vectors. Token probabilities are computed by `` `Q(z_t = j | z_<t, d) = Softmax_j(d_t E_t^T)` ``, and the actual docid token is selected by `` `z_t = argmax_j Q(z_t = j | z_<t, d)` ``.
- **Reconstruction via discrete auto-encoding**: the chosen docid is embedded as `` `z = {e_(1,z1), ..., e_(M,zM)} in R^(M x D)` `` and scored by a reconstruction model that retrieves the original document from a prefix-matched candidate set `` `S(z_<t)` `` using Eq. 6. This forces each prefix to capture residual semantics missing from earlier tokens.
- **Progressive optimization**: instead of optimizing all docid positions jointly, GENRET learns step `T` while freezing optimized prefixes `z_<T`. The objective at each stage is `` `L = L_Rec + L_Com + L_Ret` ``, combining reconstruction loss, commitment loss `` `L_Com = -sum_(t=1)^T log Q(z_t | z_<t, d)` ``, and retrieval loss with in-batch negatives.
- **Gradient handling**: because `` `argmax` `` is non-differentiable, the paper uses a straight-through estimator so gradients on `` `z_T` `` are copied back to document state `` `d_T` `` and the selected codebook vector.
- **Diversity mechanisms**: GENRET warm-starts each codebook with constrained `K`-means centroids after a `5K`-step warm-up, then applies docid re-assignment using Sinkhorn-normalized scores `` `H* = Diag(u) exp(H / epsilon) Diag(v)` `` with `` `epsilon = 1.0` `` and 100 Sinkhorn iterations to avoid collapsed assignments.
- **Training / inference recipe**: the model is trained with AdamW, learning rate `` `5e-4` ``, batch size `256`, and up to `500k` steps per timestep. Query augmentation uses DocT5Query with nucleus sampling `` `p = 0.8` ``, `` `t = 0.8` ``, generating 5 synthetic queries per document. Inference uses [[constrained-decoding]] with a prefix tree and [[beam-search]] of size `100`.
- **Docid length and retrieval**: the docid length `M` is dataset-dependent so each collection can be assigned sufficiently unique identifiers. During decoding, invalid next tokens are masked to zero probability, ensuring outputs correspond to existing corpus docids.

## Key Results

- **NQ320K SOTA**: GENRET reaches `68.1` R@1, `88.8` R@10, `95.2` R@100, and `75.9` MRR@100 on the full test set, beating NCI (`66.4` / `85.7` / `92.4` / `73.6`) and GTR-Base (`56.0` / `84.4` / `93.7` / `66.2`).
- **Better generalization to unseen documents**: on NQ320K unseen queries, GENRET gets `62.5` R@1 and `70.4` MRR@100, versus NCI's `54.5` and `62.4`; the paper highlights at least `+14%` relative improvement in R@1 over the best generative baseline.
- **Competitive on MS MARCO**: GENRET achieves `47.9` R@1, `79.8` R@10, `91.6` R@100, and `58.1` MRR@10, outperforming prior generative systems such as GENRE (`35.6` R@1, `42.3` MRR@10) and slightly surpassing GTR-Base on R@1 / R@10 / MRR@10.
- **Strong BEIR transfer**: on six BEIR datasets, GENRET averages `41.1` nDCG@10, outperforming BM25 (`37.8`), ST5 GPL (`39.0`), and GTR-Base (`39.3`), with especially strong scores on Covid (`71.8`) and SciFact (`63.9`).
- **Zero-shot new-document retrieval**: without retraining on unseen documents, GENRET's dAE docids yield `34.2` R@1 on NQ unseen and `12.1` / `12.1` / `12.3` nDCG@10 on BEIR Arg / NFC / SciDocs, far above title- or clustering-based docids in the same setting.
- **Efficiency trade-off**: on MS MARCO, GENRET uses `860MB` memory, `220min` offline indexing time, and `0.16s` online latency for Top-100, faster at retrieval time than GENRE (`1.41s`) and DSI (`0.32s`) while remaining much smaller than dense retrievers.

## Limitations

- The paper is a preprint and explicitly marked as work in progress, so some claims should be treated as provisional.
- GENRET still depends on offline docid tokenization and progressive multi-stage training, which is substantially more complex than title-based identifiers and slower to index than GENRE.
- Zero-shot tokenization of new collections can produce duplicate docids; the paper resolves this by retrieving all matching documents and shuffling them arbitrarily, which weakens exact ranking fidelity.
- The empirical backbone is mainly T5-Base with synthetic query augmentation, so it does not establish whether the method remains optimal under larger contemporary LMs or different pretraining regimes.
- Experiments focus on subsets such as NQ320K, sampled MS MARCO, and six BEIR tasks rather than full web-scale retrieval, so scalability to much larger corpora remains future work.
