## Summary

This paper reframes software test generation as issue reproduction and introduces SWT-Bench, a Python benchmark derived from SWE-Bench with GitHub issues, golden bug-fix patches, and golden tests. The evaluation formalizes a successful generated test as fail-to-pass against the golden patch and adds change coverage over executable patched lines as a finer metric. Across 1,983 usable instances and a 276-instance Lite subset, code agents adapted from repair systems outperform or match specialized test-generation baselines: SWE-Agent reaches `S = 15.9%`, rises to `18.5%` when asked to execute generated tests, and generated tests can more than double fix precision when used as filters. The paper argues that test generation is distinct from code repair and provides a robust custom diff format to improve LLM patch applicability.

## Problem & Motivation

Automatic code generation and code repair with large language models have progressed rapidly, but issue-reproducing test generation remains underexplored despite its practical value for regression prevention, developer productivity, and automatic validation of candidate fixes. Existing real-world bug reproduction benchmarks are either outdated, small, focused on Java, or not designed for Python code agents. The authors therefore seek a benchmark and evaluation protocol that can assess whether LLMs and code agents can transform informal GitHub issues into tests that fail before the gold fix and pass afterward, while also exposing how test generation differs from code repair.

## Method

- **Benchmark construction**: start from `~90,000` pull requests across `12` popular Python repositories; keep PRs that are merged, linked to an issue, modify at least one test file, and contain at least one golden `F -> P` test. This yields `2,294` instances, then `311` unreliable cases are removed, leaving `1,983` SWT-Bench instances; SWT-Bench-Lite contains `276` issues.
- **Task formalization**: for a generated test `s`, execution is `exec(s, R) in {P, F}`. A test reproduces issue `I` fixed by patch `X*` iff `exec(s, R) = F` and `exec(s, R o X*) = P`, i.e. an `F -> P` transition.
- **Success metric**: a generated test set `T` is successful when it contains at least one `F -> P` test and no `x -> F` test after applying the golden patch. The paper separately reports `S`, `F -> P`, `F -> x`, and `P -> P` rates.
- **Coverage metric**: change coverage tracks executable added/removed lines touched by the golden patch. The main score is `Delta C_T^X = (covered removed executable lines + covered added executable lines) / (all executable changed lines)`, excluding non-executable modified lines and the `~1%` of cases with zero executable changed lines.
- **Dataset statistics**: issue descriptions average `318` words; repositories average `1,563` files and `337K` lines; existing tests average `123.2` cases with `70.1%` coverage of to-be-patched code; golden tests add `2.9` tests, remove `0.3`, edit `1.5` files and `31.8` lines on average.
- **Custom diff format**: instead of unified diff hunks, the model emits blocks with file path, operation type (`rewrite` or `insert`), approximate location (`line`, `BOF`, `EOF`), and a full function/class body. This is optimized for LLM robustness because `71%` of golden test patches only add new test functions.
- **Direct LLM baselines**: ZEROSHOT uses unified diff plus `BM25` retrieval; ZEROSHOTPLUS swaps in the custom diff format; `PASS@5` samples `k = 5` tests and uses an oracle; LIBRO samples multiple tests and selects the one whose failing trace is most semantically aligned with the issue description.
- **Code-agent adaptation**: the authors adapt SWE-Agent, Aider, and AutoCodeRover by changing prompts from "solve the issue" to "create unit tests that cover the issue". Default evaluation uses GPT-4 unless otherwise noted, temperature `t = 0` for zero-shot methods and agents, `t = 0.7` for LIBRO and `PASS@5`, with agent budgets of `20` API calls for SWE-Agent, `3` reflection steps for Aider, and `10` interaction rounds for AutoCodeRover.

## Key Results

- SWT-Bench contains `1,983` usable instances after excluding `311` unreliable cases; SWT-Bench-Lite contains `276` instances.
- ZEROSHOT improves from only `W = 48.6%` and `S = 3.6%` under unified diff to `W = 89.5%` and `S = 9.4%` with the custom ZEROSHOTPLUS format.
- Among practical automatic methods, LIBRO reaches `S = 14.1%`, Aider `12.7%`, AutoCodeRover `9.1%`, SWE-Agent `15.9%`, and SWE-Agent+ `18.5%`; `PASS@5` gives an oracle upper bound of `20.3%`.
- Generated-test coverage on successful instances is high for several methods: ZEROSHOTPLUS `76.7%`, SWE-Agent `64.7%`, SWE-Agent+ `69.4%`, versus `72.0%` for golden tests.
- Providing the exact test file to edit nearly doubles ZEROSHOTPLUS success from `8.1%` to `15.1%`, while providing a golden or incorrect patch only raises success to `10.5%`.
- Using self-generated tests as a filter for generated fixes yields only `20%` recall but more than doubles SWE-Agent fix precision to `47.8%`.
- Solved-instance overlap between SWT-Bench and SWE-Bench is small (`1` overlap for ZEROSHOTPLUS, `7` for SWE-Agent), suggesting test generation and code repair are related but distinct tasks.

## Limitations

- The benchmark is limited to Python, so cross-language generalization remains untested.
- Repositories are popular public GitHub projects, which may not represent private or less standardized development environments.
- The benchmark focuses on issue reproduction and patch-specific coverage, not broader edge-case discovery or global test-suite improvement.
- Many source issues predate modern model knowledge cutoffs, so data contamination remains a plausible threat even though the authors do not find strong statistical evidence in their small post-cutoff analysis.
- The evaluation still relies on the correctness and executability of golden patches and test harnesses; `311` originally collected instances had to be discarded due to flakiness, setup errors, or timeouts.
