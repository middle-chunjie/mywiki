## Summary

This paper studies long-horizon agentic search, where an LLM must search the web over many turns and synthesize evidence across many sources. The authors show that strong open-source frameworks fail mainly because they overfill context with noisy retrieved content, exhaust tool budgets, or stop too early. They propose Slim, a simple single-agent framework that separates search from browsing and periodically summarizes the full trajectory to keep context compact. Search returns only concise snippets, browsing extracts the most relevant chunk from a chosen page, and summarization compresses the accumulated interaction history. On BrowseComp and Humanity's Last Exam, Slim matches or exceeds prior open-source baselines at lower cost and with far fewer tool calls, and the paper also introduces a trajectory-level error taxonomy for diagnosing failures such as hallucination, unfocused search, abstention, and answer ignoring.

## Problem & Motivation

Long-horizon search tasks require many sequential tool calls, diverse evidence gathering, and final synthesis across multiple sources, so naive retrieval interfaces quickly accumulate noisy text. The paper argues that the main bottleneck is not only model capability but context management: existing systems often scrape too much irrelevant content per step, which causes context-window overflow, wasted tool budget, and premature stopping. The goal is therefore to design a simpler agent interface that preserves exploration depth while keeping each turn and the running context concise.

## Method

- **Task setup**: formalizes long-horizon search as answering query `q` over document collection `D` via tool calls `T_i(x) -> y`, under a maximum tool budget `T`; each tool call corresponds to one trajectory turn, while the final answer does not consume budget.
- **Search tool**: Slim exposes a search operator that returns only the top `k = 10` search results as `(title_i, url_i, snippet_i)`, instead of scraping every result immediately.
- **Browse tool**: for URL `u` and query `q`, Slim scrapes the page, splits it into chunks `{d_i}_{i=1}^m`, and returns the chunk `d' = argmax_{d_i} ROUGE-L(d_i, q)`; the default maximum returned content is `L = 10,000` characters, with ablations over `L ∈ {3,000, 10,000, 20,000}`.
- **Summarization module**: after every `n = 50` tool turns, the full interaction context `C` is replaced by a trajectory summary `theta(C; summarize)`; ablations compare `n = 25` and token-threshold triggers `tau ∈ {32,768, 65,536}`.
- **Agent loop**: a single agent chooses between `search` and `browse` at each step until it emits a final answer or reaches `T`; main evaluations sweep `T` up to `150` turns.
- **Implementation details**: search is backed by Google via Serper, browsing by `crawl4ai`; default browse chunking splits on natural paragraphs and scores with `ROUGE-L`, with BM25 and 100-word chunking evaluated as alternatives.
- **Evaluation setup**: experiments use o3, o4-mini, and Claude-4-Sonnet with `temperature = 1.0` and maximum output length `32,768`; o3/o4-mini use medium reasoning effort and Claude uses up to `30,000` thinking tokens.
- **Analysis pipeline**: the paper adds automated trajectory-level annotation using rules plus an o3 judge to classify failure modes such as confirmation bias, unfocused search, answer ignored, abstention, and hallucination across 300 sampled instances from each benchmark.

## Key Results

- With o3 as the base model, Slim reaches `56.0` on BrowseComp and `30.7` on HLE, outperforming the best open-source baseline by `+8` and `+4` absolute points, respectively.
- At comparable cost on BrowseComp, Slim with `T = 150` uses `75.9` tool calls versus Search-o1's `306.2`, while scoring `56.0` versus `48.3`; on HLE it uses `12.0` versus `49.8` tool calls while scoring `30.7` versus `27.0`.
- Trajectory-level analysis shows a much lower hallucination rate for Slim (`19.0`) than Search-o1 (`46.8`), ReAct (`56.7`), and HF-ODR (`96.2`) under matched-budget comparison settings.
- Slim remains strongest across other base models: with o4-mini at `T = 150`, it reaches `37.0` on BrowseComp and `24.7` on HLE while still using fewer tool calls than baseline frameworks.
- In ablations on a 50-example subset with o4-mini, the default Slim setting scores `50.0` on BrowseComp; reducing browse length to `L = 3,000` raises BrowseComp to `54.0`, while word-level chunking plus BM25 raises HLE to `32.0`.

## Limitations

- Slim reduces hallucination and context overflow but still shows high `abstention = 27.7` and `answer ignored = 30.7`, so finding the correct answer in the trajectory is still not enough for correct final synthesis.
- The framework relies on heuristic design choices such as top-`k` search truncation, ROUGE-L chunk selection, and periodic summarization instead of a learned or theoretically grounded context manager.
- Evaluation is limited to two text-only long-horizon benchmarks, with 300 sampled instances per dataset rather than full benchmark coverage.
- The system depends on external search and scraping infrastructure, so part of the observed behavior may be tied to Serper result quality and crawl4ai extraction quality rather than agent policy alone.
