## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

MathCoder trains open-source LLMs to solve math problems by interleaving natural-language reasoning, executable Python code, and execution feedback in a single solution trace. The paper constructs the `80k`-example MathCodeInstruct dataset from GPT-4-generated GSM8K and MATH solutions plus synthetic interpolation problems that bridge the difficulty gap between the two benchmarks. It then fine-tunes Llama-2 and CodeLlama models with special block tokens and an execution-aware inference loop that runs generated code in a notebook and feeds results back into decoding. The resulting MathCoder family reaches open-source state of the art at the time, with up to `83.9%` on GSM8K and `45.2%` on MATH, showing that tool-mediated execution materially improves mathematical reasoning.

## Problem & Motivation

Open-source LLMs substantially lagged behind closed-source systems such as GPT-4 Code Interpreter on math benchmarks, especially when problems required both multi-step reasoning and precise computation. Prior open-source math tuning pipelines mainly relied on chain-of-thought supervision, while program-oriented approaches often separated code from reasoning rather than letting the model alternate between them. The paper targets this gap by asking whether open-source models can be taught to behave more like GPT-4 Code Interpreter: reason in natural language, emit code when computation is needed, execute that code, and continue reasoning conditioned on the execution result.

## Method

- **Dataset decomposition**: MathCodeInstruct is defined as `D = {D_0, D_1}`, where `D_0` is seed data from GSM8K and MATH with GPT-4-generated solutions, and `D_1` is additional synthetic data generated by the paper's interpolation prompting strategy.
- **LCE supervision format**: each solution is represented as an interleaved sequence `y_i = (L, C, E, L, C, E, ...)`, where `L` is natural-language reasoning, `C` is Python code, and `E` is the execution output consumed by later reasoning steps.
- **Special tokens**: the model is trained to delimit blocks with `<|text|>`, `<|code|>`, `<|execution|>`, and `<|endofblock|>`, so it can identify reasoning blocks, code blocks, and execution results at generation time.
- **Seed-data filtering**: GPT-4 solutions for GSM8K and MATH training examples are filtered so that the produced answer matches the gold answer before being retained in `D_0`.
- **Problem interpolation**: GPT-4 is prompted with one easier GSM8K problem and one harder MATH problem, then asked to create a new problem whose difficulty falls between them; the paper reports `83.2%` of generated problems are harder than GSM8K and `95.6%` are easier than MATH.
- **Self-distillation step**: CodeLlama-34B is first fine-tuned on `D_0` to create MathCoder-Initial, then used to generate LCE solutions for the interpolation problems in `D_1`.
- **Final model family**: the combined corpus reaches `80k` training examples and is used to fine-tune Llama-2 (`7B`, `13B`, `70B`) and CodeLlama (`7B`, `13B`, `34B`) into MathCoder-L and MathCoder-CL variants.
- **Training objective**: supervised fine-tuning uses standard cross-entropy, but the loss is computed only on reasoning-language and code tokens; execution-result tokens are masked out because the deployed system does not need to predict them.
- **Training hyperparameters**: the paper uses learning rate `2e-5`, context length `2048`, global batch size `128`, cosine decay for `3` epochs, and `50` warmup steps, with DeepSpeed ZeRO-3 and FlashAttention for efficiency.
- **Inference loop**: generated code blocks are executed in a persistent Jupyter notebook so later blocks can reuse variables; each execution result is appended to the context before the model continues autoregressive decoding.
- **Inference limits**: decoding is greedy, each block can generate up to `512` new tokens, and each solution can contain at most `32` LCE blocks.

## Key Results

- MathCoder-CL-34B reaches `81.7%` on GSM8K, `45.2%` on MATH, `82.5%` on SVAMP, `75.9%` on Mathematics, and `65.8%` on SimulEq, for an average of `70.2%`.
- MathCoder-L-70B reaches `83.9%` on GSM8K and `45.1%` on MATH, outperforming ChatGPT-3.5 (`80.8%`, `34.1%`) and PaLM-2 (`80.7%`, `34.3%`) on both benchmarks; the CodeLlama-based 34B model also exceeds GPT-4's reported `42.5%` on MATH.
- The smallest MathCoder-L-7B surpasses WizardMath-70B on three of five datasets and improves average score from `46.2%` to `50.7%`, despite being far smaller.
- Adding interpolation problems improves CodeLlama-34B from `66.2%` average on the `49k` seed-only set to `70.2%` on the `80k` full set, including gains of `+4.4` on GSM8K and `+1.2` on MATH.
- Actual code execution during inference is crucial: with identical training that includes execution-result tokens, average accuracy jumps from `35.1%` without execution to `69.1%` with execution, a `+34.0` point gain.
- Masking execution-result tokens out of the training loss further improves the execution-enabled model from `69.1%` to `70.2%` average.

## Limitations

- The pipeline depends on GPT-4 to generate seed solutions and interpolation problems, so data quality, cost, and capability ceilings inherit teacher-model limitations.
- Despite strong open-source results, there remains a large gap to GPT-4 Code Interpreter on core benchmarks, especially MATH (`45.2%` vs. `69.7%`).
- The paper explicitly notes weak performance on geometry and theorem-proving-style problems; as a unimodal system, MathCoder does not address diagram-heavy reasoning.
- Training data is anchored in GSM8K and MATH, so the approach may under-cover broader mathematical domains or non-math tool-use tasks.
- Evaluation focuses on benchmark accuracy and ablations, but gives limited analysis of execution failures, notebook-state robustness, or deployment safety.
