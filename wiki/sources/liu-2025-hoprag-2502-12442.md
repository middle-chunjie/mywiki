## Summary

HopRAG proposes a logic-aware retrieval framework for multi-hop RAG that augments similarity search with graph exploration over passages. Instead of building a knowledge graph over extracted entities, it stores passages as graph vertices and uses LLM-generated pseudo-queries to define directed logical edges between them. Retrieval follows a `retrieve -> reason -> prune` pipeline: hybrid retrieval seeds the search, an LLM performs multi-hop traversal by choosing helpful outgoing edges, and a Helpfulness score re-ranks visited passages by combining textual similarity with traversal importance. On MuSiQue, 2WikiMultiHopQA, and HotpotQA, the method improves average answer quality over BM25, dense retrieval, reranking, RAPTOR, GraphRAG, HippoRAG, and SiReRAG in most tested settings while keeping the graph sparse enough for practical traversal.

## Problem & Motivation

Standard RAG retrievers optimize lexical or semantic similarity, but multi-hop QA needs logical relevance: a passage can be indirectly useful because it leads to another passage that contains the missing fact. The paper argues that imperfect retrieval hurts both precision and recall: dense retrieval often returns semantically similar but only indirectly relevant passages, and even the best recall in the authors' setting saturates around `0.45`. HopRAG is motivated by the observation that these partially useful passages can serve as stepping stones if retrieval is allowed to hop through a passage graph instead of stopping at a one-shot top-`k` list.

## Method

- **Graph index**: build a passage graph `G = (V, E)` where each vertex stores one passage and each directed edge represents a logical hop between passages rather than a schema-bound KG relation.
- **Query simulation**: for each passage `p_i`, use an LLM to generate out-coming questions `Q_i^+` that the passage raises but cannot answer, and in-coming questions `Q_i^-` that the passage can answer. In the implementation details, each vertex is prompted to produce at least `2` in-coming questions and `4` out-coming questions.
- **Hybrid edge matching**: convert each pseudo-query into keywords via `NER(.)` and embeddings via `EMB(.)`, then pair out-coming and in-coming triplets with `SIM(r_s^+, r_t^-) = (Jaccard(k_s^+, k_t^-) + cosine(v_s^+, v_t^-)) / 2`. The best-matching pair defines a directed edge and its attached pseudo-query payload.
- **Reasoning-augmented traversal**: start from `top_k` seed vertices retrieved for query `q`; perform breadth-first local search for `n_hop` rounds. For each current vertex, an LLM selects the most helpful outgoing edge to enqueue one neighbor, and a counter `C_count` records how often each vertex is visited.
- **Pruning by Helpfulness**: compute `H_i = (SIM(v_i, q) + IMP(v_i, C_count)) / 2`, where `IMP(v_i, C_count) = C_count[v_i] / sum_j C_count[v_j]`, then keep the final top-`k` vertices with highest `H_i`.
- **Experimental settings**: use BGE embeddings with `768` dimensions, GPT-4o-mini for pseudo-query generation and traversal reasoning, GPT-4o and GPT-3.5-turbo as readers, `top_k = 20`, `n_hop = 4`, `max_tokens = 2048`, `temperature = 0.1`, and a sparse graph budget retaining only `O(n log n)` edges.

## Key Results

- With GPT-3.5-turbo readers and `top_k = 20`, HopRAG reaches average `EM/F1 = 54.00/66.76`, compared with `53.93/65.88` for SiReRAG, `52.97/64.03` for HippoRAG, `49.43/60.38` for RAPTOR, and `34.67/43.60` for reranking.
- With GPT-4o readers and `top_k = 20`, HopRAG achieves average `EM/F1 = 55.10/66.40`, outperforming BGE (`36.17/45.14`), RAPTOR (`49.40/61.21`), and GraphRAG (`22.10/30.15`).
- The paper reports that over `60%` of retrieved passages are indirectly relevant or irrelevant in their motivating analysis, supporting the need for logic-aware retrieval beyond one-shot similarity search.
- HopRAG remains competitive at smaller contexts: with `top_k = 12`, average GPT-3.5-turbo answer scores are still `51.57/63.78`, while traversal expands evidence with only `12` final passages.
- Increasing traversal depth from `n_hop = 1` to `4` raises average retrieval F1 from `8.08` to `14.96`, but also increases average traversal-model calls from `19.92` to `38.53`.

## Limitations

- The gains come with extra traversal cost: at `top_k = 20` and `n_hop = 4`, the system makes `38.53` average LLM calls per query, which the appendix estimates as a major latency bottleneck.
- Edge quality depends on pseudo-query generation, keyword extraction, and embedding quality; noisy pseudo-queries or weak NER can create poor graph connectivity.
- The evaluation is concentrated on multi-hop QA benchmarks, so it does not establish effectiveness on open-ended generation, summarization, or non-QA enterprise RAG workloads.
- Retrieval F1 decreases as `top_k` grows because larger candidate sets pull in more negatives, so better answer quality is partly purchased by tolerating noisier retrieval sets.
- HopRAG is not uniformly best on every individual setting; the paper itself notes exceptions such as HotpotQA against SiReRAG and 2WikiMultiHopQA against HippoRAG in some tables.
