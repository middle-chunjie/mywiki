## Summary

⚠ Possibly outdated: published 2022; re-verify against recent literature.

This paper studies natural-language-to-code generation as a program selection problem rather than only a decoding problem. Starting from few-shot Codex samples, the authors introduce MBR-EXEC, which scores each candidate by how well its execution results agree with other sampled programs on a small set of test inputs. The key idea is that pretrained code models often distribute probability mass over multiple surface forms that are semantically equivalent, so consensus under execution can recover a stronger single answer than plain likelihood or greedy decoding. Across MBPP, Spider, and NL2Bash, the method consistently improves single-program performance over execution-unaware baselines, with especially large gains on benchmarks evaluated by execution accuracy.

## Problem & Motivation

Pretrained code models can generate many plausible candidate programs from a natural-language specification, but picking one correct program remains difficult because string overlap and token likelihood are poor proxies for semantic correctness. The paper targets a realistic setting where only one submission can be returned, assumes no execution supervision during training, and asks whether limited inference-time execution can exploit semantic agreement across samples to improve final program choice.

## Method

- **Candidate collection**: query Codex with `5` different prompts, each formatted as `3-shot` demonstrations plus the new problem, then sample candidate programs with temperature `0.3`.
- **MBR objective**: given candidates `\mathcal{P} = \{p_i\}_{i=1}^N`, select `\hat{p} = \arg\min_{p \in \mathcal{P}} \sum_{p_{ref} \in \mathcal{P}} \ell(p, p_{ref})`, i.e. the candidate with the lowest Bayes risk against other samples.
- **Execution-based loss**: define `\ell(p_i, p_j) = \max_{t \in \mathcal{T}} \mathbf{1}[p_i(t) \neq p_j(t)]`, so two programs incur zero loss only when they produce identical outputs on all available test inputs.
- **Tie-breaking**: if multiple candidates share the same minimum risk, choose the one with the highest model likelihood.
- **Approximate semantic equivalence**: exact program equivalence is intractable, so the method uses execution on a small test set as a behavioral proxy for semantic sameness.
- **Dataset-specific prompting**: MBPP uses one visible assertion as `[INFO]`; Spider prepends schema text; NL2Bash omits `[INFO]` and substitutes execution similarity with token-level BLEU over bashlex parses.
- **Baselines**: compare against greedy decoding, temperature sampling, maximum likelihood (`ML`), mean token log-likelihood (`MALL`), and BLEU-based MBR (`MBR-charBLEU` / `MBR-tokenBLEU`).
- **Inference budget**: default experiments use `15` training examples split into groups of `3`, execution limits of `128GB` memory and `10s` runtime for MBPP and Spider, and one test input for Bayes-risk computation unless otherwise noted.

## Key Results

- On the main comparison, `MBR-EXEC` reaches `58.2 ± 0.3` on MBPP, `63.6 ± 0.8` on Spider, and `58.5 ± 0.3` on NL2Bash.
- Relative to greedy `3-shot` decoding, the gains are `+10.9` on MBPP (`47.3 -> 58.2`), `+12.8` on Spider (`50.8 -> 63.6`), and `+5.7` on NL2Bash (`52.8 -> 58.5`).
- Relative to temperature sampling without semantic selection, the gains are `+10.5` on MBPP (`47.7 -> 58.2`), `+15.1` on Spider (`48.5 -> 63.6`), and `+5.5` on NL2Bash (`53.0 -> 58.5`).
- When MBR-EXEC is run over greedy vs sampled candidates, sampled candidates still perform better: MBPP `58.2` vs `56.0`, Spider `63.6` vs `62.1`, NL2Bash `58.5` vs `58.4`.
- Lower sampling temperatures work best: the paper recommends temperatures below `0.5`, and shows that temperatures near `1.0` sharply degrade MBPP and Spider.
- Using multiple groups of `3-shot` prompts outperforms concatenating all `15` examples into one prompt on MBPP and NL2Bash.

## Limitations

- The method sits entirely on top of a frozen Codex model and does not incorporate execution information during training or fine-tuning.
- Semantic equivalence is approximated with limited test cases, so two incorrect programs can still agree, and incomplete tests can miss behavioral differences.
- The strongest evidence is on inference-time reranking; the paper does not study joint training with execution feedback.
- NL2Bash is not executed in a true sandboxed environment, so its "execution" signal is a proxy based on bash parsing and BLEU rather than real command behavior.
- The approach assumes access to usable test inputs and enough execution budget at inference time, which may not hold in every deployment.
