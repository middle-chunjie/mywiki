## Summary

This paper introduces CodeI/O, a data-construction and intermediate-training framework that converts real Python programs into executable natural-language reasoning tasks. Each sample pairs cleaned reference code, a textual query, and either an input-prediction or output-prediction target, with chain-of-thought rationales synthesized by DeepSeek-V2.5. Starting from `810.5K` raw code files, the pipeline yields `454.9K` executable functions and `3.5M` supervised instances spanning logic, symbolic, scientific, mathematical, and commonsense reasoning. The authors further add execution-verified revision turns to build CodeI/O++, then train four base models in a two-stage setup. Across `14` benchmarks and model sizes from `7B` to `27B`, CodeI/O delivers more balanced reasoning gains than raw-code pretraining or several strong synthetic-data baselines.

## Problem & Motivation

The paper targets a data bottleneck in general reasoning for large language models: math and code generation have abundant structured supervision, but logical, symbolic, scientific, and commonsense reasoning data are much sparser and more fragmented. The authors argue that raw code contains rich procedural reasoning patterns, but standard raw-code pretraining leaves those signals implicit and entangled with programming syntax. They therefore reformulate code as a natural-language input/output prediction task so the model learns reusable reasoning structure without needing to emit executable syntax.

## Method

- **Raw-code collection**: aggregate `810.5K` Python files from CodeMix (`~427K`), PyEdu-R (`~369K`), and other curated sources (`~14.5K`) to maximize diversity of reasoning patterns rather than only algorithm practice.
- **Unified transformation**: use DeepSeek-V2.5 to rewrite each file into a cleaned executable package containing a reference program, a main entrypoint with JSON-serializable inputs/outputs, an input/output specification, a rule-based input generator, and a concise natural-language query.
- **Input/output sampling**: execute transformed functions on multiple generated test cases, discard non-executable or random programs, and enforce constraints such as runtime `< 5s`, total object size `< 1024` bytes, list/dict length `< 20`, string length `< 100`, and other object size `< 128` bytes.
- **Training objective**: build roughly balanced input-prediction and output-prediction tasks (`50%/50%`) where the prompt contains the query, code, and given I/O context, and the response is a free-form natural-language CoT rather than code.
- **Scale after filtering**: retain `454.9K` raw files and `3.5M` trainable instances after removing failed transformations, over-complex objects, and samples exceeding runtime limits.
- **Execution-verified refinement**: form CodeI/O++ by checking predictions against ground-truth outputs or by re-executing the code with predicted inputs, then appending execution feedback and prompting one extra revision turn; about `50%` of first-turn responses are already correct, and roughly `10%` of the incorrect ones are fixed after revision overall.
- **Training setup**: apply a two-stage regime with stage 1 on CodeI/O for `1` epoch at learning rate `1e-5` for smaller models or `4e-6` for Gemma 2 27B, batch size `1024`; then stage 2 instruction-tuning for `700` steps at learning rate `3e-5` or `1e-5`, cosine decay to `1e-6` or `3e-7`, no warmup, and maximum sequence length `4096`.

## Key Results

- On Qwen 2.5 Coder 7B, average score rises from `54.8` (instruction tuning only) to `57.2` with CodeI/O and `57.7` with CodeI/O++; LeetCode-O improves from `20.7` to `23.7/24.2`, and KorBench from `38.7` to `44.3/45.7`.
- On LLaMA 3.1 8B, the average improves from `49.3` to `51.2` with CodeI/O and `52.1` with CodeI/O++; CRUXEval-O increases from `46.9` to `53.3/53.1`, and LeetCode-O from `4.1` to `9.3/10.0`.
- On DeepSeek Coder v2 Lite 16B and Gemma 2 27B, the average moves from `51.6 -> 53.6/53.5` and `59.5 -> 60.9/61.5`, showing gains across architectures and parameter scales.
- Scaling matters: a `~50%` subset of CodeI/O (`1.59M` samples) scores `56.7` average versus `57.2` for the full `3.52M` set; input-only and output-only variants score `56.1` and `56.4`, both below the full mixed objective.
- Rejection sampling is not the best use of verifiability: removing wrong responses drops the average to `56.5`, and replacing wrong outputs with ground truth reaches only `56.6`, both below the main `57.2` result.
- Leakage analysis reports `21.5%` potential overlap on LeetCode-O, `5.1%` on KorBench, and `0.0-0.1%` on most other benchmarks; similar gains on full and non-leaked subsets suggest the improvements are not mainly contamination-driven.

## Limitations

- The pipeline depends heavily on DeepSeek-V2.5 for code refactoring and CoT synthesis, so teacher errors or stylistic biases can propagate into the training corpus.
- The method is limited to executable Python programs with JSON-serializable I/O and fairly strict complexity caps, which excludes many richer software behaviors and non-code reasoning settings.
- Gains are broad but not universal: the paper reports small regressions or near-no-change on some benchmark/model pairs, and CodeI/O++ is slightly worse than CodeI/O when the second-stage dataset is switched to Tulu-3.
- Potential benchmark overlap is not fully eliminated for LeetCode-O and KorBench, and extending revision beyond one extra turn shows rapidly diminishing returns.
