## Summary

ReCo extends [[generation-augmented-retrieval]] for [[code-search]] by augmenting not only the natural-language query with LLM-generated exemplar code but also every candidate program with an LLM-rewritten variant intended to normalize implementation style. The rewriting pipeline first summarizes code into a concise purpose statement and then regenerates code from that summary so rewritten candidates better match the latent style of query-conditioned exemplars. The paper also introduces [[code-style-similarity]], a metric that combines variable-name distance, API-invocation distance, and AST tree-edit distance to quantify stylistic alignment. Across BM25, zero-shot dense retrievers, and fine-tuned dense retrievers on CoNaLa, MBPP, APPS, and MBJP, ReCo consistently improves MRR over GAR, with especially large gains on MBPP and APPS.

## Problem & Motivation

Code search suffers from strong modality mismatch between short natural-language queries and code snippets. Prior [[generation-augmented-retrieval]] helps by asking a [[large-language-model]] to generate exemplar code from the query, but the authors argue that code search is harder than passage retrieval because many implementations satisfy the same intent while differing substantially in naming, library usage, and control structure. As a result, exemplar code can be semantically correct yet stylistically far from the relevant code stored in the corpus, limiting lexical overlap for sparse retrievers and weakening representation alignment for dense retrievers. ReCo is motivated by the idea that style normalization on the codebase side can make GAR-style query expansion materially more useful.

## Method

- **Base scoring setup**: code search ranks a query `q` and code snippet `c` by dot-product similarity `sim(q, c) = <G(q), G(c)>`, where `G` maps text and code into a shared embedding space.
- **Exemplar-code generation**: for each query `q`, the model uses [[few-shot-prompting]] with `K = 4` in-context query-code pairs and the instruction "Please generate a Java/Python code snippet according to the given description." to sample exemplar code `c_q = LLM(q, GEN)`.
- **Code rewriting via summarize-then-generate**: each corpus code `c` is first converted into a one-sentence summary `q_sum = LLM(c, SUM)`, then regenerated as rewritten code `c_c = LLM(q_sum, GEN)`. The paper explicitly chooses this summarize-then-generate path instead of direct rewriting because direct rewrite prompts changed code only marginally.
- **Sparse retrieval augmentation**: the augmented query is `q^+ = concat({q} x N, {c_q1, ..., c_qN})`, and each candidate code is analogously expanded with rewritten variants. Repeating the original text `N` times is meant to preserve original semantics while adding generated code tokens.
- **Dense retrieval training**: fine-tuned models optimize [[infonce-loss]] with in-batch negatives, `L = -E[log exp(v_qi^+ · v_ci^+) / (exp(v_qi^+ · v_ci^+) + sum_{j != i} exp(v_qi^+ · v_cj^+))]`.
- **Augmented dense representations**: the paper assumes LLM-generated code styles are roughly unimodal and average-pools original and generated representations, `v_q^+ = (N · G(q) + sum G(c_q)) / (2N)`, with the same construction applied to `v_c^+`.
- **Code Style Similarity (CSSim)**: style distance averages three normalized terms, `CSDis(c1, c2) = (Dis_Var + Dis_API + TED) / 3`, and `CSSim = 1 - CSDis`. `Dis_Var` and `Dis_API` are IDF-weighted nearest-neighbor edit distances over variable names and API calls; `TED` is tree-edit distance over ASTs.
- **Prompting and training hyperparameters**: prompting uses temperature `1`, output length `128` for code summarization and `256` for code generation, with re-sampled in-context examples across repeated generations. Dense models use maximum query/code length `256`, Adam with learning rate `5e-6`, mean pooling, L2 normalization, batch size `32`, and `10` fine-tuning epochs on a single Nvidia Tesla A100 GPU.

## Key Results

- On BM25, ReCo improves MRR over GAR from `71.7` to `75.8` on CoNaLa, `35.1` to `70.8` on MBPP (`+35.7`), `17.6` to `22.6` on APPS (`+5.0`), and `33.5` to `65.3` on MBJP (`+31.8`) under the paper's best setting.
- In zero-shot dense retrieval, ReCo raises Contriever from `14.0` to `41.6` on APPS (`+27.6`), `71.3` to `87.4` on MBPP (`+16.1`), and `62.3` to `76.6` on MBJP (`+14.3`); UniXcoder also improves from `85.0` to `92.4` on MBPP.
- In fine-tuned dense retrieval, ReCo lifts UniXcoder from `34.5` to `58.1` on APPS (`+23.6`) and from `89.0` to `94.2` on MBPP (`+5.2`); CodeBERT improves from `84.1` to `89.1` on MBJP (`+5.0`).
- CSSim better tracks retrieval gains than CodeBLEU, ROUGE-L, or BLEU: when BM25 chooses the best exemplar by CSSim instead of random selection, MRR rises from `41.3` to `43.6` on CoNaLa and from `26.6` to `29.9` on MBJP.
- The ablation using only LLM-generated codes still remains competitive: zero-shot UniXcoder with ReCo reports `83.1` on CoNaLa, `84.4` on MBPP, `26.8` on APPS, and `78.1` on MBJP without the original query/code text.

## Limitations

- ReCo still depends on real-time generation of exemplar code, so latency remains a practical bottleneck even though rewritten corpus code can be pre-computed offline.
- The method assumes summarize-then-generate preserves program semantics; the paper gives intuition and theory, but does not directly verify semantic equivalence of rewritten code at scale.
- Evaluation excludes very large codebases such as CodeSearchNet because rewriting an entire corpus is expensive; the authors estimate that generating a single exemplar per query there would take more than two months.
- Gains are not uniform across all settings: for example, fine-tuned CodeBERT on CoNaLa drops from `88.6` with GAR to `85.0` with ReCo in the best reported table.
- Performance depends on LLM behavior; Code Llama-34B can underperform because it sometimes continues generating code for in-context examples and truncates the actual target output.
