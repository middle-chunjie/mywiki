## Summary

FrugalRAG studies multi-hop QA under an efficiency objective rather than optimizing only answer accuracy or retrieval recall. The paper argues that a strong prompted [[retrieval-augmented-generation]] baseline based on ReAct is already competitive, so the real opportunity is to reduce unnecessary searches while preserving evidence quality. Its solution splits training into an exploration stage that maximizes evidence coverage and a reinforcement-learning stage that learns when to stop retrieving. With only `1000` training examples, the method achieves competitive or better answer and retrieval metrics than stronger baselines on HotPotQA, 2WikiMultiHopQA, and MuSiQue, while substantially lowering average search count. The central claim is that adaptive test-time retrieval is more data-efficient and latency-aware than scaling retrieval budget uniformly.

## Problem & Motivation

Multi-hop QA over large corpora requires the model to iteratively issue search queries, gather evidence, and decide when enough information has been collected to answer. Prior work mainly optimizes answer quality or recall and often relies on tens of thousands of labeled examples, but production RAG systems also care about inference latency, which in this setting is closely tied to the number of retrieval calls. The paper argues that recent RL-based RAG systems often spend more searches to improve quality, while a stronger few-shot ReAct baseline already covers much of the accuracy gap. FrugalRAG therefore targets the efficiency-performance frontier: preserve evidence quality and answer quality, but spend fewer searches on easy questions and more only when the question actually needs it.

## Method

- **Setup**: a policy LM `f` iteratively emits thought-action-search triples `(T_h, A_h, S_h)` for hops `h = 1 ... B`, retrieves documents `D_h = R(S_h)`, and stops when it outputs `finish`; a separate generator `g` then answers from `Q` plus the accumulated context.
- **Context state**: at hop `h`, the model conditions on the question and prior trajectories, i.e. `Q ∪ ⋃_{k=0}^{h-1}(D_k, T_k, A_k, S_k)`, so retrieval and reasoning are explicitly interleaved.
- **Stage 1, evidence coverage maximization**: FrugalRAG uses ReAct rollouts with bootstrapped few-shot prompts, evaluates multiple candidate queries per hop against ground-truth supporting documents, and keeps the highest-recall trajectory to build the supervised dataset.
- **Prompt/data construction**: prompt optimization uses `50` seed examples and DSPy `BootstrapFewShotWithRandomSearch`, selects the best `4` prompt sets out of `15`, then generates a Stage-1 dataset from `1000` training questions with maximum budget `B = 6`.
- **Stage-1 supervised training**: full-parameter SFT runs for `1` epoch with `lr = 2e-5`, `weight_decay = 0.01`, batch size `4`, gradient accumulation `2`, effective batch size `8`, max sequence length `4096`, and `20` warmup steps on `8` H100 `80GB` GPUs.
- **Stage 2, stop-policy RL**: instead of using RL to encourage more searches, the paper learns when to stop retrieving; it defines the optimal stopping hop `h*` as the smallest hop count that matches FrugalRAG-Explore's recall within budget.
- **Reward design**: the main reward depends on the normalized stop error `Δ = (h_term - h*) / B`, recall `c`, threshold `τ`, and cap `R_max`; it rewards exact stopping when `Δ = 0` and penalizes both late stopping and early stopping via clipped `log((1 - Δ) / Δ)` terms.
- **Format reward**: a separate `R_f ∈ {-1, +1}` rewards adherence to the ReAct output format so malformed tool calls are directly penalized.
- **Stage-2 optimization**: GRPO samples `v = 8` rollouts per hop, uses KL penalty weight `0.1`, `R_max = 2.0`, max completion length `256`, max prompt length `1024`, total batch size `48`, learning rate `1e-6`, and `400` RL steps with DeepSpeed-Zero2 on `7` H100 GPUs plus `1` sampling GPU.
- **Evaluation protocol**: experiments use HotPotQA, [[2wiki-multihopqa]], and [[musique]] with a [[colbertv2]] retriever over Wikipedia-derived indices, and report answer metrics (`F1`, `EM`, `Match`), retrieval metrics (`Recall`, `Support F1`), and average search count.

## Key Results

- On HotPotQA with Qwen2.5-7B-Instruct, FrugalRAG reaches `63.03` F1, `48.81` EM, `58.25` Match, `83.86` Recall, and `87.43` Support F1 using only `2.75` searches on average.
- On HotPotQA with Llama3.1-8B-Instruct, FrugalRAG attains `62.95` F1 and `79.62` Recall with `2.96` searches, versus FrugalRAG-Explore's `60.58` F1 and `83.11` Recall with `5.96` searches.
- On 2WikiMultiHopQA with Qwen2.5-7B-Instruct, FrugalRAG reports `45.00` F1, `36.13` EM, `47.00` Match, `54.38` Recall, and `64.32` Support F1 at `4.86` searches.
- On MuSiQue with Qwen2.5-7B-Instruct, FrugalRAG achieves `24.78` F1, `14.89` EM, `21.84` Match, `36.71` Recall, and `31.86` Support F1 with `3.98` searches.
- Averaged over models, the Stage-2 policy reduces search count by `53.05%` on HotPotQA, `19.87%` on 2WikiMultiHopQA, and `21.17%` on MuSiQue relative to FrugalRAG-Explore, while improving F1 by `3.5%`, `5.65%`, and `0.68%` respectively.
- Against LeReT on HotPotQA, FrugalRAG reports higher recall (`79.62` vs `77.10`) with only `+0.96` extra searches on average for the Llama3.1-8B setting.

## Limitations

- Training still needs ground-truth supporting documents, so the method is not label-free and may be hard to port to domains without document-level supervision.
- The answer generator `g` is off-the-shelf rather than jointly optimized with the retriever/decomposer, which the authors cite as one reason for weaker 2Wiki answer metrics than Search-R1.
- Evaluation is limited to three multi-hop QA benchmarks and mostly one retrieval stack family, so robustness to different domains, retrievers, and corpora remains only partially tested.
- The method assumes FrugalRAG-Explore defines the best achievable recall under budget when computing `h*`, which is a practical but heuristic target for RL.
- The paper leaves open whether the two-stage recipe scales cleanly to much larger training sets or to settings with no in-domain examples.
