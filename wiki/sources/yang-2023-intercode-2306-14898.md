## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

InterCode reframes natural-language-to-code as an interactive decision-making problem with execution feedback instead of a single-shot text generation task. The paper defines interactive coding as a POMDP, packages execution inside reproducible Docker containers, and instantiates Bash, SQL, and Python environments by adapting NL2Bash, Spider, and MBPP. It evaluates multiple frontier 2023 models and prompting strategies, showing that multi-turn interaction substantially improves task completion over single-turn generation, especially on SQL, and that reasoning style matters: ReAct helps SQL more than Bash, while rigid planning struggles to exploit feedback. The benchmark's main contribution is a standardized, extensible interface for studying code generation, debugging, context discovery, and agent-environment interaction under executable supervision.

## Problem & Motivation

Most code-generation benchmarks at the time treated the task as static sequence transduction from a natural-language instruction to a final program. The authors argue that this misses how human programming actually works: people iteratively write, execute, inspect outputs, debug, and revise. Static evaluation therefore hides three important issues: small errors can propagate without recovery, generated code is disconnected from its real execution environment, and there is little room for adaptive exploration or collaboration. InterCode is proposed to standardize this interactive setting so methods that use execution feedback, context discovery, and multi-step reasoning can be compared under a common benchmark and environment API.

## Method

- **Task formalization**: InterCode models interactive coding as a POMDP `(\mathcal{U}, \mathcal{S}, \mathcal{A}, \mathcal{O}, \mathcal{T}, \mathcal{R})`, where natural-language instruction `u ∈ \mathcal{U}` is given, code is the action `a_t ∈ \mathcal{A}`, execution feedback is observation `o_t ∈ \mathcal{O}`, and a special `submit` action ends the episode with reward `r ∈ [0, 1]`.
- **Environment design**: each task is built from `(1)` a Dockerfile, `(2)` a dataset with at least `query` and `gold`, and `(3)` a reward function. Docker provides stateful, reproducible, and isolated execution; Bash and SQL environments are described as implemented in under `200` lines each.
- **Task instantiations**: the benchmark adapts three static datasets into interactive settings: Bash on NL2Bash with `200` tasks, SQL on Spider dev with `1034` tasks, and Python on MBPP with `117` tasks.
- **Bash environment**: actions run in an Ubuntu terminal with four swappable file systems. The reward combines execution-output similarity, file-system-diff overlap, and correctness of modified files; the extracted markdown mangles part of the denominator, but the paper clearly assigns weights `0.34`, `0.33`, and `0.33` to those three components.
- **SQL environment**: actions execute against a MySQL setup converted from Spider. Reward is `R = IoU(A, G) * ((τ + 1) / 2)`, where `IoU = |A ∩ G| / |A ∪ G|` scores record overlap and Kendall-style ordering adjustment penalizes misordered outputs.
- **Python environment**: actions interact with a Python interpreter inside Ubuntu; correctness is inherited from MBPP unit tests, so task completion corresponds to passing all tests.
- **Interface and logging**: `InterCodeEnv` inherits the OpenAI Gym API with `reset`, `step`, `close`, trajectory saving, and overridable `execute_action`, `get_reward`, and `reset_container` hooks for new tasks.
- **Prompting/evaluation setup**: the paper compares Single Turn, Try Again, ReAct, and Plan & Solve. For Try Again and ReAct, the horizon is `n = 10` turns; models include OpenAI (`text-davinci-003`, `gpt-3.5-turbo`, `gpt-4`), PaLM-2 (`text-bison-001`, `chat-bison-001`), and open-source (`Vicuna-13B`, `StarChat-16B`).
- **Inference hyperparameters**: OpenAI calls use `temperature = 0`, `top_p = 1`, `max_tokens = 512`, `n = 1`; PaLM uses `temperature = 0`, `top_p = 1`, `candidate_count = 1`; open-source models use `max_new_tokens = 100`, `temperature = 0.01`; observations are truncated to `1000` tokens.
- **Validation**: the authors run dummy-agent checks to ensure gold actions are admissible and receive reward `1`, and unit-test the data loader to confirm dataset-format enforcement.

## Key Results

- On InterCode-SQL, GPT-4 improves from `9.1%` Single Turn success to `73.7%` Try Again overall; on hard and extra-hard subsets it rises from `4.6% → 66.7%` and `3.0% → 52.4%`.
- On InterCode-SQL, `gpt-3.5-turbo` improves from `10.5%` to `47.3%` overall under interaction; on extra-hard queries it goes from `3.6%` to `21.1%`.
- On InterCode-Bash, overall success improves for every tested model; GPT-4 rises from `34.0%` to `48.5%`, and `gpt-3.5-turbo` from `34.5%` to `46.5%`.
- For `gpt-3.5-turbo`, ReAct improves SQL from `47.3%` success / `7.25` turns / `46.4%` error to `58.7%` / `5.30` / `6.94%`, but on Bash ReAct underperforms Try Again (`20.5%` vs. `46.5%` success).
- Even when full SQL schema is given, interaction still helps: `gpt-3.5-turbo` reaches `67.9%` Single Turn vs. `72.8%` Try Again overall, indicating that execution feedback adds more than just missing schema context.
- A human SQL worker solves `13/15` sampled tasks (`0.87` success rate), well above model baselines, with errors dominated by syntax mistakes rather than schema hallucinations.

## Limitations

- The benchmark currently covers only Bash, SQL, and Python action spaces, so claims about general interactive coding remain limited across programming languages and software-engineering settings.
- Performance plateaus across longer trajectories: later turns often repeat earlier mistakes, fail to leverage recent observations, or continue unproductive lines of reasoning.
- Strong gains are shown mostly from prompt engineering and interaction, not from new training algorithms or learned policies.
- The benchmark depends on executable sandbox construction and reward engineering, which can be labor-intensive for new domains.
- The paper itself notes that its CTF extension is still limited in scope and requires more systematic task curation and analysis.
