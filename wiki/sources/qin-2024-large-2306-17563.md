## Summary

This paper argues that large language models can be strong zero-shot text rankers when ranking is reformulated as pairwise comparison rather than pointwise scoring or listwise permutation generation. The proposed Pairwise Ranking Prompting (PRP) asks the model to decide which of two passages is more relevant to a query, supports both generation and scoring APIs, and aggregates pairwise preferences with `O(N^2)`, `O(N log N)`, or `O(N)` variants. On TREC-DL2019 and TREC-DL2020, PRP with open FLAN models matches or exceeds strong supervised rerankers and clearly surpasses prior zero-shot prompting methods. The central claim is that LLMs handle relative relevance judgments much better than calibrated point estimates or direct list permutation outputs.

## Problem & Motivation

Prior LLM ranking methods mainly used pointwise or listwise prompting, but both mismatch how off-the-shelf LLMs operate. Pointwise relevance generation needs calibrated scores that are comparable across prompts and usually requires access to token probabilities, while listwise prompting often yields missing, repeated, rejected, or unparsable rankings and is highly sensitive to document order. The paper targets zero-shot text reranking with accessible open models and asks whether a simpler comparison unit, pairwise preference, can preserve LLM reasoning ability while avoiding calibration and formatting failures.

## Method

- **Pairwise prompting unit**: define `u(q, d_1, d_2)` that asks the LLM which passage is more relevant to query `q`; PRP reduces ranking to repeated pairwise comparisons instead of asking for calibrated scores or full permutations.
- **Bidirectional consistency check**: query both `u(q, d_1, d_2)` and `u(q, d_2, d_1)` to mitigate order effects; infer `d_1 > d_2` only when both prompt orders agree, otherwise treat the pair as a tie `d_1 = d_2`.
- **Scoring mode and generation mode**: in scoring mode, compare log-likelihoods of the only two target strings, `"Passage A"` and `"Passage B"`; in generation mode, parse the generated choice directly. The main experiments use scoring mode.
- **PRP-Allpair**: compare all document pairs and aggregate global scores with `s_i = \sum_{j \ne i} \mathbb{I}_{d_i > d_j} + 0.5 \sum_{j \ne i} \mathbb{I}_{d_i = d_j}`; complexity is `O(N^2)` but comparisons can be parallelized.
- **PRP-Sorting**: plug pairwise preferences into Heapsort so reranking uses `O(N log N)` LLM calls while keeping relative order decisions local.
- **PRP-Sliding-K**: perform backward sliding-window passes over the initial ranking with stride `1`; one pass is `O(N)`, and `K` passes target top-`K` quality. The paper emphasizes `K = 10` for top-10 ranking metrics.
- **Experimental setup**: rerank the top `100` BM25 passages per query on [[trec-dl-2019]] and [[trec-dl-2020]], both over the `8.8M`-passage MS MARCO corpus, using FLAN-T5-XL `3B`, FLAN-T5-XXL `11B`, and FLAN-UL2 `20B`.

## Key Results

- On [[trec-dl-2020]], PRP-Sliding-10 with FLAN-UL2 reaches `85.80 / 75.35 / 70.46` on `NDCG@1 / NDCG@5 / NDCG@10`; PRP-Allpair reaches `85.19 / 74.73 / 70.68`, outperforming the reported GPT-4 RankGPT result `78.40 / 74.11 / 70.56` at `NDCG@1`.
- On [[trec-dl-2019]], PRP-Sliding-10 with FLAN-UL2 achieves `78.29 / 75.49 / 72.65`; PRP-Allpair reaches `73.64 / 74.77 / 72.42`. The method is below GPT-4 only on `NDCG@5` and `NDCG@10`, while staying competitive with or better than supervised rerankers on several metrics.
- Efficient variants remain strong: PRP-Sorting with FLAN-UL2 gets `84.57 / 72.52 / 69.43` on TREC-DL2020, and backward sliding passes steadily improve top-`K` metrics from `1` pass to `10` passes.
- PRP is much less order-sensitive than listwise prompting: on TREC-DL2019 with inverse BM25 order, RankGPT `gpt-3.5-turbo` drops from `82.17` to `36.43` at `NDCG@1`, while PRP-Allpair stays essentially unchanged (`73.64` to `74.42` at `NDCG@1`).
- Scoring and generation modes are nearly identical, and only about `0.02%` of pairwise generations violate the expected output format.

## Limitations

- PRP-Allpair still requires `O(N^2)` LLM calls, and even the more efficient variants can be costly for practical deployment at large candidate set sizes.
- The experiments only cover standard relevance-based reranking benchmarks (`TREC-DL2019`, `TREC-DL2020`) on BM25 top-`100` candidates, so non-standard ranking settings and full retrieval pipelines are not directly evaluated.
- The paper compares against GPT-based systems using prior reported numbers rather than running those closed models directly.
- The study remains zero-shot and does not address how to train LLMs to become more ranking-aware without sacrificing generality.
- Domain transfer beyond standard ad hoc passage relevance, such as counter-argument ranking, is left open.
