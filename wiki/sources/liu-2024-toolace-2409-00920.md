## Summary

ToolACE proposes an automated pipeline for training function-calling LLMs with synthetic data that is simultaneously diverse, appropriately difficult, and executable. The pipeline combines Tool Self-evolution Synthesis (TSS) to build a broad API pool, Self-Guided Dialog Generation (SDG) to generate single-turn, multi-turn, parallel, dependent, and non-tool-use dialogs, and Dual-Layer Verification (DLV) to filter structural and semantic errors. The resulting API pool contains 26,507 APIs across 390 domains. Fine-tuning LLaMA-3.1-8B-Instruct on ToolACE data yields 59.22 overall on BFCL-v3, including 89.27 non-live AST and 90.07 non-live executable accuracy, making the 8B model competitive with much larger or proprietary systems and substantially stronger than prior open-source function-calling models.

## Problem & Motivation

Real function-calling traces are hard to collect and annotate at scale, while existing synthetic pipelines often over-focus on simple single-turn tasks, limited public API sets, and weak accuracy control. The paper argues that tool-use training data should simultaneously cover broader API diversity, more realistic interaction structures, and executable parameter formatting. It also argues that training samples should be matched to the learner's capability: overly easy data is wasteful, while overly hard data is difficult to absorb. ToolACE is proposed to improve zero-shot generalization to unseen APIs and harder tool-use scenarios by synthesizing APIs and dialogs with explicit complexity control and verification.

## Method

- **Overall pipeline**: ToolACE combines Tool Self-evolution Synthesis (TSS), Self-Guided Dialog Generation (SDG), and Dual-Layer Verification (DLV) to synthesize tool descriptions and dialogs for function-calling supervision.
- **TSS for API diversity**: TSS performs `speciation -> adaptation -> evolution` over an API context tree extracted from pretraining-style documents. The final API pool reaches `26,507` APIs over `390` domains and includes nested parameter types such as lists of lists and lists of dictionaries.
- **Speciation and adaptation**: speciation recursively expands a hierarchical API context tree of domains and use cases; adaptation samples subtrees so each synthetic API gets a distinct scope and difficulty level instead of copying a fixed public API set.
- **Evolution step**: an LLM generates new API definitions from sampled functionality subtrees and example APIs, while diversity operators mutate functionalities, parameter types, constraints, and return fields to continuously enrich the API buffer.
- **Multi-agent dialog generation**: SDG uses `user`, `assistant`, and `tool` agents to generate `single`, `parallel`, `dependent`, and `non-tool-use` dialogs. The assistant can call APIs, ask clarifying questions, summarize tool outputs, or answer without tools, and only consistent assistant decisions across multiple generations are retained.
- **Complexity evaluation**: the learner model `M` scores a sample `(x, y)` by average token loss, `H_M(x, y) = -(1 / n_y) sum_i log p(t_i | x, t_1, ..., t_{i-1})`, so higher loss means higher learning difficulty. The paper links complexity to the number of candidate APIs, the number of utilized APIs, and the semantic dissimilarity between query and API descriptions.
- **Self-guided complication**: if a sample is too easy, the user agent is prompted to increase complexity by requiring more APIs or more indirect phrasing; if it is too hard, the query is simplified to stay within the learner's effective training range.
- **DLV rule layer**: rule-based verification checks API-definition completeness, JSON-schema-like structural validity, function-name correctness, required arguments, parameter formats and patterns, dialog structure, and consistency between function calls and tool responses.
- **DLV model layer**: expert LLM checkers screen for hallucinated arguments, dialog inconsistency, and tool-response mismatch that rule checks miss.
- **Training setup**: the main reported model is LLaMA-3.1-8B-Instruct fine-tuned with LoRA using `rank = 16`, `alpha = 32`, `learning rate = 1e-4`, `warmup ratio = 0.1`, `cosine` scheduling, `batch size = 48`, and `epochs = 3`.

## Key Results

- API diversity: ToolACE covers `26,507` APIs and `390` domains, versus `16,464` APIs / `49` domains for ToolLLM and `3,673` APIs / `21` domains for xLAM.
- BFCL-v3: ToolACE-8B reaches `59.22` overall, `89.27` non-live AST, `90.07` non-live executable, `73.21` live AST, `14.37` multi-turn, `85.37` relevance, and `83.81` irrelevance, ranking third on the leaderboard snapshot reported in the paper.
- API-Bank: ToolACE-8B reaches `75.94` on Call and `47.41` on Retrieval+Call, compared with `71.18` and `37.04` for raw LLaMA-3.1-8B-Instruct.
- Same-data comparison (`25,000` samples): ToolACE training reaches `58.19` BFCL overall, outperforming ToolLLM (`24.90`) and xLAM (`40.51`) under the paper's controlled comparison.
- Data-type ablation: removing parallel data drops BFCL overall to `50.60` and multi-turn to `1.75`; removing multi-type data keeps non-live AST high (`89.46`) but collapses irrelevance detection to `6.99`, showing that negative and heterogeneous samples matter.
- Complexity evaluator ablation: using the learner itself as evaluator gives the best reported BFCL overall (`59.22`) versus `57.61` with Qwen-7B and `57.67` with Qwen-14B as external evaluators.

## Limitations

- Complexity evaluation scales with both model size and dataset size, so its computational overhead becomes expensive as the learner or corpus grows.
- The paper notes that one-round non-uniform sampling can bias training toward easier examples and may keep the model in its comfort zone instead of progressively learning harder cases.
- ToolACE improves function calling strongly, but it still trails GPT-4 on broader reasoning and understanding benchmarks.
- Nested and dependent data help diversity, but benchmark coverage for those cases is limited, so their measured gains are smaller than for parallel or multi-type data.
