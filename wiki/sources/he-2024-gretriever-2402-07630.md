## Summary

The paper targets question answering over textual graphs, where nodes and edges carry natural-language attributes and the graph may be too large to serialize directly into an LLM prompt. It introduces the GraphQA benchmark by standardizing ExplaGraphs, SceneGraphs, and WebQSP into a common graph-plus-question format, and proposes G-Retriever, a retrieval-augmented graph QA architecture. The pipeline embeds node, edge, and query text with SentenceBERT, retrieves relevant graph elements with k-nearest neighbors, constructs a connected subgraph through Prize-Collecting Steiner Tree optimization, and answers with a frozen Llama2-7b conditioned on both a GNN-derived graph token and the textualized subgraph. Across all three datasets, the method improves accuracy or Hit@1 over prompt tuning baselines, sharply reduces prompt length on large graphs, and materially lowers hallucinated graph references.

## Problem & Motivation

Existing graph-LLM systems either focus on conventional prediction tasks such as node or graph classification, or answer only simple questions on small or synthetic graphs. For real textual graphs, directly flattening the full graph into text is brittle: token counts can exceed LLM context windows, irrelevant structure can drown out useful evidence, and graph-soft-prompt baselines can hallucinate nonexistent nodes or edges because they compress the whole graph into a single embedding.

The paper therefore frames the target problem as flexible graph question answering over real-world textual graphs and argues that the model should retrieve query-relevant graph evidence explicitly, keep the retrieved subgraph connected for interpretability, and preserve the language capabilities of the frozen LLM through parameter-efficient adaptation.

## Method

- **Task formalization**: a textual graph is defined as `` `G = (V, E, {x_n}_{n in V}, {x_e}_{e in E})` ``, where node text `` `x_n in D^{L_n}` `` and edge text `` `x_e in D^{L_e}` `` are sequences over a vocabulary `` `D` ``.
- **Benchmark construction**: the proposed [[graphqa]] benchmark unifies [[explagraphs]], [[scenegraphs]], and [[webquestionssp]] into a common graph-question-answer format; the datasets span commonsense reasoning, scene understanding, and [[knowledge-graph]] reasoning.
- **Indexing**: node and edge attributes are embedded with [[sentencebert]], giving `` `z_n = LM(x_n) in R^d` `` and analogous edge vectors; query text is encoded as `` `z_q = LM(x_q) in R^d` ``.
- **Initial retrieval**: relevant nodes and edges are obtained by cosine-similarity kNN, `` `V_k = argtopk_{n in V} cos(z_q, z_n)` `` and `` `E_k = argtopk_{e in E} cos(z_q, z_e)` ``, which instantiates [[nearest-neighbor-retrieval]] over graph elements.
- **Subgraph construction with PCST**: G-Retriever maps ranked nodes and edges to prizes, e.g. `` `prize(n) = k - i` `` for the top-`` `i` `` retrieved node, and solves `` `S* = argmax_S sum_{n in V_S} prize(n) + sum_{e in E_S} prize(e) - cost(S)` `` with `` `cost(S) = |E_S| * C_e` ``. When an edge has `` `P_e > C_e` ``, it is converted into a virtual node so the Prize-Collecting Steiner Tree solver can still optimize nonnegative edge costs.
- **Graph encoder**: the retrieved subgraph `` `S* = (V*, E*)` `` is encoded by a GAT-based [[graph-neural-network]] with `` `4` `` layers, `` `4` `` attention heads per layer, and hidden size `` `1024` ``, followed by mean pooling: `` `h_g = POOL(GNN_{phi_1}(S*)) in R^{d_g}` ``.
- **Projection and soft prompting**: an MLP maps the pooled graph vector into the LLM embedding space, `` `h^_g = MLP_{phi_2}(h_g) in R^{d_l}` ``. This graph token acts as a learned soft prompt, making the method a form of [[soft-prompt-tuning]] instead of full LLM fine-tuning.
- **Textual input to the LLM**: the retrieved subgraph is also serialized as text and concatenated with the question, yielding `` `h_t = TextEmbedder([textualize(S*); x_q]) in R^{L x d_l}` ``. The decoder then generates `` `p(Y | S*, x_q) = product_i p(y_i | y_<i, [h^_g; h_t])` `` with a frozen [[large-language-model]] backbone.
- **LLM and training settings**: the main generator is Llama2-7b, run with `` `10` `` virtual tokens, maximum input text length `` `512` ``, and maximum generation length `` `32` ``. Optimization uses AdamW with learning rate `` `1e-5` ``, weight decay `` `0.05` ``, batch size `` `4` ``, `` `10` `` epochs, and early stopping patience `` `2` ``; LoRA ablations use `` `r = 8` ``, `` `alpha = 16` ``, and dropout `` `0.05` ``.
- **Dataset-specific retrieval hyperparameters**: for SceneGraphs, `` `k = 3` `` for both nodes and edges with `` `C_e = 1` ``; for WebQSP, `` `k = 3` `` for nodes, `` `k = 5` `` for edges, and `` `C_e = 0.5` ``; for ExplaGraphs, the full graph is used by setting `` `k = 0` `` because the graphs are already small.

## Key Results

- **Frozen LLM + prompt tuning**: G-Retriever reaches `0.8696 ± 0.0129` on ExplaGraphs, `0.8614 ± 0.0080` on SceneGraphs, and `0.6732 ± 0.0076` on WebQSP, outperforming vanilla prompt tuning (`0.5876`, `0.6851`, `0.4975`) by `47.99%`, `22.97%`, and `35.32%`, respectively.
- **Best overall tuned setting**: G-Retriever + LoRA achieves `0.8768 ± 0.0274` on ExplaGraphs, `0.9077 ± 0.0104` on SceneGraphs, and `0.7011 ± 0.0050` on WebQSP, improving over plain LoRA by `0.31%`, `5.62%`, and `13.56%`.
- **Retrieval efficiency**: on SceneGraphs, average prompt size drops from `1,396` to `235` tokens and from `19` to `5` nodes (`-83%` tokens, `-74%` nodes); on WebQSP, it drops from `100,627` to `610` tokens and from `1,371` to `18` nodes (`-99%` for both tokens and nodes).
- **Hallucination mitigation**: in a manual study of `100` SceneGraphs responses, valid cited nodes improve from `31%` to `77%`, valid cited edges from `12%` to `76%`, and fully valid cited graph evidence from `8%` to `62%`.
- **Ablations on WebQSP**: removing the textualized graph lowers Hit@1 to `0.5414` (`-19.58%`), removing the graph encoder lowers it to `0.5963` (`-11.42%`), and removing edge retrieval lowers it to `0.5837` (`-13.29%`), indicating that both text and structure matter.
- **Robustness checks**: changing the graph encoder still keeps performance competitive, with GCN / GAT / GraphTransformer scoring `0.6853 / 0.6732 / 0.6910` on WebQSP and `0.8290 / 0.8696 / 0.9016` on ExplaGraphs; upgrading the LLM from Llama2-7b to 13b increases WebQSP Hit@1 from `0.6732` to `0.7371`.

## Limitations

- The retrieval module is static rather than trainable; the paper explicitly lists trainable retrieval as future work.
- The primary experiments rely on a frozen Llama2-7b backbone, with only limited appendix evidence for stronger LLMs, so the conclusions are somewhat coupled to that model family.
- Hallucination is evaluated through manual inspection of only `100` SceneGraphs outputs, which is informative but still small-scale.
- The benchmark aggregates and reprocesses three pre-existing datasets rather than introducing fully new raw collections, so coverage of graph QA settings remains limited.
- ExplaGraphs is small enough that retrieval is disabled with `` `k = 0` ``, so the advantages of graph retrieval are demonstrated most clearly on SceneGraphs and WebQSP rather than uniformly across all benchmark components.
