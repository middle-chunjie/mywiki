## Summary

Web2Code introduces a large-scale instruction-tuning resource and evaluation suite for multimodal large language models that must read webpage screenshots and generate faithful HTML. The dataset contains `1,179.7K` instruction-response pairs, combining `884.7K` webpage image-to-code examples with `295K` webpage question-answer pairs, and mixes newly generated data with refined prior web datasets. The evaluation suite separates webpage understanding (WUB) from webpage code generation (WCGB), replacing fragile string-level code matching with rendered screenshot comparison scored by GPT-4V across structure, aesthetics, textual fidelity, and UI consistency. Across several LLaVA-style backbones, adding Web2Code consistently improves webpage understanding and screenshot-to-HTML generation without clearly hurting general multimodal performance.

## Problem & Motivation

Existing multimodal large language models perform well on broad vision-language tasks, but they remain weak at webpage screenshots because the task requires OCR, layout understanding, spatial reasoning, and long-form HTML generation at the same time. Prior webpage datasets either lack instruction-following format, contain noisy text, or are too limited for tuning general-purpose MLLMs. The paper addresses this gap by building an instruction-tuning dataset specialized for webpage understanding and webpage-to-code translation, and by defining benchmarks whose metrics better reflect rendered webpage fidelity than raw code similarity.

## Method

- **Dataset composition**: Web2Code contains `1,179.7K` instruction pairs, split into `884.7K` webpage image-code pairs and `295K` question-answer pairs. The four main components are `DWCG = 60K`, `DWCG_R = 824.7K`, `DWU = 243.5K`, and `DWU_R = 51.5K`.
- **DWCG synthetic generation**: GPT-3.5 generates `60K` HTML webpages following CodeAlpaca-style prompts and `10` detailed diversity criteria; Selenium WebDriver renders each HTML page into a screenshot, and GPT-4 produces the final instruction-following prompts in a two-stage process that adds stylistic instructions.
- **DWCG_R refinement**: Existing code-generation datasets, mainly WebSight and Pix2Code, are converted into LLaVA-style instruction data; GPT-4 replaces noisy or random text, webpages are re-rendered into screenshots, and the refined pairs retain both HTML and visual supervision.
- **DWU question-answer generation**: GPT-4 generates `10` QA pairs for each of `24.35K` webpages, yielding `243.5K` question-answer instances that supervise webpage understanding using the rendered page and the question, without exposing HTML at input time.
- **DWU_R refinement**: WebSRC question-answer data are deduplicated and quality-filtered with GPT-4, reducing noisy repeated questions and producing `51.5K` higher-quality webpage-understanding examples.
- **Evaluation framework**: WUB evaluates webpage understanding on `1,198` screenshots and `5,990` binary QA pairs using accuracy. WCGB asks the model to generate HTML from a screenshot, re-renders the output with Selenium, and scores the result with GPT-4V on `10` criteria grouped into four matrices: `VSA`, `CAD`, `TCC`, and `UII`, each on a `0-10` scale.
- **Model/training setup**: The MLLM follows the LLaVA-1.5 recipe with a CLIP ViT-L/14 visual encoder at `336 x 336` resolution and patch size `14`, an LLM backbone, and a projector. Pretraining optimizes only the projector with batch size `256` and learning rate `1e-3`; instruction tuning optimizes projector + LLM with batch size `128`, learning rate `2e-5`, cosine decay, warmup ratio `0.03`, AdamW, `0` weight decay, and `1` epoch for each stage.

## Key Results

- On WCGB with LLaMA3-8B, progressively adding Web2Code components improves the overall score from `1.790` to `6.402`, `6.716`, `7.806`, and finally `8.530`.
- On WUB with the same LLaMA3-8B backbone, accuracy rises from `65.56%` on general-domain data alone to `74.84%` with the full Web2Code mixture.
- On CrystalChat-7B, adding DWU to the baseline improves general-domain evaluation from `POPE 86.86 -> 87.10`, `SciQA 67.77 -> 68.27`, and `TextVQA 57.84 -> 58.15`, suggesting the web data do not obviously damage general multimodal ability.
- Against WebSight baselines on LLaMA3-8B, full Web2Code reaches `WCGB Overall = 8.530` and `WUB = 74.84`, compared with WebSight v0.1 at `4.125` / `70.13` and WebSight v0.2 at `5.115` / `57.33`.
- On low-level Design2Code metrics with LLaMA3-8B, the full dataset yields `Block-Match 77.1`, `Text 96.9`, `Position 77.0`, `Color 66.2`, and `CLIP 86.7`.
- In a manual privacy audit of `1,000` samples, only `2.7%` contained privacy-sensitive content, specifically `16` email addresses, `11` phone numbers, and `0` credit card numbers.

## Limitations

- The dataset still may not cover the full diversity of real-world HTML, CSS, and interactive web behaviors, so models trained on it can miss uncommon layouts or complex dynamic pages.
- Synthetic and GPT-refined data may encode generation bias from GPT-3.5/GPT-4, which can shape both stylistic distribution and benchmark difficulty.
- WCGB evaluates rendered screenshots rather than code quality directly, so it does not measure efficiency, readability, maintainability, DOM semantics, or adherence to engineering best practices.
- Privacy risks are reduced but not eliminated: the paper's own audit still finds `2.7%` sensitive samples in a random inspection.
- The evaluation focuses on static screenshot understanding and HTML reconstruction, leaving open how well the resulting models transfer to full web agents that must act over live DOM state and interactive workflows.
