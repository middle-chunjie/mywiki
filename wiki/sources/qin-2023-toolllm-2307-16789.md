## Summary

⚠ Possibly outdated: published 2023; re-verify against recent literature.

ToolLLM is a tool-use framework for open-source LLMs that couples large-scale data construction, search-based annotation, model tuning, retrieval, and automatic evaluation. The paper builds ToolBench from `16,464` real-world REST APIs across `3,451` tools and `49` categories, generates nearly `200k` instruction-API pairs with ChatGPT, and retains `126,486` passed instruction-solution trajectories using a depth-first search-based decision tree (DFSDT). It then fine-tunes LLaMA-2 7B into ToolLLaMA, adds a Sentence-BERT API retriever, and evaluates with ToolEval. Empirically, DFSDT substantially outperforms ReACT, the retriever beats BM25 and `text-embedding-ada-002`, and ToolLLaMA approaches ChatGPT-level tool use while generalizing to unseen APIs and out-of-distribution APIBench domains.

## Problem & Motivation

Open-source LLMs had become competent at general instruction following, but they still lagged far behind closed models such as ChatGPT and GPT-4 in realistic tool use. The paper argues that the gap comes from missing supervision in three places: large-scale real-world APIs, multi-tool and multi-round tasks, and robust reasoning trajectories that actually execute tools and observe responses. Prior tool-learning resources were either small, simulated, single-tool, or dependent on manually specified API sets. ToolLLM is motivated by the need for an open pipeline that can teach models to select, call, and compose thousands of real APIs under realistic constraints, and to evaluate such behavior without relying on a single fixed ground-truth trajectory.

## Method

- **ToolBench construction**: crawl RapidAPI metadata at tool and API granularity, starting from `10,853` tools / `53,190` APIs and filtering down to `3,451` tools / `16,464` APIs after functionality and quality checks.
- **Instruction generation**: sample API subsets `\mathbb{S}_{N}^{sub} = \{API_1, \dots, API_N\}` and prompt ChatGPT to produce instructions plus relevant API subsets `\mathbb{S}_{*}^{rel} \subset \mathbb{S}_{N}^{sub}` for single-tool (`I1`) and multi-tool (`I2`, `I3`) settings.
- **Sampling strategy**: use RapidAPI hierarchy to sample `2-5` related tools from the same category or collection and up to `3` APIs per tool, reducing sparsity for multi-tool instructions.
- **Instruction data scale**: after filtering hallucinated relevant APIs, retain nearly `200k` instruction-relevance pairs: `87,413` (`I1`), `84,815` (`I2`), and `25,251` (`I3`).
- **Trajectory annotation**: cast tool use as multi-round decision making where action `a_t = ChatGPT(a_t | \{a_1, r_1, \dots, a_{t-1}, r_{t-1}\}, Inst_*)`; each step emits thought, API name, and parameters, then consumes the real API response `r_t`.
- **DFSDT reasoning**: expand a decision tree over candidate reasoning paths, let the model either continue a promising branch or call "Finish by Giving Up" to backtrack and explore a new child; use DFS instead of BFS to keep OpenAI API costs manageable.
- **Annotation output**: keep only passed trajectories after DFSDT search, producing `126,486` `(instruction, solution path)` pairs for ToolLLaMA training.
- **API retriever**: train a dense bi-encoder with Sentence-BERT on BERT-BASE, using generated relevant APIs as positives and contrastive negatives; rank APIs by embedding similarity and provide the top `5` APIs at inference.
- **ToolLLaMA training**: fine-tune LLaMA-2 7B in multi-round conversation format with learning rate `5e-5`, warmup ratio `4e-2`, total batch size `64`, maximum sequence length `8192`, position interpolation ratio `2`, and `2` epochs.
- **ToolEval**: evaluate with pass rate and win rate, using ChatGPT to judge multiple predictions per example with majority vote; human agreement reaches `87.1%` for pass rate and `80.3%` for win rate.

## Key Results

- **Retriever quality**: average NDCG improves to `78.0` / `84.9` at `@1` / `@5`, versus BM25 at `18.5` / `17.0` and `text-embedding-ada-002` at `49.6` / `45.4`.
- **DFSDT vs ReACT**: ChatGPT-based annotation pass rate rises from `35.3` with ReACT and `44.5` with ReACT@N to `63.8` with DFSDT.
- **Main ToolBench evaluation**: ToolLLaMA + DFSDT reaches average pass / win `66.7` / `60.0`, outperforming Text-Davinci-003 + DFSDT (`43.1` / `46.3`) and Claude-2 + DFSDT (`22.6` / `43.5`).
- **Retriever-integrated inference**: ToolLLaMA + DFSDT + retriever improves further to average pass / win `67.3` / `63.1`, slightly above the oracle-API variant.
- **OOD APIBench generalization**: with its own retriever, ToolLLaMA obtains AST accuracy `16.77` on HuggingFace, `51.16` on TorchHub, and `40.59` on TensorHub, exceeding Gorilla-ZS + BM25 on all three domains.
- **Oracle-retriever ceiling**: ToolLLaMA + Oracle reaches AST `88.80`, `85.88`, and `88.62` on HuggingFace, TorchHub, and TensorHub respectively.

## Limitations

- The paper is a `2023` preprint in a fast-moving domain; model, benchmark, and API baselines may now be outdated.
- ToolBench construction, trajectory annotation, and ToolEval all rely heavily on ChatGPT, so data quality and evaluation may inherit teacher-model biases.
- Main ToolBench experiments often assume oracle API access; the practical setting with real retrieval is tested, but not for every baseline under identical conditions.
- ToolEval is useful but imperfect: human agreement is `87.1%` / `80.3%`, which leaves non-trivial evaluator noise.
- RapidAPI endpoints are temporally unstable, so reproducibility depends on API availability, latency, and changing responses.
- The core open model is only LLaMA-2 `7B`; strong results do not necessarily imply the method scales or transfers cleanly to all larger or newer models.
