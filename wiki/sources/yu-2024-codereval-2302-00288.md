## Summary

CoderEval introduces an execution-based benchmark for pragmatic code generation, targeting the gap between standalone toy problems and real repository development. The benchmark contains `230` Python tasks from `43` projects and `230` Java tasks from `10` projects, with each task paired with function signatures, docstrings, tests, and contextual metadata. A central contribution is its six-level runnable taxonomy, which operationalizes how much external context a function depends on, plus the distinction between `oracle_context` and `all_context`. The authors also add human-labeled docstrings to reduce training-data leakage concerns and build a project-level evaluation platform that injects model outputs back into runnable repositories. Results show a substantial drop from standalone to non-standalone code generation, even for ChatGPT, making contextual dependency a primary bottleneck for practical code LLMs.

## Problem & Motivation

Existing code-generation benchmarks such as HumanEval mainly measure standalone function synthesis, but the paper finds that more than `70%` of functions in popular open-source Java and Python projects are non-standalone. That mismatch means benchmark scores can overstate practical usefulness in IDE or repository settings, where functions routinely depend on project-local types, APIs, variables, and constants. The authors therefore aim to build a benchmark that better matches real software development, reduces docstring memorization concerns, and supports fine-grained analysis of whether models can recover the specific contextual dependencies needed for correct code.

## Method

- **Benchmark construction**: curate `230` Python tasks from `43` projects and `230` Java tasks from `10` projects after mining high-star GitHub repositories across `14` frequent tags and manually screening for realistic development value.
- **Runnable taxonomy**: classify each task into `6` levels of dependency, `self-contained`, `slib-runnable`, `plib-runnable`, `class-runnable`, `file-runnable`, and `project-runnable`, where higher levels require progressively more external context.
- **Task filtering**: keep functions with English comments, successful execution in the verification platform, original tests passing, implementations longer than `3` lines, and typically fewer than `10` contextual tokens to avoid making every task impossible.
- **Context extraction**: run whole-project static analysis to identify `TypeReference`, `APIInvocation`, and `VarReference`, using a knowledge base spanning Python `3.0.0` to `3.10.0`, Java `1.8` to `17`, plus public packages from PyPI and Maven.
- **Dual context views**: record `oracle_context` for the code elements actually used in the gold implementation and `all_context` for all accessible definitions/imports in the current file.
- **Prompt design**: retain the original docstring and add a human-labeled docstring for every task, written by `13` engineers with at least `3` years of Python/Java experience to mitigate memorization and study prompt sensitivity.
- **Execution platform**: evaluate Python in per-project environments created with `pyenv`, `venv`, and `pip`; evaluate Java via incremental `javac`/`java` compilation and autogenerated `NewTestFile` wrappers inside Dockerized project environments.
- **Inference protocol**: generate `n = 10` samples per task with nucleus sampling at `temperature = 0.6`, use `max_window_length = 1024` for CodeGen and PanGu-Coder, run each experiment `10` times, and report the median group by `Pass@10`.
- **Metrics**: compute unbiased `Pass@k` for `k ∈ {1, 5, 10}` and propose `Acc@k` to measure recall of required contextual tokens in generated code.

## Key Results

- The corpus-level motivation is empirically strong: non-standalone functions exceed `70%` of functions in the top `100` open-source Java and Python projects the authors analyzed.
- CoderEval has much richer dependency structure than HumanEval: in Python, `84%` of tasks are `file_runnable`, `54%` are `class-runnable`, and only `17%` are `slib-runnable`; in Java, `slib-runnable` is only `35%`.
- On CoderEval, ChatGPT is best overall but still modest: Python `Pass@1/5/10 = 21.04/27.31/30.00`, Java `35.39/42.77/46.09`; PanGu-Coder beats CodeGen on both languages.
- The same ChatGPT model is far stronger on HumanEval than on CoderEval: Python `39.21/64.09/72.96` on HumanEval versus `21.04/27.31/30.00` on CoderEval; Java `38.21/59.35/67.23` versus `35.39/42.77/46.09`.
- Standalone tasks are much easier than non-standalone ones. For ChatGPT, Python `Pass@1` drops from `35.87%` on standalone tasks to `15.45%` on non-standalone ones; Java drops from `64.88%` to `19.67%`.
- Model complementarity exists but does not remove the gap: across `230` tasks, at least one model solves `91` Python tasks and `136` Java tasks, while only `32` Python tasks and `56` Java tasks are solved by all three models.
- Human-labeled docstrings change outcomes non-trivially: for ChatGPT on Python, `Pass@1` rises from `21.13%` to `26.61%`, but on Java it falls from `35.39%` to `26.96%`.

## Limitations

- The benchmark still covers only `460` tasks and `2` languages, so it does not fully represent repository-scale code generation across ecosystems.
- Model coverage is narrow: experiments use CodeGen, PanGu-Coder, and ChatGPT (`gpt-3.5-turbo`), leaving newer or larger open models for future work.
- The public-library knowledge base may be incomplete, so standalone versus non-standalone statistics can deviate slightly from ground truth.
- The paper releases `all_context` but does not experimentally test prompting with it, so the usefulness of richer repository context remains unresolved.
- Some dependency slices are sparse, especially in Java (`5` `plib-depend` tasks and `1` `file-depend` task), which limits per-level conclusions.
